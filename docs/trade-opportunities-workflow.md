# Trade Opportunities Workflow

Trade Opportunities is one of the two local-suite jobs ([local-models.md](local-models.md)). This document specifies its end-to-end control flow; the feature's design rationale — the archetype lens, the 3×3 matrix, the two detection modes, the opportunity schema, the continuity vocabulary — lives in [trade-opportunities.md](trade-opportunities.md).

Trade Opportunities is **two on-demand jobs sharing one page** — **Discover (DTO)** and **Audit (ATO)** ([trade-opportunities.md §The two jobs](trade-opportunities.md#the-two-jobs)). The step table below specifies **DTO's** end-to-end control flow (the full funnel); **ATO** is a reduced, user-directed flow over selected opportunities that **reuses DTO's stages** — its **Deep Audit** is the Step-5 per-candidate loop, its **Quick Audit** the engine-only cheap re-derivation — and is described in [§ATO: the audit flow](#ato-the-audit-flow).

The **DTO** job:
- loads the house view, investor profile, and run-level macro / positioning context
- **discovers** candidates through three converging feeders — bottom-up structured screens, a model-led hypothesis-research lane (its edge), and a carried-forward watchlist (the persisted discovery memory) — rather than reading a fixed list
- classifies each candidate by archetype, then computes a deterministic, archetype-weighted financial picture for it
- researches each candidate on the open web — building the leading-metric case **and a mandatory bear case**
- scores, gates, and assigns each survivor a deterministic risk tier and horizon → a 3×3 matrix cell
- computes **deterministic outcome labels** on prior picks (matured-window realized return vs sector / market, drawdown, whether the leading metric continued, failure mode) for calibration — plus a **continuous since-flagged read** refreshed each run for every carried-forward idea (live from its first subsequent run, for the matrix display and the carried-forward re-score) — carries prior opportunities forward with an updated status, and flags any overlap with current holdings

Both jobs run **on demand only**; their model work runs **entirely on local models** (ATO's Quick Audit makes no model call at all), with **no cost at the model layer**. A **single global run slot** serializes both against the report and Portfolio Analysis (only one runs at a time). For job states, the global run slot, cancellation, and error handling, see [scheduling.md](scheduling.md) and [run-tracking.md](run-tracking.md); for the failure posture (per-candidate checkpoint/resume, fail-soft research and discovery), see [trade-opportunities.md §Failure posture](trade-opportunities.md#failure-posture).

Unlike Portfolio Analysis, the feature is **not tied to current holdings** — **DTO discovers new ideas**, while ATO maintains the existing matrix. It still **requires a connected Schwab account**, whose option chains supply the per-candidate options-activity signal ([schwab-integration.md](schwab-integration.md)); holdings are read only for the deterministic owned/not-owned **cross-reference** (Step 8 in DTO, the touched names in an ATO audit), which never influences what is discovered or chosen.

## How to read this workflow

Every step below is tagged with a **Type** so it is obvious what the step actually does — the same three Types the Portfolio Analysis and report flows use:

- **Computed (app layer)** — deterministic Rust logic, with no model and no external network: local SQLite and filesystem reads, the candidate dedup / tradability filter, the deterministic **risk-tier and horizon assignment**, and the **financial-analysis engine** (every sub-score, target, and derived read).
- **API retrieval** — fetches from external sources: the **FMP discovery layer** (screener / peers / industry classification — the universe scan; **no `*-bulk` endpoint, off-plan**) and the **FMP per-symbol surface** (fundamentals, the revision signal, positioning, segments); company cross-checks from **SEC EDGAR / Stooq / FINRA**; run-level macro and positioning from **FRED / CFTC / CBOE**; the keyless **FMP Articles** feed the discovery scan reads (**no GDELT**); option chains from **Charles Schwab**; and the **web tool** (**keyless SearXNG for discovery**; Tavily only as the per-candidate loop's degraded fallback) the orchestrator runs *on a model's behalf*. The full per-source endpoint surface, with each call's **discovery / per-candidate / run-level** cardinality, is in [data-sources.md §Trade Opportunities — endpoint surface](data-sources.md#trade-opportunities--endpoint-surface).
- **Local-model call** — invokes a model on the app-supervised **Ollama** daemon ([local-models.md §Serving runtime](local-models.md#serving-runtime)): the primary reasoner **`Qwen3.5-122B-A10B`** in **thinking** mode (theme research, candidate research, archetype confirmation, scoring) or **non-thinking** mode (firm, directed consolidation), or the fixed **`Qwen3-Embedding-4B`** embedder (vectorization only). Every generative call is **schema-constrained** via Ollama's native `format` parameter — the model picks values, never structure.

Two load-bearing architectural rules frame the whole table, the same ones the report and Portfolio pipelines hold: **agents are pure stages, and the application layer owns all I/O** — a model stage consumes the structured input handed to it and emits a schema-validated result; when a research stage needs the web it *requests* a tool call and the orchestrator performs the fetch. And **the engine computes every number** ([local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)) — the model interprets computed values and never invents one; the **risk tier and horizon are likewise rule-derived, not model-picked**, so the same asset lands in the same cell run to run ([trade-opportunities.md §The opportunity space](trade-opportunities.md#the-opportunity-space)). For each model stage, the **Local-model call** block lists what the prompt includes and what the model returns. Per-step progress, per-request rows, and token/reasoning output stream to the run tracker over the shared `progress` seam ([run-tracking.md](run-tracking.md)), with **per-cell progress** as the matrix fills.

The job is a **funnel**: broad, cheap discovery narrows to a small candidate set, then each survivor gets expensive per-name validation — the condense-as-you-go discipline the rest of the suite uses, and the shape the data budget requires (the richest per-symbol signals are rate-relevant, so they are spent only on the narrowed set). The cardinality bands in the endpoint table track this directly: **discovery** calls scan the universe a handful of times, **per-candidate** calls are the budget driver, and **run-level** calls fire once.

## Step 1: Job Start and Gate

**Type:** Computed (app layer) — the local-suite execution gate. No model and no external API (credential and daemon *presence/reachability* are checked, not analysis).

The job will not start unless three preconditions hold, the same gate Portfolio Analysis clears ([portfolio-workflow.md §Step 1](portfolio-workflow.md#step-1-job-start-and-gate)):
- the **single global run slot** is free (no report or other local job is running — see [scheduling.md §Concurrent Job Protection](scheduling.md#concurrent-job-protection));
- the **local-model daemon is reachable and the configured roster is present** (the 122B reasoner + the embedder) — health-checked at the Ollama endpoint ([local-models.md §Serving runtime](local-models.md#serving-runtime));
- a **connected Schwab account** with a valid (≤7-day) refresh token ([schwab-integration.md §A connected Schwab account is required](schwab-integration.md#a-connected-schwab-account-is-required)) — Trade Opportunities needs it only for the per-candidate options-activity signal and the closing holdings cross-reference, but the connection is a hard precondition all the same.

This gate is **independent of the cloud-report gate** — a machine with no OpenAI/Anthropic keys can still run the local suite. Missing **configuration** (the Ollama endpoint or a roster id unset, or Schwab not connected / refresh token lapsed) locks the local-suite Run buttons and shows a persistent warning *before* this step — **local models not configured** and **Schwab connection**, one per category; a live **local-model connectivity** failure caught here at the run-gate (daemon unreachable, a rostered model not pulled) blocks the attempt **inline**, not as a persistent warning (ATO's **Quick Audit** reuses this gate but, making no model call, skips the daemon-connectivity check — [§ATO: the audit flow](#ato-the-audit-flow)); Schwab *API* reachability is not tested at this step — it surfaces later, when the options-activity signal / holdings cross-reference fetch runs ([portfolio-workflow.md §Step 1](portfolio-workflow.md#step-1-job-start-and-gate), [interface.md §Connection status](interface.md#connection-status-local-suite)).

## Step 2: Load Shared Context

**Type:** Computed (local read — house view, investor profile, prior run) + API retrieval (run-level FRED / CFTC / CBOE). No model.

Loaded **once per run and shared across every candidate**, not re-requested per name:
- the **Market Signal house view** — the latest report's Thesis, Investment Strategy, and Forward Outlook sections plus recent report summaries (`thesis_stance`, `forward_outlook_themes`, `key_risks`), loaded **deterministically** from the report store (retrieve-don't-dump — never by vector-searching the report's memory; see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)). The report's **creation date** rides into the context, and the same **one-week freshness window** Portfolio applies holds here: a house view older than one week is **omitted and recorded as a gap** rather than fed as current ([portfolio-workflow.md §Step 5](portfolio-workflow.md#step-5-load-shared-context)). The house view supplies the **macro / regime backbone of the job's worldview** — its `market_cycle` and `risk_posture` are a compact growth × inflation regime read ([trade-opportunities.md §The research method](trade-opportunities.md#the-research-method)) — and steers *where the job hunts* (it biases the top-down theme scan and the archetype-appropriate read) without being a number the engine consumes; the **forward thematic map** that completes the worldview is the job's own, built in the Step-3b hypothesis research;
- the **investor profile** — for now a **fixed default preset** (long-term horizon, profit-maximization objective, medium-to-high risk tolerance, **cash treated as always available**, no tax adjustment; user configuration is deferred — see [configuration.md §Investor Profile](configuration.md#investor-profile)) — which shapes the **entry framing and conviction emphasis**, never which opportunities qualify; because cash is unconstrained, full-size and *add-style* entries are never gated on observed Schwab cash;
- run-level market context — the **risk-free rate** (FRED `DGS10` / `DGS2`) that anchors the engine's discounting and scenario targets; **cyclical commodity prices** (FRED daily energy + monthly IMF metals, Stooq futures) read both as the **commodity-cyclical discovery feeder** (a price turn surfaces the sleeve) and as per-candidate context; **CFTC Commitments-of-Traders positioning** on the bellwether contracts (the commodity sleeve's positioning read); and the **CBOE venue-level put/call backdrop** (a broad-market sentiment context, not a per-name signal);
- the **prior run's persisted opportunity matrix**, loaded for the Step-7 carry-forward continuity check; and the **persisted opportunity graph** — the job's **discovery memory** (prior hypotheses with their economic value-chain traces, and the **watchlist** of worthy-but-unpicked names with each one's leading metric + falsifiers), loaded so Step 3b can extend or retire existing theses rather than re-deriving blind and Step 3c can re-check the watchlist ([trade-opportunities.md §Discovery memory](trade-opportunities.md#discovery-memory-the-opportunity-graph)).

## Step 3: Candidate Discovery

**Type:** mixed — three converging feeders (see 3a–3c). This is where ideas are **found**, not merely vetted: model-led research is a first-class *discovery* feeder that runs here, before candidate selection, not only as a downstream validation step ([trade-opportunities.md §The pipeline](trade-opportunities.md#the-pipeline)). The three feeders are the **bottom-up structured screens** (3a), the **model-led hypothesis research lane** (3b — the edge: it forms investable *hypotheses* and reasons its way to names), and the **carried-forward watchlist** (3c — the persisted discovery memory that re-checks worthy-but-unpicked names from prior runs). All are **fail-soft** — a failed screen, sweep, or re-check degrades to fewer candidates, never a failed run.

### Step 3a: Bottom-Up Structured Screens

**Type:** API retrieval (FMP discovery layer + FRED / Stooq + FINRA) + Computed (app-layer slicing). No model.

Cheap, broad, structured feeders over the whole universe, run deterministically and tagged with the signal that surfaced each name. **The plan carries no `*-bulk` endpoint, and the company screener exposes only coarse market-structure fields** (market cap, sector / industry, beta, price, dividend, volume, exchange — no valuation / profitability / growth filter; [data-sources.md §FMP — current paid-plan tier audit](data-sources.md#fmp--current-paid-plan-tier-audit)), so this stage **cannot pre-score the universe**. It *generates and stratifies* a candidate longlist; the multi-factor quant composite and the forensic gate move entirely to the per-candidate engine (Step 5c), computed only on the narrowed set the budget can afford:
- **Company screener → universe definition + breadth stratification** — `company-screener` applies the **hard tradability gates** (price / volume / market-cap floors, `isActivelyTrading`, the allowed exchange set, equities only) and tags every eligible name with its **market-cap band and sector / industry**. This is the breadth backbone — it defines the strata Step 4 fills, **not a ranked shortlist** (there is no fundamental field to rank on).
- **Event / catalyst & positioning feeds** — the market-wide **insider-buy feed** (`insider-trading/latest`) for cluster buys, new institutional / activist stakes and **short-interest extremes** (FINRA's consolidated biweekly file, fetched once per run), notable **congressional** buys (FMP), and the **event feeds** (`mergers-acquisitions-latest`, `sec-filings-8k`, `ipos-calendar`, the report's free movers / earnings-calendar) for fresh catalysts. This is the most selective structured feeder — a name surfaces because something **already moved**, so it skews coincident / lagging; the *early* names come from the Step-3b research lane.
- **Commodity-price turns** (FRED / Stooq) for the **cyclical sleeve** — a spot / contract-price turn at washed-out sentiment surfaces commodity-cyclical candidates.

Each feeder reduces to a candidate set with its surfacing signal attached. **The per-symbol rate discipline now binds** — the rich factors `*-bulk` once amortized across the universe are spent per candidate downstream, which is exactly what the Step-4 budget rations ([data-sources.md §Trade Opportunities — endpoint surface](data-sources.md#trade-opportunities--endpoint-surface)).

### Step 3b: Model-Led Hypothesis Research

**Type:** Local-model call (122B, thinking) + API retrieval (news stack + web tool), **looped**. The job's **edge**: a research-active feeder that forms investable **hypotheses** and reasons its way to names — not a "theme → tickers" lookup. It runs **worldview → hypothesis → mechanism → value-chain node → leading metric → candidate**, so the model commits to a *hypothesis first* and only then to the public-company expressions of it. It builds the **forward thematic map** — the job's worldview layer ([trade-opportunities.md §The research method](trade-opportunities.md#the-research-method)) — inside the **same bounded per-topic research loop the per-candidate stage uses** ([web-research.md](web-research.md)), here aimed at *discovery*, in three movements (**each topic / route its own model call over a clean context** so the local model is never handed a sprawling prompt):

1. **Research-strategy planning** — rather than a fixed agenda, the reasoner (given the Step-2 house view and the carried-forward opportunity graph) **chooses which research routes to spend the discovery budget on** — e.g. a *policy / regulatory* route, a *supply-chain* route, a *technical-bottleneck* route, a *procurement / grant / capex* route, a *customer-capex* route, an *industry-history* route, a *failure-analogue* route, and an *event-impact / value-chain-repricing* route — **each carrying its own source strategy** so a route targets the right source types rather than leaning on generic web search: *policy* → legislation, agency notices, procurement databases; *supply-chain* → trade journals, filings, customer / supplier commentary; *technical-bottleneck* → standards bodies, engineering blogs, patent / product docs, trade publications; *event-impact* → the announcing company's primary materials (spec sheets, reference designs, keynote / launch docs), standards bodies, teardown / engineering analysis, and the affected names' segment disclosures (and so on per route; at implementation these per-route source strategies are expressed as a **schema / rubric**, not free prose, so a route reliably targets its source types). The **event-impact route is gated by a materiality test** so it doesn't chase every product headline — it fires only on a material technology / product / standard announcement **plus** at least one corroborating condition (meaningful group repricing, credible primary-source documentation, a customer-adoption signal, or clear value-chain exposure); absent the gate the route stays dormant and the run behaves exactly as before ([trade-opportunities.md §The event-impact / value-chain repricing lens](trade-opportunities.md#the-event-impact--value-chain-repricing-lens)). **One route is always a mandatory *outside-view challenge*, run graph-blind** — "assume the carried-forward graph is stale: what world-change are we missing?" — so stateful memory can't anchor the job to its own prior hypotheses (the live complement to Step 3c's carry-horizon pruning). Route selection is itself bounded (a capped number of routes per run, priority-ordered); this is where aggressive deep research becomes real rather than "search the web for themes."
2. **Hypothesis formation & scoring** — each route's research produces structured **hypothesis cards**, the worldview reasoning made explicit and auditable: *what is changing in the world* → *the affected system / mechanism* → *who gains pricing power / where the margin is captured* (the economic value-chain trace — margin capture, bargaining power, capacity constraint / bottleneck, and pricing power **versus mere exposure**, deliberately past the crowded pure-plays to the **picks-and-shovels enablers** at the constrained, margin-capturing nodes, often mid / small cap) → *the leading metric that would prove it* (tagged with its **re-check class** — see Step 3c) → *the likely public-company expressions* → *bear case* → *the key falsifiers + sources to verify*. An **event-impact route's cards are explicitly two-sided** — it traces the chain into **beneficiaries**, **feared losers** (the names that sold off, with the *actually-exposed* revenue / profit pool sized), and **latent names** (chain nodes that did not move but should be affected) — and each affected name carries a typed **`technology_read`** sub-object — the full typed schema is in [trade-opportunities.md §The event-impact / value-chain repricing lens](trade-opportunities.md#the-event-impact--value-chain-repricing-lens) — so the panic-vs-real and euphoria-vs-real reads are *sized*, not narrative. Each card is then **scored on its own merits, before any ticker** — magnitude, durability, time horizon, **leading-metric observability**, crowding / priced-in risk, and **margin-capture clarity** — a **model-produced score the app validates against two named thresholds** (the same model-proposes / app-validates split as Steps 5g/5h): the higher **promotion threshold** (the score a hypothesis must clear for its names to enter validation this run) and the lower **watchlist threshold** (the score below which a still-live hypothesis is merely remembered — Step 3c). Names are therefore promoted from *strong* hypotheses, not merely interesting narratives. The model commits to a hypothesis before it names a ticker; the value-chain trace is the centerpiece, not an afterthought.
3. **Adversarial vetting & promotion** — before a hypothesis's names flow on, the reasoner runs explicit adversarial passes alongside the mandatory bear case: **why is this already priced?**, **why might the obvious beneficiary be the wrong expression?**, and **who actually captures the margin instead?** — and, for any **feared-loser** name an event-impact route surfaced, the **symmetric pass**: **is the impairment real or panic, and what is the actually-exposed revenue / profit pool?** (the mirror of the beneficiary-side challenge, so an oversold incumbent is promoted as an early-detection candidate only when its real exposure is genuinely narrow, and a truly impaired one is left alone). A hypothesis that **clears the promotion threshold *and* survives the adversarial passes** maps to candidate names (resolved against FMP's industry classification / peers); one that doesn't is **recorded in the opportunity graph** (with its score, falsifiers, and the failing pass) rather than forced into a name — a hypothesis below the promotion threshold but above the **watchlist threshold** can still seed a **watchlist** node if it meets the watchlist bar (Step 3c).

The research nests **route ⊃ topic ⊃ pass ⊃ fetch** — a route is worked as one or more **topics** (each its own model call over a clean context), each topic is a root **pass** plus ≤2 app-governed follow-ups, and each pass is itself a bounded multi-turn loop of web-tool **fetches** ([web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management)). The two ceilings therefore bind at different levels: **per-topic depth ≤2 (≤3 passes/topic — counting passes, not searches, and not per route)** and a **per-run discovery fetch + wall-clock budget that binds first**, capping fetches across every route, spent in route-priority order, fail-soft on exhaustion — so the discovery research has a hard ceiling and can never run away. The Step-2 house view steers the hunt (aligning it with the current regime / thesis) without confining it. Search is **keyless SearXNG only** — **no Tavily and no GDELT for discovery**: GDELT's escalating rate-limit / IP-lockout makes it unreliable (the same reason the report job dropped it), and Tavily is **not a discovery dependency**; if SearXNG is unavailable this lane **fail-softs to fewer candidates** rather than falling back to a keyed provider. The **FMP structured news feeds** (`news/general-latest`, `news/stock-latest`, on the paid key — `news/press-releases-latest` is off-plan, [data-sources.md §FMP — current paid-plan tier audit](data-sources.md#fmp--current-paid-plan-tier-audit)) plus the keyless **FMP Articles** feed and the macro-release calendar **seed** the route research with ticker-tagged, dated headlines, which the web tool then **deep-reads**. Those seeds are captured as the route's **seed-lineage**, never as evidence: a seed whose URL the route deep-reads stamps the resulting evidence-ledger claim with a **`surfaced_by`** back-pointer, and each hypothesis card additionally records a bounded, config-capped **`seeded_by`** list of the seeds the reasoner judges to have oriented it (even ones it never fetched) — **each entry validated against the route's fed seed-ID set, with unknown references dropped and logged** so lineage can't be fabricated — so the headline→hypothesis path is auditable while the seed itself is never scored as conviction ([web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management)). The lineage flows onto the hypothesis card and persists on the opportunity-graph node.

**Heavy routes sub-distill before emitting cards.** Most routes resolve in a single card-formation call, but a route can accumulate more research than one call can consolidate cleanly — most often the **event-impact route**, which spans three sides (beneficiaries / feared-losers / latent) and many entities. The orchestrator classifies a route **heavy deterministically**, *after* its research loop completes and *before* card formation, by the same tier-1 primitive the per-candidate distillation uses ([web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management)): heavy when **either** its accumulated full findings + evidence ledger would **overflow the input budget** of a single card-formation call (a byte/token measure against a configured fraction of that budget), **or** the route spans **more than one substantial sub-agenda** — where *substantial* is itself measured deterministically (a sub-agenda whose accumulated evidence-ledger size crosses a configured per-side threshold), with the **event-impact route's sides counted substantial whenever populated** (beneficiary / feared-loser / latent are semantically distinct and warrant per-side focus regardless of size), or any route resolving to more than *K* distinct hypotheses. A heavy route is **tier-1 sub-distilled along its natural seam** (per side / per sub-agenda), then a bounded **route-level reduce emits the route's cards**. That reduce still emits **many distinct cards** — it structures *within* the route and is **not** a cross-route merge; the cross-route consolidation stays the deterministic Step 4 (no model collapses distinct hypotheses, so breadth and auditability are preserved). Sub-distillation is **bounded** — a per-route sub-distillation cap (config), spent in priority order from the existing per-run discovery budget (wall-clock binds), fail-soft to a recorded gap on exhaustion — and the **heavy classification and sub-unit count are logged to the audit record**, so the fan-out is never silent ([configuration.md §Local Analysis Suite Configuration](configuration.md#local-analysis-suite-configuration)).

#### Local-model call — Hypothesis research & exposed-name discovery (Qwen3.5-122B, thinking)

**Model.** The resident 122B reasoner in thinking mode, requesting `web_search` / `web_fetch` tool calls the orchestrator executes (**keyless SearXNG only for discovery — no Tavily fallback**; SSRF-guarded; untrusted page text inserted as quoted evidence, never as instructions — see [web-research.md §Safety and provenance](web-research.md#safety-and-provenance)). The orchestrator owns every request and the per-route depth / fetch / wall-clock budget, exactly as in the per-candidate loop. This lane is where the model **reasons its way to names** — forming a hypothesis, tracing it down the value chain to the operators that capture the margin, and surfacing under-covered / early-inflection names no structured feed carries — rather than ranking names handed to it; each surfaced name carries its hypothesis lineage + rationale + source URLs and is verified (exists, US-listed, clears the tradability gate) before it can earn enrichment budget.

**Prompt — input.** **One route's questions at a time** over a clean context — the Step-2 house view (regime, themes, forward outlook), the **carried-forward opportunity graph** (prior hypotheses + watchlist, so the model extends or retires existing theses rather than re-deriving blind — **except the mandatory outside-view route, which is run graph-blind** so it can't anchor on prior theses), and the keyless news / macro-release inputs relevant to that route; routes never share a context.

**Returns.** A schema-validated set of **hypothesis cards** (each: world-change → mechanism → economic value-chain trace [margin capture / bargaining power / capacity constraint / pricing power vs exposure] → leading metric [tagged with its **re-check class** — `structured_metric` / `filing_metric` / `research_metric`, Step 3c] → candidate expressions → bear case → key falsifiers → sources, the bounded **`seeded_by`** seed-lineage [the structured-feed headlines that oriented the hypothesis, config-capped and validated against the route's fed seed-ID set — [web-research.md](web-research.md#the-research-loop-and-context-management)], the **hypothesis score** [magnitude / durability / horizon / metric-observability / crowding / margin-capture clarity], plus the adversarial-pass verdicts; an **event-impact card additionally carries each affected name's side** [beneficiary / feared-loser / latent] **and its typed `technology_read`** [substitute · complement · mix-shift / exposed revenue · profit / deployment timeline / switching costs / margin-capturing node / source confidence], with the symmetric feared-loser pass verdict where applicable) and the **candidate names** they resolve to — each with its hypothesis lineage, surfacing rationale, and source URLs — flowed into Step 4 alongside the bottom-up screens and the watchlist, and **written to the opportunity graph** (the persisted discovery memory — [trade-opportunities.md §Discovery memory](trade-opportunities.md#discovery-memory-the-opportunity-graph)). The thematic map / hypothesis set is retained as run-level worldview context for the per-candidate thematic-fit read (Step 5d). The model proposes hypotheses and names; it neither fetches per-symbol data nor scores them here.

### Step 3c: Carried-Forward Watchlist Re-check

**Type:** Computed (engine re-pull of leading metrics) + app-layer promotion. No model.

Discovery is **no longer stateless**. The persisted **opportunity graph** (loaded at Step 2) carries forward every worthy-but-unpicked candidate from prior runs as a **watchlist node**. The **discovery-worthiness bar is app-enforced, not model discretion** — so the model can't over-watchlist vague but interesting ideas: a node qualifies only with a **named, countable, dated leading metric** (tagged by re-check class, below), **a stated economic mechanism**, **at least one falsifier**, and a **hypothesis score above the watchlist threshold** (Step 3b). Each qualifying node is kept with its hypothesis lineage, the **specific leading metric to monitor**, its **key falsifiers**, and source timestamps. This closes the silent-loss gap in stateless re-discovery: a deferred name that quietly starts compounding would otherwise depend on chance to re-surface in a later sweep, and there is no promise it would.

Not every leading metric is equally cheap to re-check, so each node tags its metric by **re-check class**:
- **`structured_metric`** (revision velocity, relative strength, short interest, earnings surprise) — re-pulled **free every run** from the engine's structured feeds;
- **`filing_metric`** (segment revenue, margins, backlog disclosed in filings) — re-checked on **filing cadence** (SEC / FMP), not every run;
- **`research_metric`** (product ASPs, channel checks, scuttlebutt with no structured feed) — re-checked only within a **bounded research sub-budget**, at lower frequency.

For each node the engine re-pulls the leading metric **at its class's cadence** and the app applies a rule:
- **metric inflected / continued** (the hypothesis is confirming) → the node is **promoted as a priority feeder** into Step 4, so a maturing thesis is never missed for want of a fresh trigger;
- **falsifiers tripped, metric dead / stale, or the carry horizon elapsed** (a configurable maximum number of runs a hypothesis is monitored without confirming) → the node is **retired** (kept in history for outcome-learning, removed from active monitoring), so the watchlist self-prunes and cannot grow unbounded;
- **otherwise** → the node **stays on the watchlist**, carried forward with a refreshed timestamp.

The watchlist is bounded by a configurable retention cap ([configuration.md §Local Analysis Suite Configuration](configuration.md#local-analysis-suite-configuration)). It is the **discovery-memory** analog of the validated matrix's Step-7 carry-forward — *upstream* (worthy candidates remembered before validation) rather than *downstream* (validated picks tracked); the two are distinct stores ([trade-opportunities.md §Discovery memory](trade-opportunities.md#discovery-memory-the-opportunity-graph)).

## Step 4: Candidate Consolidation

**Type:** Computed (app layer) — a deterministic merge before any per-candidate stage. No model.

The three feeders' outputs — the bottom-up structured screens (3a), the model-led hypothesis names (3b), and the promoted carried-forward watchlist nodes (3c) — are **deduped** into one candidate list, **sanity-filtered for tradability** (exchange listing, a liquidity / price floor, an instrument-type filter — funds and non-equities drop out, since the job hunts operating businesses by archetype), and each surviving name is **tagged with every signal that surfaced it** (which screen, which hypothesis, the positioning flag, whether it is a maturing watchlist name). This is the **cross-route / cross-feeder reduce, and it is deliberately deterministic** — distinct hypotheses are never collapsed by a model (which would destroy auditable breadth and could silently drop a name); the only model-side consolidation in discovery is *within* a route's own card formation (the Step-3b heavy-route sub-distillation). The contrast with the per-candidate stage is intentional: Step 5e's reduce *is* a model call because it fuses one candidate's lenses into one verdict, whereas discovery's reduce must keep many distinct candidates and so stays computed here.

The narrowed set is bounded by a **research budget** — a ceiling on how many names get the expensive per-candidate validation in one run — and the budget exists for **compute, not quality**: deep per-candidate web research runs on a local reasoner, so a single run cannot validate the whole universe ([trade-opportunities.md §The pipeline](trade-opportunities.md#the-pipeline)). Two properties follow, and both matter: the budget is **generous and user-configurable** (Settings → Trade Opportunities discovery breadth — [configuration.md §Local Analysis Suite Configuration](configuration.md#local-analysis-suite-configuration)); and it is **never a quality verdict** — nothing is validated at this point, so a name that doesn't make the cut is **not rejected, only deferred**. Crucially, deferral is **no longer a coin-flip on re-surfacing**: a genuinely worthy deferral (a real hypothesis + an identified leading metric) is **written to the persisted opportunity graph as a watchlist node** and **re-checked every later run** (Step 3c), its leading metric monitored, so a deferred name that quietly compounds is caught rather than lost to chance. This **discovery-memory carry-forward** is distinct from — and upstream of — the *validated* carry-forward (the opportunity matrix reloaded at Step 2 for the Step-7 continuity check). A name not even worth watchlisting carries no state and is simply re-derivable by a later sweep; the watchlist itself is bounded (retention cap + self-pruning of falsified / stale / horizon-elapsed nodes, Step 3c), so the store never grows unbounded.

Within that budget the slots are filled **not by a flat top-N**. There is no universe-wide composite to rank on at this point — the fundamental multi-factor score and the forensic gate are computed per candidate only at Step 5c, on the names that clear this stage — so with bulk pre-scoring off-plan the **stratification itself is the breadth mechanism**: a pure rank on the cheap surfacing signals alone (feeder strength, theme fit, the coarse screener fields) would collapse the funnel onto whatever is loudest (mega-cap momentum, the most-covered AI names, one crowded macro theme) and throw away the breadth that is the job's edge ([trade-opportunities.md §The research method](trade-opportunities.md#the-research-method)). The slots are therefore filled **under deliberate diversity guardrails**, each a floor-and-ceiling rather than a single ranking:
- **market-cap band** — a **floor for mid- and small-cap** slots so the mega-cap tail can't fill the slate (the enabler the worldview prizes is frequently sub-large-cap), and a **ceiling on mega-cap** names;
- **feeder** — no single feeder (a structured screen, the model-led hypothesis lane, the carried-forward watchlist, a positioning scan) may supply more than a set share, so the narrowed set reflects *converging* signals rather than one loud one;
- **archetype** — each of the five archetypes gets representation up to a cap, so a single archetype (today, ai-infra) can't monopolize the set; since the model-confirmed archetype is not set until Step 5a, the quota uses a **provisional archetype** assigned deterministically here from the cheap classification features already in hand (sector / industry + the Step-4 surfacing tags), with Step 5a's confirmed archetype the authoritative one for 5c–5g;
- **sector / theme** — a per-sector and per-theme ceiling so one crowded macro theme (AI, GLP-1) can't dominate discovery.

Within each axis's floor and ceiling the names are still ranked by **signal strength × house-view fit**; the guardrails shape the *composition* of the narrowed set, never relax the per-name bar, and a candidate displaced by a quota is **likewise only deferred** (watchlisted if worthy and re-checked next run, else re-derivable), not dropped. The result is the narrowed candidate set the expensive per-name loop runs over — the funnel's waist, kept deliberately broad. None of this caps the *output*: the budget bounds only how many names get **researched** this run; how many **validated** opportunities reach the matrix is set by the gates alone (Step 6).

The budget is **distributed equally across the cap-band × sector buckets by default** — an explicit, configurable allocation rule: equal-per-bucket maximizes breadth (the design default), while proportional-to-population and signal-adaptive variants are calibration knobs tuned against the outcome-learning scorecard ([trade-opportunities.md §Outcome learning](trade-opportunities.md#outcome-learning-calibration)), so no single crowded stratum consumes the per-name compute.

**New candidates are prioritized within the budget.** The feeders surface brand-new names *and* existing live opportunities that re-tripped a feeder. DTO spends the deep-research budget on **new names first** (filled under the diversity guardrails above); only **leftover** budget goes to the re-surfaced existing opportunities, ordered by **oldest `last_deep_researched_at`** (the stalest first, ties broken by `became_opportunity_at` then ticker). **Both classes — new names and budget-winning re-surfacers — enter the same Step-5 validation loop as candidates** (a re-surfacer flagged carried-forward, so Step 5b loads its prior record + `continuity_weight`); a re-surfacer's deep pass therefore runs **in Step 5, before matrix assembly**, not in a separate Step-7 pass. A re-surfaced existing opportunity that doesn't win a leftover slot is **not researched** — it falls to the Step-7 cheap re-derivation like any other live name, and its re-surfacing raises the attention warning so the user can choose a Deep Audit. Every ticker DTO deep-researches — new or re-surfaced — is recorded in a **run-scoped deep-research set** so Step 7 never also cheap-sweeps it: within a run a ticker is deep-researched **at most once** ([trade-opportunities.md §The two jobs](trade-opportunities.md#the-two-jobs)).

## Step 5: Per-Candidate Validation Loop

Each candidate from Step 4 is processed through the chain below. Candidates are independent, so the loop **checkpoints per candidate** — each candidate's completed stages persist, so a cancellation or a single model failure resumes the unfinished candidates rather than restarting the (potentially long) run, and recent research is cached within a freshness window ([trade-opportunities.md §Failure posture](trade-opportunities.md#failure-posture)). The resident **122B reasoner fills every model role in this loop** by switching mode (thinking ↔ non-thinking), so moving a candidate across archetype confirmation (thinking), its research passes (thinking), distillation (non-thinking — single or hierarchical, Step 5e), and scoring (thinking) pays no model-swap cost ([local-models.md §The model roster and per-task routing](local-models.md#the-model-roster-and-per-task-routing)). The shared **financial-analysis engine** is the same one Portfolio Analysis uses ([portfolio-analysis.md §The per-holding pipeline](portfolio-analysis.md#the-per-holding-pipeline)); the difference is the **archetype** selects which signals it weights and which valuation lens it applies.

### Step 5a: Archetype Classification

**Type:** Computed (deterministic features) + Local-model call (confirmation). The lens that decides which signals matter for this candidate.

The application computes the candidate's **classification features** — sector / industry (FMP `profile`), margin and recurring-revenue structure, cyclicality, and the signals that surfaced it (Step 4 tags) — and the reasoner **confirms** one of the five archetypes (secular-compounder / ai-infra / commodity-cyclical / disruptor / quality-compounder). The confirmed archetype selects the **signal weighting and valuation lens** for 5c–5g ([trade-opportunities.md §Archetype](trade-opportunities.md#archetype--the-lens-that-decides-which-signals-matter)) — e.g. a commodity cyclical is judged on P/B, P/NAV, and mid-cycle EPS with trailing P/E suppressed, while an ai-infra name is judged on segment-revenue acceleration and forward P/E against its revision rate — and informs the deterministic risk-tier and horizon reads. Archetype is a first-class dimension (deterministic features, model-confirmed); it feeds the risk-tier rule rather than replacing it.

#### Local-model call — Archetype confirmation (Qwen3.5-122B, thinking)

**Model.** The resident 122B in thinking mode; schema-constrained output.

**Prompt — input.** The candidate's classification features and surfacing signals only — a compact, clean context. No web access (this is a classification, not research).

**Returns.** The schema-validated **archetype label** plus a short rationale and a confidence; a low-confidence or contradictory classification flags the candidate for a wider valuation read rather than committing to one lens.

### Step 5b: Dossier Assembly

**Type:** API retrieval (FMP per-symbol / SEC EDGAR / Stooq / FINRA / Schwab chains) + Local-model call (embedding, for continuity retrieval) + Computed (assemble the packet).

The application builds the candidate's evidence packet deterministically: the **FMP per-symbol surface** on the paid key — fundamentals (statements / ratios / key metrics, `financial-scores` for Altman Z + Piotroski, owner earnings, enterprise values, DCF, **multi-year `financial-growth`** per-share CAGRs), **product / geographic revenue segments**, the **analyst / revision signal** (forward estimates snapshotted for velocity, price-target consensus + summary trend, the `grades-historical` rating distribution + consensus, upgrades / downgrades, earnings surprises), **symbol-keyed positioning** (insider, **SC 13D / 13G activist filings**, Senate / House congressional — FMP **13F + holder-level analytics are off-plan** → SEC EDGAR 13F or omitted, and held out of the grade until calibrated regardless; [data-sources.md §FMP — current paid-plan tier audit](data-sources.md#fmp--current-paid-plan-tier-audit)), **stock peers**, **share float / liquidity**, the **next earnings date** (catalyst), **earnings-call transcript language** (backlog / book-to-bill / guidance / supply-discipline — **transcripts are off-plan**, so this rides the Step-5d web-research lane, not an FMP pull), the **symbol-scoped structured news** (`news/stock` — the narrative / sentiment / catalyst feed the research lane deep-reads; `news/press-releases` is off-plan, primary-source disclosures via `sec-filings-8k` + web), and **M&A involvement** (acquirer or target — via market-wide `mergers-acquisitions-latest` + `sec-filings-8k`, since the per-symbol `mergers-acquisitions-search` is off-plan) — joined with **SEC EDGAR** (10-K/Q/8-K + XBRL company facts) as the authoritative cross-check; **FINRA** short interest (looked up in the once-per-run file); **Stooq** deep price history and a live **FMP `quote`**; the **Schwab option chain** for the candidate if it is optionable (→ the options-activity signal, computed in 5c); the Step-2 shared context; and vector-retrieved continuity from **this job's own prior runs** for this candidate. For a **carried-forward candidate** (an existing opportunity being deep-re-evaluated — via a DTO leftover-budget re-surface or an ATO Deep Audit), the dossier also carries a derived **`continuity_weight`**, banded from the age of the prior record's `last_deep_researched_at` (≤ ~1 week *continued research* → > ~4 weeks *fresh look*, mirroring the `research_forward_assumption` freshness window), that frames how hard Step-5g scoring leans on the prior thesis / conviction / bear case — it weights interpretation only, never an engine number, and a name re-entering from the archive carries none (a fresh start — [trade-opportunities.md §The opportunity](trade-opportunities.md#the-opportunity)). The full input list and every endpoint, with cardinality, is in [trade-opportunities.md §Signal inputs](trade-opportunities.md#signal-inputs) and [data-sources.md §Trade Opportunities — endpoint surface](data-sources.md#trade-opportunities--endpoint-surface). This is the **per-candidate** surface — the budget driver — so it runs only for the narrowed set, never the discovery longlist.

#### Local-model call — Vector continuity retrieval (Qwen3-Embedding-4B, fixed)

**Model.** The fixed local embedder — vectorization only, no reasoning. Shares the `Embedder` trait the report pipeline defines; only the vector space differs.

**Prompt (input text).** A query string built deterministically from the candidate (symbol, archetype, sector/industry, and the prior opportunity's thesis themes if carried), byte-capped before the call.

**Returns.** A fixed-dimensionality vector; the application runs a brute-force cosine search scoped to the **Trade Opportunities** memory partition (the job namespace — never the report's or Portfolio Analysis's — see [local-models.md §Run history and continuity](local-models.md#run-history-and-continuity)) and carries the relevant prior analysis into the dossier.

### Step 5c: Deterministic Analysis (Archetype-Weighted Engine)

**Type:** Computed (the financial-analysis engine, shared with Portfolio Analysis). No model.

The engine computes the candidate's quantitative picture, **weighted by the Step-5a archetype** — the archetype selects which sub-scores dominate and which valuation lens applies. It produces: the **multi-factor quant composite** (value / quality anchored on gross-profitability / price + fundamental momentum / low-volatility / revision flow — **computed here for the first time**, since discovery only stratified by coarse fields; built as a **winsorized rank-sum** scored **within the candidate's cap-band × sector cohort** (sector-neutral, so the composite isn't a disguised sector bet), with any per-symbol factor that failed to resolve **imputed to the neutral median rank** so a missing call can't silently sink a name, and the factors **integrated at one final rank** rather than chained hard cutoffs — value and momentum are negatively correlated, so sequential cutoffs would collapse breadth) and the **value-creation read** (**ROIC vs cost of capital**, the reinvestment runway *g* ≈ ROIC × reinvestment, owner earnings with **R&D capitalized** so research-heavy names aren't mis-scored, and the moat-source features); the **leading-metric series** (revision velocity, segment-revenue acceleration, commodity-price turn, or margin decoupling, per archetype); the **earnings surprise / SUE**; **positioning** (insider net, congressional, activist 13D/G, short interest, COT for the cyclical sleeve, and the **options-activity signal** from the Step-5b chain — put/call by volume and open interest, IV/skew; FMP 13F institutional is off-plan → SEC EDGAR or omitted); scenario **price targets** discounted off the run-level risk-free rate (methodology recorded); a **price-action confirmation read** (relative strength vs the market / sector and proximity to a multi-year base breakout, from Stooq deep history — a confirmer, not a trigger); and the **two derived reads selection leans on** — the **narrative-vs-reality ratio** (estimate-revision pace vs multiple change, with the **operating-reality-vs-price** fallback for thinly-covered names whose estimates are absent or stale — both defined in [trade-opportunities.md §The two non-negotiables](trade-opportunities.md#the-two-non-negotiables)) and the **forensic flags** (margin compression while revenue accelerates, net-income-vs-operating-cash-flow divergence, receivable / inventory build outpacing sales, restatement / auditor-change history, plus `financial-scores` Altman Z + Piotroski). A **tradability flag** (Amihud-style illiquidity + days-to-cover) rides alongside, so a small / illiquid name is flagged rather than silently excluded. For a **carried-forward candidate** (one with a prior opportunity record), the engine additionally computes its **since-flagged performance read** — running return since it became an opportunity (`became_opportunity_at`; absolute and vs sector / market), maximum drawdown over that window, and leading-metric-continuation state, from the same Stooq history (the identical primitive Step 7's scorecard uses — [trade-opportunities.md §Outcome learning](trade-opportunities.md#outcome-learning-calibration)) — so scoring can weigh how the idea has actually done since the job first called it. The forward targets are a **provisional scenario menu** at this point. The model interprets these reads; it never invents one ([trade-opportunities.md §The pipeline](trade-opportunities.md#the-pipeline)).

### Step 5d: Bounded Web Research

**Type:** Local-model call (122B, thinking) + API retrieval (the web tool), **looped**. This is the only stage in the loop that itself loops.

The reasoner sets a small **agenda** that works the three research lenses of [trade-opportunities.md §The research method](trade-opportunities.md#the-research-method), **one topic at a time** — **each topic a separate model call and research loop over a clean context** (the dossier facts plus that topic's own questions; no other topic's findings are fed in), so the local model reasons over a tight prompt, never a sprawling one:
- **leading-metric validation** — confirm the engine's leading metric is real, countable, dated, and inflecting from a third-party source (the mandatory anchor);
- **macro / thematic-fit** — which theme the name rides and where it sits on the S-curve (against the Step-3b thematic map), pure-play vs enabler, bottom-up TAM, and the economist's front-running indicators (capex commentary, book-to-bill, freight, the cycle) the structured feeds don't carry;
- **investor-judgment** — the driving **narrative and market sentiment** (how much of the price is emotion about *what might come* vs present fundamentals), management / capital-allocation quality, and the pre-consensus tells — seeded by the candidate's symbol-scoped FMP news (`news/stock` — the press-release feed is off-plan) and deep-read on the open web;
- **pattern / case-study** — check the candidate against the early-winner library and the look-alike failures (the third of the five lenses' research reads — [trade-opportunities.md §The lenses](trade-opportunities.md#the-lenses)): the recurring early tells (a high-margin segment compounding inside a lower-margin whole, an oligopoly's supply discipline turning, a usage metric inflecting ahead of revenue) against the red-flag set the selection gate enforces;
- **external corroboration** the feeds can't give — customer / hyperscaler capex, supply discipline, transcript backlog / TAM / inflection language, and **DRAM/NAND ASP direction**;
- **mandatory — the contemporaneous bear case** — why the name might fail (the winning traits also rode the famous failures down).

The orchestrator — not the model — owns every request, so the loop has a hard ceiling: **per-topic depth ≤2 (≤3 passes/topic)** and a **per-candidate fetch + wall-clock budget that binds first**, spent in topic-priority order (leading-metric and bear case first), fail-soft on exhaustion — the lowest-priority topics drop to a recorded gap rather than overrunning the run. Grounded throughout by the engine's quant and value-creation numbers so research builds the case around the leading metric rather than substituting a story for it. The full loop and its bounds are in [web-research.md](web-research.md).

#### Local-model call — Per-candidate research (Qwen3.5-122B, thinking)

**Model.** The resident 122B reasoner in thinking mode, requesting `web_search` / `web_fetch` tool calls the orchestrator executes (SearXNG-primary, Tavily fallback; SSRF-guarded; untrusted page text inserted as quoted evidence, never as instructions). **One call per agenda topic** — topics do not share a context.

**Prompt — input.** The candidate's dossier facts, its archetype and computed leading-metric reads, and **that topic's questions only** — a clean context per topic. Within a pass the model reasons over the fetched, readability-extracted page text and an **append-only evidence ledger** (each extracted claim + its source URL / timestamp); there is **no in-loop re-distillation of findings** — the heavy consolidation is deferred to the Step-5e distillation, so research is never planned over already-distilled, lossy notes.

**Returns.** The topic's **full findings response**, preserved whole (with its evidence-ledger entries), plus any **follow-up proposal** (a structured field the orchestrator decides whether to spend) and any **material forward fact** flagged for the Step-5f refinement. The **bear-case topic is non-optional** — the candidate cannot reach scoring without a stated, sourced bear case (the winning traits also appeared in the famous failures). Every topic's full response flows intact to distillation — nothing is summarized away in between — where it is consolidated in a single pass or, when the per-candidate research is large, **hierarchically** (a tier-1 distillation per topic-tree → a final reduce, Step 5e).

### Step 5e: Distillation

**Type:** Local-model call(s) (122B, non-thinking; the optional 35B fast tier if resident) — a single pass, or **hierarchical** (tier-1 per topic-tree → a reduce) when the per-candidate research is large. Consolidation, not new reasoning.

The reasoner in non-thinking mode consolidates the topics' **full findings responses** into the compact object the scoring stage reads. This is the *only* place research is condensed before scoring, and it always consolidates **full-context research, never already-distilled notes** ([web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management)). The orchestrator picks the shape **deterministically** from the accumulated evidence-ledger size against the call's input budget — a **single pass** for ordinary candidates, or **map-reduce** (a **tier-1** distillation per topic-tree → a **tier-2 reduce**) when the research is large, with tier-1 outputs **structured and field-preserving** (per-lens claims + sources + confidence + any internal-tension flag) so nothing the reduce needs is lost — the shared primitive in [web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management).

Either way the **cross-lens contradiction check lives at the consolidating pass that first sees all five lenses** — the single pass when flat, the **tier-2 reduce** when hierarchical (it cannot run at tier-1, which sees only one topic-tree): it flags which lenses disagree (with a **severity**), the thesis's **key falsifiers**, and any stale / conflicted source, so the check still costs **no dedicated model call** ([trade-opportunities.md §Reconciling the lenses](trade-opportunities.md#reconciling-the-lenses-the-contradiction-check)). Hierarchical distillation is **bounded** by a per-candidate sub-distillation cap and the run's wall-clock, and the chosen shape + tier count are **logged to the audit**.

#### Local-model call(s) — Distillation (Qwen3.5-122B, non-thinking)

**Model.** The same resident 122B in non-thinking mode by default (no model-swap cost); the fast 35B tier is a benchmark-gated option ([local-models.md §The model roster and per-task routing](local-models.md#the-model-roster-and-per-task-routing)).

**Prompt — input.** *Single pass:* the **full findings response from every topic** (including the bear-case topic) plus the append-only evidence ledger (claims + sources), **and the engine's Step-5c computed reads — the quant composite, the value-creation read, the narrative-vs-reality ratio, and the forensic flags — as the engine lenses the research findings are reconciled against, so the contradiction check can span all five lenses (per the *all-five-lens* rule above)**. *Hierarchical:* each **tier-1** call gets one topic-tree's complete findings + that tree's ledger entries (no engine reads — tier-1 sees only one tree); the **tier-2 reduce** gets the tier-1 structured outputs **plus those same engine reads**, each carrying the citations the contradiction check and downstream gate depend on.

**Returns.** A single schema-validated **distilled findings object** for scoring — the leading-metric validation, the narrative/sentiment read, the forward-opportunity read, and the bear case, each cited; a **typed cross-lens contradiction read** (which lenses disagree, with a **severity**), the thesis's **key falsifiers** (the specific monitorable conditions that would break it), and any stale / conflicted source — produced by the final consolidating pass (the single pass, or the tier-2 reduce), so scoring and the gate reason over *reconciled*, not merely concatenated, findings; surfacing any **material forward fact** the structured feeds lacked as a typed **`research_forward_assumption`** (schema in [portfolio-workflow.md §Step 6d](portfolio-workflow.md#step-6d-distillation)) — so only a structured, sourced, numeric claim (never loose prose) can reach the engine's target refinement; and, when research validated an **engine-unscored leading indicator** — a `research_metric`-class signal **absent from the Step-5c quant composite / leading-metric series** (so distinct from a structured anchor the engine already scored) — a typed **`validated_leading_indicator`** `{ metric, value / level, direction, as-of date, source URL, confidence }`, the **only** field a Step-5g conviction *raise* may cite and the one Step 5h validates against, so the raise is never prompt-only.

### Step 5f: Deterministic Target Refinement

**Type:** Computed (the engine). No model.

If distillation produced a typed **`research_forward_assumption`** (Step 5e — a guidance figure, a signed-contract value, a commodity / ASP turn, each with value, units, as-of date, source, and confidence), the **engine — not the model —** recomputes the affected scenario target with it as an explicit, **logged** assumption. A malformed, unsourced, or non-numeric claim is **rejected** (it cannot move a target), and a fact that **conflicts** with a structured feed is resolved by the assumption's declared conflict-handling rule. So the number stays engine-computed while the forward view reflects what research learned. The backward-looking sub-scores and derived reads are untouched; absent a valid assumption, the Step-5c targets stand. This is the same refinement contract Portfolio Analysis uses ([portfolio-workflow.md §Step 6e](portfolio-workflow.md#step-6e-deterministic-target-refinement)).

### Step 5g: Scoring & Gating

**Type:** Local-model call (122B, thinking). The opportunity-authoring call — but it does **not** pick the risk tier or the horizon (those are deterministic, Step 5h).

The reasoner interprets the computed analysis and the distilled research into the candidate's **opportunity record**, under explicit discipline: **score the conjunction, never a single signal** (base rates are brutal — most stocks underperform Treasuries over their life, and the winner traits recur in losers); **require the leading-metric anchor plus external validation**; **apply the narrative-vs-reality ratio**; treat **price action (relative strength / base breakout) as a confirmation overlay** that adjusts conviction but never substitutes for the leading-metric anchor; for a **carried-forward name**, weigh its **since-flagged performance** (Step 5c) the same disciplined way — a gain unmatched by leading-metric progress reads as building multiple-unwind risk and **caps** conviction (the asymmetry has narrowed), a drawdown with the metric intact reads as improved asymmetry rather than a reason to abandon, and a gain **matched** by leading-metric progress is held **neutral — never boosted**: the confirming metric is already scored directly through the engine's leading-metric series and quant composite (Step 5c), so crediting the since-flagged gain on top would **double-count the metric** and re-admit price as a conviction input — the exact reflexive momentum-chasing this read exists to police. The since-flagged read is therefore **cap-only by design** — it can hold or lower conviction, never raise it — so the job's own realized track record sharpens the re-score without becoming momentum-chasing; and **resolve the cross-lens contradiction** the distillation surfaced — a candidate whose loud lens (usually the thematic narrative) is contradicted by a weak value-creation, management, or unit-economics read is **capped, not promoted**, its conviction shaped down by the unreconciled disagreement rather than averaged out. It reads every number from the engine rather than inventing it. The scoring runs the candidate down the archetype's **track** — *proven-economics* (trailing returns on capital + a margin of safety) or *emerging-economics* (a forward TAM × penetration × margin model clearing a return hurdle) — but **both tracks pass through the same moat / management / price-asymmetry gate** ([trade-opportunities.md §The research method](trade-opportunities.md#the-research-method)), so a strong-numbers name and a revolutionary one are judged on one spine without collapsing into pure value or pure momentum. **Conviction is bidirectional, but asymmetrically gated.** Every discipline lever above — narrative-vs-reality, the cross-lens contradiction, the since-flagged read — can only *cap or lower* it. A conviction **raise** is admitted only on the **emerging-economics track** (disruptor, ai-infra) and only when the distilled research validated a leading indicator **absent from the Step-5c quant composite / leading-metric series** — a `research_metric`-class signal with no structured feed (DRAM/NAND ASPs, channel checks, transcript backlog), carried as the typed **`validated_leading_indicator`** (Step 5e). The test is *whether the engine already scored it*, **not** whether it is the mandatory leading-metric anchor: a structured anchor already in the 5c composite cannot also raise conviction (that is the double-count the since-flagged read guards against), while a research-only signal the engine could not score can. The raise is **bounded (≤ one level), cited to that field, and logged**. Price action, narrative, and any metric already in the Step-5c composite can **never** raise conviction — the since-flagged read stays cap-only precisely because its metric is already scored, and the same no-double-count / anti-reflexivity guard governs here ([trade-opportunities.md §Balancing the revolutionary against the proven](trade-opportunities.md#balancing-the-revolutionary-against-the-proven)).

#### Local-model call — Scoring & gating (Qwen3.5-122B, thinking)

**Model.** The resident 122B in thinking mode; schema-constrained output.

**Prompt — input.** The engine's computed analysis (archetype-weighted sub-scores, the refined scenario targets with exposed methodology, the narrative-vs-reality and forensic reads, the price-action confirmer, the positioning / options signal, and — for a carried-forward name — the **since-flagged performance read** (Step 5c) as context); the distilled research findings including the mandatory bear case **and the cross-lens contradiction read / key falsifiers**; the candidate's archetype and surfacing signals; the house view and investor profile; and any prior opportunity record for this name. The **absolute street opinions** (consensus target level, current rating consensus, FMP's ratings snapshot) are presented as *evidence to weigh against the engine's own read*, not as numbers to adopt.

**Returns.** The schema-validated **opportunity record** ([trade-opportunities.md §The opportunity](trade-opportunities.md#the-opportunity)) — directional thesis, **detection mode** (early / continuation), the **leading operating metric** and its trend, the catalyst, **conviction** as a **decomposed triple** so the raise is app-checkable, not a single opaque value — **`base_conviction`** (capped by narrative-vs-reality **and the cross-lens contradiction read**, lowered on thin evidence, abstaining as `insufficient-evidence` below the evidence floor), an **optional `conviction_raise`** (emerging-economics track only — a `+1` level, the cited **`validated_leading_indicator`**, and a rationale), and **`final_conviction`** (= `base_conviction`, or one level higher on a valid raise), with **Step 5h re-deriving `final_conviction` deterministically** from the base ± the validated raise, never the model's bare number, the **narrative-vs-reality read**, the **forward outlook** (the engine's base-case scenario target and bear / bull range from Steps 5c / 5f over the engine's fixed **end-of-year ~12-month** window — distinct from the matrix horizon — methodology exposed; the forward read, surfaced distinct from the backward engine anchor), the **mandatory bear case**, the **key falsifiers** (the monitorable conditions that would invalidate it), the entry consideration, and any tripped **risk / forensic flags** — plus a proposed **carry-forward status** (`new` / `still-valid` / `invalidated` — no "target met" exit; an `invalidated` name leaves the matrix for the archive) with each status / conviction move attributed to an input that changed (Step 7 validates the attribution). The model proposes the record; it does **not** assign the risk tier or horizon.

### Step 5h: Deterministic Risk-Tier, Gate Validation & Checkpoint

**Type:** Computed (app layer). No model.

This step is an **app-layer validator and tier-assigner**, not just a recorder:
- the **risk tier** (high / medium / low) is assigned **deterministically by rule** from the engine's measurable inputs (profitability, market cap, liquidity, volatility, leverage, drawdown, event exposure — archetype-informed), and the **horizon** (short / mid / long) from **catalyst timing + archetype** (a dated near-term catalyst → short; a multi-year secular / quality-compounder archetype → long; else mid) — *not* the risk-input list, which carries no time signal — so the model's conviction never sets the cell ([trade-opportunities.md §The opportunity space](trade-opportunities.md#the-opportunity-space), [§Starting parameters](trade-opportunities.md#starting-parameters-calibratable)). Tier + horizon → the matrix cell;
- the **forensic / risk gate** is enforced in the app, not left to the model: a candidate tripping the forensic flags (Step 5c), or whose move is mostly multiple-expansion (a `hype` narrative-vs-reality read with no leading-metric anchor), **or carrying a high-severity unresolved cross-lens contradiction flagged at distillation** (a strong narrative over a failing value-creation / forensic / unit-economics read — [trade-opportunities.md §Reconciling the lenses](trade-opportunities.md#reconciling-the-lenses-the-contradiction-check)), is **capped or excluded** rather than promoted — a hard rule the model's enthusiasm can't override;
- a **carried-forward candidate** (a re-surfaced existing opportunity) whose Step-5g status is **`invalidated`** is **held out of the matrix and flagged for archival** here — the same gate that drops `insufficient-evidence`, so an invalidated re-surfacer never reaches Step-6 assembly (Step 6 only orders/dedups the gated survivors); Step 7 then moves it to the archive (this deep pass is the only archival path — [trade-opportunities.md §Archived opportunities](trade-opportunities.md#archived-opportunities));
- a candidate that abstained as `insufficient-evidence` (below the **archetype-aware evidence floor** — missing the floor-bearing price + validated leading metric, or statements-or-substitute per archetype, or carrying stale / conflicting data — see [trade-opportunities.md §Evidence floor](trade-opportunities.md#evidence-floor)) is held out of the matrix rather than promoted as a low-conviction guess;
- a model-authored **conviction raise** is validated against the **raise invariant** and `final_conviction` is **computed by the app**, not trusted from the model: a `conviction_raise` is honored only on the **emerging-economics track**, only **≤ one level**, and only if it **cites a `validated_leading_indicator`** present in the Step-5e distilled findings (a `research_metric` signal absent from the Step-5c composite) → `final_conviction = base_conviction + 1` (capped at the ceiling); a raise failing any clause is **dropped, so `final_conviction = base_conviction`** (the well-defined un-raised level), the drop logged to the audit. This runs **per candidate, here** — so even a **debut** opportunity, which Step 7's carry-forward continuity check never sees, cannot enter the matrix on an invalidly raised conviction (the per-candidate analog of Portfolio's Step-6g validation);
- the surviving candidate is **checkpointed** so the run can resume here.

## Step 6: Matrix Assembly & Completeness Validation

**Type:** Local-model call (122B — ranking + dedup proposal) + Computed (app-layer completeness validation). The risk tier and horizon are already deterministic (Step 5h), so each survivor's **cell is predetermined** — Step 6 does **not** choose *which* opportunities appear, only their **order within a cell** and which near-duplicates **collapse**. Because removing the per-cell output cap makes "every gate-clearer appears" a hard guarantee, completeness is enforced by the **app layer, not the model's good behavior** (the same app-validator stance as Step 5h and Step 7): every Step 5h survivor must appear in its assigned cell, **or** be explicitly collapsed into a named peer with a recorded dedup reason. A survivor that is neither present nor accounted-for is a **validation failure**, so the model cannot quietly drop a name the way an output quota once could.

#### Local-model call — Per-cell ranking & dedup proposal (Qwen3.5-122B)

**Model.** The resident 122B reasoner; schema-constrained output.

**Prompt — input.** All gated opportunity records from Step 5 with their **deterministically-assigned cells**; the house view and investor profile; and, for context, the count of candidates competing for each cell.

**Returns.** Per cell, a **conviction ranking** of that cell's gated survivors, plus any **dedup-collapse proposals** — each naming the merged-away candidate, the peer it collapses into, and the reason (near-identical thesis / shared leading metric / shared catalyst). The model **cannot drop or hide a survivor** and **cannot move one to a different tier or horizon** (both deterministic) — it only **orders** and **proposes merges**.

#### Computed — Completeness validation & matrix assembly

**Type:** Computed (app layer). No model. The app assembles the final **3×3 matrix** from the deterministic cell assignments and the model's ranking, applying each accepted dedup collapse (recorded in the audit with its reason — [trade-opportunities.md §Storage and display](trade-opportunities.md#storage-and-display)). It then **validates completeness**: every Step 5h survivor is either listed in its cell or recorded as a collapsed peer — a survivor absent from both **fails the run's matrix validation** rather than vanishing silently. There is **no per-cell output cap**: the gates set a cell's count, not a quota, so a cell holds as many or as few validated ideas as qualify. **A cell may be empty** when nothing qualified — honest, not a failure — and the matrix never pads itself to fill them ([trade-opportunities.md §The pipeline](trade-opportunities.md#the-pipeline)); equally, a rich cell that genuinely earned many anchored ideas is never trimmed to a fixed number.

## Step 7: Continuity Check & Carry-Forward

**Type:** Computed (app layer) — an **app-layer validator**, not just a recorder. No model: the re-surfacers' deep passes already ran in the Step-5 loop, so this step only reconciles their verdicts, cheap-sweeps the rest, and archives.

Prior opportunities (from the Step-2 prior run) are reconciled with this run's matrix, mirroring the discipline of Portfolio Analysis's what-changed audit ([portfolio-workflow.md §Step 6g](portfolio-workflow.md#step-6g-continuity-check-and-checkpoint)) at the opportunity-list level. DTO splits the carried matrix **by cost**, and the dividing rule is that **only a deep pass can archive** ([trade-opportunities.md §Archived opportunities](trade-opportunities.md#archived-opportunities)):
- every opportunity carried forward keeps its identity and gets an **updated status** (`new` / `still-valid` / `invalidated`); an **addition or an `invalidated` departure must be justified by what changed**. There is **no "target met" exit** — an opportunity persists while it still clears re-evaluation, however far its price has run; only a **deep** re-evaluation that judges it no longer an opportunity ends it (it then leaves the matrix for the **archive** — [trade-opportunities.md §Archived opportunities](trade-opportunities.md#archived-opportunities));
- **the re-surfacers were already deep-re-evaluated in Step 5.** An existing opportunity that re-surfaced through discovery (Step 3) and won leftover deep-research budget (Step 4) ran the **full Step-5 loop as a carried-forward candidate** (its prior record carried in under the age-banded `continuity_weight`, Step 5b), so its fresh verdict is already in hand here — Step 7 does **not** re-invoke the loop. That Step-5 deep pass is the **only path in DTO that can judge a name `invalidated` → archive** (a qualitative erosion the cheap read can't see, or a deterministic exhausted-upside / continuation-failure finding *confirmed* under fresh research). A still-valid result reconciles into the matrix; an `invalidated` one was **held out at Step 5h** and is **moved to the archive here**. Every such ticker is in the **run-scoped deep-research set**, so the cheap sweep skips it;
- **every other live opportunity gets the cheap re-derivation, which never archives.** For each carried opportunity **not** in the deep-research set the engine (off-budget, no model) re-pulls price + estimates, **re-derives the scenario targets** (Steps 5c / 5f, carrying the last deep pass's persisted **`research_forward_assumption`** — **decayed to structured-only past its freshness window** so a stale prop can't keep a target inflated), **re-runs the entry asymmetry / valuation gate** against the live price, and refreshes the since-flagged read — but it **never archives and never re-rates a survivor**. When it detects **exhausted upside** (re-derived base-case forward return no longer clearing the entry gate), a **continuation-failure alarm** (estimate revisions rolling over, a beat-and-raise streak breaking, shipments diverging below sell-through), or a **tripwire** (a forensic flag newly tripping, a since-flagged gain diverging hard from leading-metric continuation, the narrative-vs-reality read crossing into `hype`, a margin-of-safety compression near the exhaustion line, or a drawdown breach), it raises the deliberately **high-bar attention warning** on the card — *consider a Deep Audit* — rather than acting. A deep pass (a later re-surface, or an ATO Deep Audit) clears it. Both readings are **fail-soft**: a *missing* input holds the opportunity on its last verdict, never an escalation; only an *affirmative* signal **confirmed under a deep pass** ends an opportunity;
- each status / conviction move the Step-5g record claims as **external** must resolve to a concrete entry in the engine's deterministic input delta (this run's metrics / positioning / price vs the prior run's stored values), a source-backed research finding, or the logged `research_forward_assumption`; an attribution that resolves to nothing is **downgraded to a self-correction** (or fails schema validation), so the model cannot launder a no-new-facts swing as "the thesis changed";
- **deterministic outcome labels** are computed for every prior opportunity old enough to have a forward window — forward return at 1 / 3 / 6 / 12 months **vs its sector and the market**, **maximum drawdown** over the window, **whether the leading metric continued**, and (for losers) a **decision-tree failure mode** (`leading-metric-rollover` / `multiple-unwind` / `forensic-materialization` / `thesis-played-out` [an internal-calibration label for a winner, **not** a matrix status — hitting a target is not an exit] / `market-beta`) — all **engine-computed** from price history and the input delta, never model-judged, and tested against the opportunity's stored **key falsifiers**. They anchor on the opportunity's first-surfaced run (= the `became_opportunity_at` run — its first matrix entry, recoverable from the carry-forward identity), feed the **planned** archetype-weight / gate calibration, and — guarding against overfitting a handful of outcomes — **do not yet auto-adjust weights**; early runs stay shadow / calibration ([trade-opportunities.md §Outcome learning](trade-opportunities.md#outcome-learning-calibration));
- the **continuous since-flagged read** is refreshed for every carried-forward opportunity — running return since it became an opportunity (`became_opportunity_at`; absolute and vs sector / market), maximum drawdown, and leading-metric-continuation — from the same daily Stooq reconstruction, **live from the idea's first subsequent run** (its first continuity pass after being added — a debut pick has no elapsed window yet) rather than only once a horizon window matures, so the matrix display ([trade-opportunities.md §Storage and display](trade-opportunities.md#storage-and-display)) and the Step-5g re-score of a carried-forward name read the same numbers (the matured-window labels above attach to it as they elapse);
- the **opportunity graph is reconciled in the same pass** — this run's picks link to their matrix entry, this run's worthy-but-unpicked names are added or refreshed as **watchlist** nodes (with their leading metric + key falsifiers), and nodes whose falsifiers tripped or whose carry horizon elapsed are **retired** (Step 3c) — so the discovery memory stays current and bounded alongside the validated matrix ([trade-opportunities.md §Discovery memory](trade-opportunities.md#discovery-memory-the-opportunity-graph)).

Output stays **firm and does not churn** between runs absent hard data ([thesis-continuity.md](thesis-continuity.md)) — an idea persists with an updated status rather than silently reappearing or vanishing.

An `invalidated` opportunity is **moved to the archive** rather than dropped — the job's price-tracked record of departed picks, bounded to the most recent 100 ([trade-opportunities.md §Archived opportunities](trade-opportunities.md#archived-opportunities)). The archive is **distinct from the opportunity-graph `retired` nodes** above: the graph retires *upstream discovery hypotheses* (worthy-but-unpicked watchlist names), while the archive holds *downstream validated picks* that left the matrix. An archived pick keeps only the **price-derived part** of the since-flagged read refreshed each run — return vs sector / market and drawdown, **not** leading-metric continuation (which would need a structured re-pull) — the same stateless reconstruction, no deep research, no model call; if a later run **independently re-discovers** it, it is a **fresh** opportunity (new `became_opportunity_at`, removed from the archive) and none of the archived record influences the new one.

## Step 8: Holdings Cross-Reference

**Type:** Computed (app layer) — a deterministic post-step. No model.

After selection and continuity, the application flags any opportunity that **overlaps the user's current holdings** (owned / not-owned), reading only the holdings list from the most recent Schwab pull — **never** the Portfolio Analysis memory partition. Crucially this runs *after* discovery and selection, so holdings **never influence which opportunities are found or chosen** ([trade-opportunities.md §Holdings cross-reference](trade-opportunities.md#holdings-cross-reference)); the job stays genuinely independent of the account, surfacing an already-held idea as such only for the user's awareness.

## Step 9: Persist Run and Audit, with Memory Embeddings

**Type:** Computed (persist the matrix and audit record) + Local-model call (embeddings for continuity).

The application persists the run: the **3×3 matrix** of opportunities, and an **audit record** that makes the run traceable — the report(s) and sources used with retrieval timestamps, the **discovery and screening inputs** (which screens / themes surfaced each candidate), the distilled findings, the computed signals and the **bear case** behind each pick, each pick's conviction decomposition (`base_conviction` + any `conviction_raise` with its cited `validated_leading_indicator` + the app-computed `final_conviction`), the **input delta and the carry-forward attribution**, the **matrix dedup-collapse decisions** (each merged-away candidate, the peer it merged into, and the reason — so the Step-6 completeness guarantee is auditable), the **deterministic outcome labels on prior opportunities** (matured-window forward return vs sector / market, drawdown, leading-metric continuation, failure mode, plus each carried-forward idea's continuous since-flagged read — [trade-opportunities.md §Outcome learning](trade-opportunities.md#outcome-learning-calibration)), the price-target methodology including its discount-rate assumption and any research-sourced forward assumption (with source), the model ids and quantizations, the prompt/schema version, and degraded-input flags ([trade-opportunities.md §Storage and display](trade-opportunities.md#storage-and-display)). The **opportunity graph** (the discovery memory — hypotheses with their economic value-chain traces, and the watchlist nodes with each one's leading metric, falsifiers, and status) is persisted alongside, carried into the next run's Step 2. The **archive of departed picks** is the persisted third store — the most recent 100 opportunities a `failed-reevaluation` removed from the matrix, each a frozen verdict snapshot (distinct from both the matrix and the graph), with the since-flagged read recomputed statelessly rather than stored ([trade-opportunities.md §Archived opportunities](trade-opportunities.md#archived-opportunities)). Retention keeps the last N runs ([storage.md](storage.md)).

#### Local-model call — Run-result embeddings (Qwen3-Embedding-4B, fixed)

**Model.** The fixed local embedder — vectorization only.

**Prompt (input text).** Each opportunity's record summary, embedded individually; **plus each prior opportunity's outcome label** (a durable learning), so realized outcomes are semantically recallable for the planned calibration.

**Returns.** Vectors stored in the **Trade Opportunities** memory partition (the job namespace), so a later run of this job can semantically recall the relevant prior analysis **and outcome history** for a name ([local-models.md §Run history and continuity](local-models.md#run-history-and-continuity)). Best-effort: a failed embedding costs the memory row, never the persisted run.

## Step 10: Generate Trade Opportunities Page and Update UI

**Type:** Computed (frontend). No model.

The **Trade Opportunities page** renders the 3×3 matrix — high-, medium-, and low-risk sections, each containing short-, mid-, and long-term ideas — each cell listing its opportunities with archetype, directional thesis, leading metric, catalyst, conviction, the **forward outlook / price prediction** (base-case scenario price target and bear / bull range over the EOY ~12-month window), narrative-vs-reality read, entry consideration, bear case, status, its **`became_opportunity_at` and `last_deep_researched_at` dates**, the **since-flagged performance read** for carried-forward ideas (return since it became an opportunity, vs sector / market, with a compact running curve and maximum drawdown), and the owned/not-owned flag, with empty cells shown as empty. A separate **Archived opportunities** view lists departed picks with their frozen record, departure date, and live since-flagged return — price-only, **no forward prediction** ([trade-opportunities.md §Archived opportunities](trade-opportunities.md#archived-opportunities)) ([interface.md](interface.md)). The page also carries the **two jobs' controls** — a **Discover** button (DTO) and a selection-gated **Audit** button forking to **Quick Audit** / **Deep Audit** (with a confirmation on a large Deep-Audit selection) — and renders each card's lifecycle affordances: the **selection control** (plus select-all / deselect-all), an amber **Consider Deep Audit** badge when the attention warning is set, a green **Deep-researched today** badge when `last_deep_researched_at` is the current local-timezone day, and a quiet **Research stale** badge when the last deep pass is older than the ~4-week freshness window. The **price-derived** since-flagged read on each card is **re-derived at render** from a **per-ticker Stooq daily-bar cache** (refreshed fail-soft only after 8 PM ET and not within the prior 24 h — keyless, no per-render fetch) as a freshness floor, so a card's performance number is current even when no job has run since the last matrix write ([trade-opportunities.md §Storage and display](trade-opportunities.md#storage-and-display)); leading-metric continuation and the live FMP-quote-based gate read refresh only when a job runs. While a job ran, the run tracker replaced the page (latest-run-only); on completion the page shows the persisted matrix. A **run is never a report**: the matrix appears only on persisted success, so a cancel or failure removes nothing ([run-tracking.md](run-tracking.md)).

## ATO: the audit flow

**Audit Trade Opportunities** is the user-directed maintenance job ([trade-opportunities.md §The two jobs](trade-opportunities.md#the-two-jobs)). It has no discovery: the user selects one or more **existing** matrix opportunities and chooses **Quick Audit** or **Deep Audit**; the job re-evaluates exactly that selection, reusing DTO's stages, and re-renders. It holds the **same single global run slot** and clears the **same presence gate** as the other local jobs (daemon + roster configured, Schwab connected). The fork differs only at the **run-gate connectivity** layer: **Deep Audit** clears the full Step-1 gate (daemon reachable + roster pulled, since it makes model calls), while **Quick Audit** is engine-only and **skips the daemon-connectivity check**, so it runs even when the daemon is configured-but-down; its cheap re-derivation reads no Schwab data (the options signal is held out of the grade), though the Schwab connection stays a presence precondition (see step 2).

1. **Gate & load (Steps 1–2, reused).** The execution gate is uniform on **presence** and fork-aware only on **run-gate connectivity** (above); the **Step-2 shared-context load** runs as in DTO (house view, profile, run-level FRED / CFTC / CBOE, the prior matrix and opportunity graph — Quick Audit loads only the subset its engine pass needs, e.g. the FRED risk-free rate). The selection is the work-list; **no discovery feeders run**.
2. **Quick Audit — the engine-only cheap re-derivation (no model).** For each selected opportunity the engine runs the **same cheap re-derivation Step 7 applies to the DTO matrix tail**: re-pull price + estimates, re-derive the scenario targets (decaying a stale `research_forward_assumption`), re-run the entry asymmetry gate, refresh the since-flagged read. It **never archives** — an exhausted-upside, continuation-failure, or tripwire reading raises the **attention warning** ([trade-opportunities.md §The opportunity](trade-opportunities.md#the-opportunity)) on that card. This path makes **no model call**, so it cannot fail on research and it is fast — and because it needs no model at runtime, it **skips the run-gate daemon-connectivity check**, so a Quick Audit runs even when the Ollama daemon is **configured-but-down** (it still needs the daemon and Schwab *configured* — the uniform presence gate). Its cheap re-derivation reads no Schwab data (the **options-activity signal is held out of the grade** and isn't an input), needing only FMP / Stooq / FRED at runtime. (If the options signal is ever calibrated *into* the grade, that's a deep-pass input regardless, so this stays correct.)
3. **Deep Audit — the full Step-5 loop.** For each selected opportunity the job runs the **Step-5 per-candidate validation loop** (5a archetype → 5b dossier, with the age-banded `continuity_weight` over the prior record → 5c engine → 5d research → 5e distill → 5f refine → 5g score → 5h validate). It rewrites the model-authored fields, writes a fresh `research_forward_assumption`, stamps `last_deep_researched_at`, and **clears any attention warning**. It is the **only** ATO path that can judge a selected name `invalidated` → archive. A **large selection prompts a confirmation first**, since it runs the loop per name (potentially long), in the spirit of the SearXNG pre-run consent ([web-research.md](web-research.md)).
4. **Continuity, persist & render (Steps 7–10, reduced).** The audited records are reconciled into the matrix with the same what-changed attribution discipline (a deep-pass status / conviction move must trace to an input delta, a research finding, or the logged assumption); an `invalidated` Deep-Audit result moves the name to the **archive**. The opportunity graph is **not** rebuilt (no discovery ran), the holdings cross-reference (Step 8) re-runs only over the touched names, and the run + audit record persist (Step 9) and the page re-renders (Step 10) exactly as for DTO. ATO's depth is bounded by the **user's selection**, not the DTO deep-research budget.
