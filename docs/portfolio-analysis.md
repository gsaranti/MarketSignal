# Portfolio Analysis

Portfolio Analysis is a **local, on-demand job** that grades every holding in the user's portfolio and recommends an action for each, grounded in the holding's fundamentals, fresh web research, and the current Market Signal house view. It runs entirely on local models (see [local-models.md](local-models.md)) and sources holdings — and the per-stock options signal — from a **connected Charles Schwab account, which is required to run the job** (manual import supplements the holdings but does not replace the connection; see [schwab-integration.md](schwab-integration.md)).

It is deliberately **prescriptive** — it issues buy/trim/hold/sell-style actions and explicit price targets — which is the opposite of the Market Signal Report's no-buy/sell stance ([report-structure.md](report-structure.md)). The report forms the *house view*; this job applies that view to the specific positions the user holds.

## Triggering

The job is manual and runs in two user-controlled steps: **pull holdings**, then **run analysis**. Only one run happens at a time across the whole app — a single global run slot serializes the report and both local jobs (see [local-models.md §Failure posture](local-models.md#failure-posture)). While it runs it streams into the run tracker with **per-holding progress**, the same observability seam the report uses ([run-tracking.md](run-tracking.md)).

## Asset eligibility

A Schwab portfolio holds more than equities — ETFs, mutual funds, options, fixed income, and cash. The equity-centric pipeline (FMP fundamentals, company financials) applies cleanly only to individual stocks, and in reduced form to funds. Each position is classified by asset type before analysis:

- **Stocks** — the full pipeline and verdict.
- **ETFs / funds** — a reduced verdict (no single-company financials; graded on strategy / **exposure**, valuation, and the house view). On the current FMP plan ([data-sources.md §FMP — current paid-plan tier audit](data-sources.md#fmp--current-paid-plan-tier-audit)) **`etf/holdings`, `etf/asset-exposure`, and the `funds/disclosure*` filings are off-plan**, so constituent-level look-through is lost. ETFs are sourced from the **on-plan** `etf/info` (expense ratio, AUM, NAV, asset class, mandate) + `etf/sector-weightings` / `etf/country-weightings` — an **exposure-tilt** read (sector / country, not single names). Single-name **concentration** and direct-vs-fund **overlap** degrade to that exposure level unless reconstructed from **SEC N-PORT** (keyless but heavy, ~60-day lag — an optional enhancement, never floor-bearing). A future **issuer-holdings adapter** — major ETF sponsors publish current constituent files fresher than N-PORT's lag, at the cost of per-issuer, messier parsing — is the practical way to buy back single-name look-through without a higher FMP tier; it is **planned, not built**, and like N-PORT would only enrich the fund path, never gate it. **Mutual funds degrade hardest**: the `etf/*` weighting endpoints don't cover them, so a mutual fund grades on **quote / NAV + profile + the house view** only (or SEC N-PORT). `etf/asset-exposure` (reverse "which ETFs hold this name" lookup) is off-plan and dropped — it was only an optional per-direct-equity cross-check.
- **Options, fixed income, cash, and unsupported types** — marked **not rated**, with a short reason, and excluded from grading. Cash and buying power still feed the investor profile and the portfolio roll-up — and a **material** not-rated position (a sizable fixed-income sleeve, a large options position) still contributes its **risk and exposure** to the whole-book aggregates ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)), so the portfolio's risk posture reflects the real book rather than only the rated sleeve; it simply carries no per-position grade.

A not-rated position never receives a fabricated grade; the eligibility decision is explicit and shown in the UI.

For a **fund** holding the engine runs a **reduced computation** in place of the equity pipeline: **expense drag** (the expense ratio as an annual return headwind), **exposure tilt** (sector / country from `etf/sector-weightings` / `etf/country-weightings`), and a **fund-level valuation** (read against the house view). **Constituent concentration** (top-N weight, Herfindahl) is **off-plan by default** — `etf/holdings` is blocked — so it is computed only when **SEC N-PORT** is available and otherwise omitted (recorded as a degraded-input flag, never a fabricated number). These feed a **reduced sub-score set** — **valuation** plus a **risk** score driven by exposure (and concentration where N-PORT supplies it), with **no company-quality score** (there is no single issuer to grade) — while the action, conviction, and continuity machinery stay identical to the equity path; the **evidence floor uses the fund analog** above, not the company-statements floor (see [§Evidence floor](#evidence-floor)).

## Evidence floor

Eligibility decides whether a position *can* be graded; the evidence floor decides whether it *should* be — and the engine's inputs split by criticality. **Floor-bearing inputs** — the company's **financial statements** (FMP / SEC) and a **current price** — are the irreducible basis for a grade: an eligible holding missing these, or carrying stale prices, too few quality sources, or conflicting identity data, returns **`insufficient-evidence`** — an explicit abstention — rather than a low-conviction grade. A **fund** holding has no company statements, so it floors on the **fund analog**: a current **quote / NAV**, `etf/info`, the expense ratio, and (for ETFs) enough **sector / country weighting** coverage to read exposure — missing those abstains the fund the same way. Constituent **concentration** is **enriching, not floor-bearing** (it needs the off-plan `etf/holdings`, recoverable only via SEC N-PORT), so its absence lowers conviction rather than forcing abstention. Every **other** input (`financial-scores`, peers, the revision / surprise reads, positioning, dividends, float, beta, and all analyst data) is **enriching**: its absence **lowers conviction** and is recorded as a degraded-input flag, but never on its own forces abstention. "Firm" means committing when the evidence supports it, not manufacturing a verdict when it doesn't — the per-holding analog of the report's Step-3 coverage floor.

## Holdings change tracking

Each run persists the **holdings snapshot it ran against** (see [§Storage and display](#storage-and-display)), so the next run diffs the current Schwab pull against the prior run's snapshot deterministically — in the application layer, before any model stage. Every current position is classified by quantity and cost basis as **new**, **increased**, **decreased**, or **unchanged**, and that delta rides into the holding's dossier so the verdict reasons over what the user actually did — added to, trimmed, or left a position — rather than re-grading it in a vacuum. A symbol present last run but **absent now** is an **exited** position: it earns no per-holding verdict (there is nothing left to grade), but it is surfaced in the portfolio roll-up as closed-since-last-run, so a sold-out name is acknowledged rather than silently vanishing from the run. The diff is the app's, not the model's — the same compute-don't-guess boundary the rest of the pipeline holds.

## The position thesis ledger

Where holdings-change tracking remembers *what the user did* with a position, the **thesis ledger** remembers *why the job holds a view on it* — a compact, persisted, per-holding record carried forward run to run, the Portfolio analog of Trade Opportunities' opportunity graph ([trade-opportunities.md §Discovery memory](trade-opportunities.md#discovery-memory-the-opportunity-graph)) but **downstream and holding-specific**: it tracks an owned position rather than discovering a new one. Without it, each run re-derives its conviction from scratch and continuity is only a value-by-value diff; with it, a run tests the *standing thesis* against fresh evidence and says precisely which part broke.

Each gradable holding carries a ledger with:

- **Thesis** — the original thesis (when the position first entered an analysis) and the current thesis, so drift is legible.
- **Key drivers** — the few variables the thesis actually depends on (e.g. data-center revenue growth, gross-margin trajectory, net retention), each tied where possible to an engine-tracked metric so the next run can read whether it moved.
- **Bear / base / bull monitor** — three compact scenarios, each with the conditions that define it, the engine's scenario price target for it, and a rough probability lean. The monitor names, per scenario, **what must improve** to migrate toward the bull case, **what must not break** to stay in the base case, and the conditions that would force a downgrade or license an upgrade — so the next run's continuity check has explicit goalposts rather than a gestalt.
- **Key falsifiers** — monitorable conditions that would invalidate the thesis (the Portfolio analog of an opportunity's key falsifiers), each typed as **quantitative** (engine-checkable against the input delta — e.g. "operating margin falls below 30%") or **qualitative** (research/model-checkable — e.g. "a credible second source ships at scale"). A **third-party technology event** that threatens a holding (a competitor's or supplier's product / standard announcement) is a first-class **qualitative falsifier class** here: when one fires, it is recorded with the same sized **`technology_read`** the Trade Opportunities lens uses (substitute / complement / mix-shift, the *actually-exposed* revenue / profit pool, deployment timeline, switching costs), so "is this a real impairment or a panic-priced threat?" is tracked as a monitorable condition rather than a vague worry ([§The per-holding pipeline](#the-per-holding-pipeline) Step 3; [trade-opportunities.md §The event-impact / value-chain repricing lens](trade-opportunities.md#the-event-impact--value-chain-repricing-lens)).
- **Action triggers** — explicit **add / trim / sell triggers** and a **target-weight range**: the pre-committed conditions under which the standing view would change the position (e.g. "trim above 8% portfolio weight," "add on a pullback to the base-case downside with the thesis intact"). The triggers are *intrinsic* conditions on the holding; the portfolio-construction stage ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)) still owns whether a fired trigger actually sizes, given the rest of the book.

A **fund** holding carries a ledger of the same shape but a different flavor of content, matching its reduced engine path ([§Asset eligibility](#asset-eligibility)): its **drivers** are exposure tilt, expense drag, mandate / tracking fidelity, role in the portfolio, and house-view fit — not company operating metrics — and its falsifiers and triggers are written against those (e.g. "mandate drift," "expense ratio rises above the category," "the exposure this fund supplies is now better held directly").

The ledger is **read at dossier assembly and re-evaluated and rewritten each run**, across the pipeline's deterministic and model stages ([§The per-holding pipeline](#the-per-holding-pipeline)): the **engine** (as part of its input delta) tests which **quantitative** falsifiers and triggers crossed this run; the **interpretation stage** reads those crossings, judges the **qualitative** conditions from research, and rewrites the thesis, re-weights the scenarios, and re-sets the triggers; and the **continuity check** validates the rewrite. A tripped falsifier or fired trigger is a first-class continuity signal held to the same app-validation as the what-changed audit — it must resolve to a real input-delta crossing or a source-backed finding, so the ledger cannot be quietly rewritten to fit a new verdict. The ledger is persisted with the run and survives within the job's retention like the rest of a holding's continuity (see [§Storage and display](#storage-and-display)).

## The per-holding pipeline

This section covers the per-holding pipeline's **design**; the full job's **control flow** — every step, its Type (Computed / API retrieval / Local-model call), and each local-model-call contract — is the canonical spec in [portfolio-workflow.md](portfolio-workflow.md) (the per-holding loop is its Step 6; the six numbered stages below map to workflow sub-steps **6a–6g**, with stage 5 here spanning the workflow's separate target-refinement [6e] and interpretation [6f] steps).

Each holding is processed through a chain of distilled, schema-validated hand-offs (see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)):

1. **Dossier assembly (application layer — deterministic, with a non-reasoning embedding lookup for continuity retrieval).** The app — not the model — assembles the holding's evidence packet, holding the pure-stage boundary (the only model touch is the fixed embedder vectorizing the continuity query — no reasoning; see [portfolio-workflow.md §Step 6a](portfolio-workflow.md#step-6a-dossier-assembly)): the position and its prior-run **position delta** ([§Holdings change tracking](#holdings-change-tracking)); the per-symbol **fundamentals / revision / positioning / price** surface (equity holdings draw the full FMP + SEC + Stooq surface, funds the reduced ETF surface — [§Asset eligibility](#asset-eligibility); every endpoint, by source and cardinality, is in [data-sources.md §Portfolio Analysis — endpoint surface](data-sources.md#portfolio-analysis--endpoint-surface) and [portfolio-workflow.md §Step 6a](portfolio-workflow.md#step-6a-dossier-assembly)), with **SEC EDGAR** the authoritative cross-check behind FMP's working feed; the **investor profile** ([configuration.md](configuration.md)); the Market Signal **house view**, loaded *deterministically* (the report's own sections + summaries, never vector-searched — [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)) and **freshness-gated** (dropped past one week — [portfolio-workflow.md §Step 5](portfolio-workflow.md#step-5-load-shared-context)); and the **prior run's verdict and thesis ledger** ([§The position thesis ledger](#the-position-thesis-ledger)). Three market inputs — the **risk-free rate**, **commodity prices**, and **CFTC positioning** — are fetched **once per run and shared across every holding** rather than re-requested per symbol, the load-relief that keeps the per-holding budget small.
2. **Deterministic financial analysis (application layer).** The shared Rust engine (also Trade Opportunities' — [trade-opportunities.md](trade-opportunities.md)) computes every number in three layers, the load-bearing decision being *which layer touches the grade*: **(a) the grade core** — fundamentals, valuation multiples, price-action, and the engine's **forensic flags** → the **quality / valuation / momentum / risk** sub-scores and scenario **price targets** (discounted off the run-level risk-free rate); **(b) a conviction layer** — revision velocity, rating drift, earnings surprise → the **narrative-vs-reality ratio** (revision pace vs multiple — *justified-expensive* vs *hype*); and **(c) positioning context** — insider / congressional / FINRA short interest / the Schwab options signal. Layers (b) and (c) are **computed but held *out* of the sub-scores** (positioning until shadow-mode calibration proves it adds value), so new signals sharpen *conviction and risk* without moving the letter grade — the three-layer engine's whole point. From the targets the engine derives the **capital-efficiency / dead-money read** (base-case forward return vs a risk-free-plus-premium hurdle), also kept out of the grade and fed to the action-sizing spine ([§The holding verdict](#the-holding-verdict)); the targets and this read are **provisional**, recomputed if research refines the base case (Step 5), while the backward-looking sub-scores are final. The engine also diffs this run against the prior run's stored values into a deterministic **input delta** — the evidence the continuity audit (Step 6) holds every verdict move to, and the same delta that evaluates the thesis ledger's **quantitative** falsifiers and triggers ([§The position thesis ledger](#the-position-thesis-ledger)). The dossier's **absolute street opinions** (consensus target *level*, buy / hold / sell rating, FMP's ratings snapshot) are deliberately **not** engine inputs — they ride into interpretation (Step 5) as *evidence the model weighs against the engine's own read*, never numbers to adopt. These computed values, not the model, are the source of every number in the verdict ([local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)).
3. **Bounded web research** — the 122B reasoner (thinking mode) over the bounded per-topic research loop ([web-research.md](web-research.md)), fail-soft, grounded by the deterministic SEC/FMP financials so research *fills the gaps the numbers leave* rather than substituting a story. Its agenda: competitive / business position, recent results and estimate revisions, catalysts and risks; the **market narrative and sentiment** as a *sourced qualitative* read (how much of the price reflects emotion about what *might* come versus present fundamentals — distinct from the quantitative options / momentum proxies); and **forward opportunity and thematic fit** (TAM, optionality, theme exposure). **Conditionally**, when the holding moved on a third-party **technology event** (a competitor's or supplier's announcement that repriced the name, the same materiality gate Trade Opportunities applies), the agenda adds a **technology-event impact assessment** — reading the actual technology and *sizing the holding's real exposure* into a typed **`technology_read`** ([trade-opportunities.md §The event-impact / value-chain repricing lens](trade-opportunities.md#the-event-impact--value-chain-repricing-lens)) — which is what separates a **panic drop from a genuine impairment** (and, symmetrically, overstated-benefit euphoria on the upside); it feeds the conviction read and a thesis-ledger falsifier ([§The position thesis ledger](#the-position-thesis-ledger)), dormant otherwise.
4. **Distillation** — the reasoner in **non-thinking mode** consolidates each topic's **complete** findings into the compact object interpretation reads (never a re-distillation of in-loop summaries), single-pass or hierarchical by the deterministic shared rule ([web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management)); Portfolio has no cross-lens contradiction check, so the reduce here is pure consolidation. Running it on the same resident 122B avoids per-holding model swaps — the 35B fast tier is a benchmark-gated option ([local-models.md §The model roster and per-task routing](local-models.md#the-model-roster-and-per-task-routing)).
5. **Interpretation and grading.** First a **deterministic target refinement**: if distillation surfaced a typed **`research_forward_assumption`** (a sourced forward fact the structured feeds lacked), the **engine — not the model —** recomputes the affected scenario target with it as an explicit, logged assumption and refreshes the capital-efficiency / dead-money read, so the forward view reflects research while the number stays engine-computed (the refinement contract is [portfolio-workflow.md §Step 6e](portfolio-workflow.md#step-6e-deterministic-target-refinement); the backward-looking sub-scores are untouched). Then the 122B in thinking mode interprets the refined analysis and the distilled research into the **intrinsic verdict** — grade, conviction, horizon, the justified base-case target, and a **standalone action lean** (what it would do if the holding stood alone, before portfolio context) — **reading every number from the engine, never inventing one**, with the *final* portfolio action deferred to the construction stage that has the whole book in view ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)). This stage also **rewrites the thesis ledger** — thesis, bear/base/bull monitor, falsifiers, and triggers — in light of what the run learned.
6. **Continuity check.** An **app-layer validator**, not just a recorder: every verdict move interpretation labels **external** (market data, company information, or research / narrative) must resolve to a concrete **input-delta** entry, a source-backed research finding, or the logged forward assumption — otherwise it is **downgraded to a labeled self-correction** (or fails schema validation), so a no-new-facts swing can't be laundered as "the market changed." Output stays firm run-to-run, the self-correction the earned, flagged exception ([thesis-continuity.md](thesis-continuity.md)). The same rule **validates the thesis-ledger rewrite** — every falsifier marked tripped and every trigger marked fired must map to the engine's deterministic crossing (quantitative) or a source-backed finding (qualitative), so the ledger can't be quietly rewritten to fit the verdict ([§The position thesis ledger](#the-position-thesis-ledger)).

## The holding verdict

Each holding's output is a structured, schema-validated record that deliberately keeps two things apart — the **intrinsic verdict** (what the holding is worth on its own merits) and the **portfolio action** (what to do about it given the rest of the book). A holding can be an A-grade business and a *trim* because it is oversized, crowded, or duplicative; a B-grade business can be an *add* because it diversifies the book at better risk-adjusted upside. Collapsing the two hides that reasoning; separating them — and auditing each on its own — is what keeps an "A-grade" from silently becoming "add aggressively."

Inside the intrinsic verdict, two reads are likewise kept **orthogonal** — the **grade** (a *backward / current* quality anchor, computed deterministically from the trailing financials, valuation, price-action, and forensic flags) and the **forward outlook** (a *forward* scenario read — the horizon outlooks and scenario price targets, refined by research). They answer different questions: *what the business is* versus *what the market is likely to pay for it next*. A low grade with a strong forward outlook is a **legitimate, first-class state, not a contradiction** — the market routinely prices the future, so a C- or D-graded business mid-inflection can carry a constructive outlook; the card shows the two side by side and never collapses the forward read into the letter (or the letter into the forward read). This is the per-holding form of the suite's **four-part model** — **grade** (backward anchor) · **forward outlook** (scenario, research-informed) · **conviction** (the evidence-weighted synthesis, which moves *both* up and down) · **action** (the portfolio-aware decision over all three). One **governing invariant** binds it: research may lift the forward outlook and conviction, but a conviction *raise* must trace to a *countable, dated, third-party leading indicator the engine's structured feeds did not already score* — **never** to price action or narrative, and never to a metric the engine already counted (the anti-reflexivity / no-double-count guard the suite holds throughout — [trade-opportunities.md §Outcome learning](trade-opportunities.md#outcome-learning-calibration)).

### Intrinsic verdict

Decided in the per-holding loop with **no whole-book context** — it is a property of the holding alone:

- **Composite grade** (A–F) over the engine's deterministically-computed sub-scores — **quality**, **valuation**, **momentum**, and **risk** — so the letter rolls up from real metrics, not a model's gestalt.
- **Conviction** — the verdict's confidence, and the one read that moves in **both** directions. It is **lowered** when evidence is thin or degraded and **capped** by the **narrative-vs-reality** read (a price driven by multiple-expansion the estimates don't support caps conviction even on a strong company), and below the evidence floor it abstains entirely (see [§Evidence floor](#evidence-floor)). It is **raised** only when research validates a *countable, dated, third-party leading indicator the engine's structured feeds did not already score* — a confirmed thesis-ledger **key driver** moving *ahead of* the trailing financials (the held-name case for the very C-/D-graded-but-inflecting business the grade alone misreads). A raise is **bounded (≤ one band), cited to that indicator, and logged**, and is held to the same continuity validation as any other verdict move ([portfolio-workflow.md §Step 6g](portfolio-workflow.md#step-6g-continuity-check-and-checkpoint)). Price action or narrative on its own can **never** raise it — the governing invariant above.
- **Forward outlook** — the holding's *forward* read, kept distinct from the backward grade and surfaced beside it: separate short-, mid-, and long-term directional reads, together with the scenario price targets below. A grade/outlook divergence — a weak business with an improving setup, or a strong one the market has already paid for — is shown as such rather than averaged into the letter ([§The holding verdict](#the-holding-verdict) intro).
- **Price targets** — end-of-month and end-of-year, computed by the financial-analysis engine as scenario outputs with their **methodology and assumptions exposed** (including any research-sourced forward assumption folded in post-research — see [§The per-holding pipeline](#the-per-holding-pipeline) Step 5), aligned to the ledger's bear/base/bull monitor; the model selects and justifies the base case rather than inventing the number.
- **Standalone action lean** — what the action *would* be if the holding stood alone (a pure read of grade × conviction × upside/downside × risk — plus the engine's **capital-efficiency / dead-money** read — on the same ladder), set **before** portfolio constraints. A holding whose forward base-case return doesn't clear the hurdle leans toward **exit here, on its own merits**; the **possible-tax-benefit and redeployment** counterweights that firm up that exit are applied with the whole book in view at construction ([§Portfolio action](#portfolio-action)). It is the input the construction stage reconciles, and showing it beside the final action is what makes the intrinsic-vs-portfolio split visible.
- **Financial analysis** — a concise read of the company's financial health.

### Portfolio action

Decided **after** the roll-up has the whole book in view (see [§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)), so the action reflects the portfolio, not just the holding:

- **Action**, on a fixed ladder: **sell all → trim → hold → add → add aggressively**, made concrete with a **target portfolio-weight range and an estimated share/dollar adjustment**. No orders are ever placed.
- **The action-sizing spine.** The action and its size are reconciled from an explicit, mostly-deterministic input set rather than a model's gestalt: the **grade** and **conviction**, the **upside / downside** the targets imply against the current price, the engine's **capital-efficiency read** (the holding's forward base-case return against a hurdle — the run-level risk-free rate plus a risk premium — flagging a position as **dead money** when its forward prospects don't clear it), the **risk tier**, the **existing weight**, the **concentration headroom** against the profile's limits, the **correlation / exposure overlap** with the rest of the book (a name held directly and inside a fund, or several names sharing one factor, size down together), and the **cash / buying-power and tax** constraints. The engine computes these and **bounds the feasible action set** — *add aggressively* is offered only when buying power and concentration headroom support it; a *sell all* sitting on a large taxable gain carries that cost into the call — and the model chooses within those bounds and justifies it. (The buying-power bound is **profile-controlled**: under the current fixed preset, cash is treated as **unconstrained**, so that bound is **inert** and only concentration headroom binds *add aggressively* — see [configuration.md §Investor Profile](configuration.md#investor-profile); a future configurable profile can re-enable it.) The spine is what prevents a strong grade from sizing past what the portfolio can bear.

  **Capital efficiency and the sunk-cost guard.** A holding flagged **dead money** is where the spine deliberately overrides the disposition reflex. The instinct to hold a deep loser "until it comes back" — or because selling "wouldn't free much cash" — treats the unrealized loss as a reason to stay; the spine treats it as the opposite. When the forward base case doesn't clear the hurdle over the holding's horizon, **realizing the position (some or all) is leaned toward, not away from**, and two *generic* counterweights support that exit beyond the weak forward read: the **possible tax benefit** of booking the loss, and the **redeployment optionality** of the freed cash — worthwhile even when the proceeds are modest, since dead capital earns nothing. These are weighed **high-level**: the job models neither the user's account type and tax rate nor a specific replacement name (naming the next idea stays Trade Opportunities' job, kept isolated — [trade-opportunities.md](trade-opportunities.md)); it **assumes the user acts on the specifics appropriately**. Two disciplines keep this from becoming a license to dump every red position: the unrealized loss **never moves the grade or sub-scores** — those stay purely forward-looking and cost-basis-agnostic ([§Intrinsic verdict](#intrinsic-verdict)), so it informs only the *action and its rationale* — and the lean applies **only once the forward prospects are independently judged poor**. A loser with an intact, improving thesis is still a hold (or an add); the lean to realize-and-redeploy is reserved for a position the forward read has already written off.

### What changed

A structured **audit of the prior verdict**, not a bare diff, **split to match the separation above**:

- The **intrinsic** half attributes every moved value (grade, any sub-score, conviction, target, horizon, a re-weighted scenario, a tripped falsifier or fired trigger) old → new to an **external change** — **market data** (price / momentum / positioning), **company information** (earnings, guidance, a new filing, a fundamentals shift), or **research / narrative** (a catalyst or narrative change), tied to the specific engine input-delta entry or source-backed finding that moved — or to a **self-correction**, where the inputs did *not* materially change and the model is revising its own prior read.
- The **action** half attributes a changed action or target weight to either a **moved intrinsic verdict** *or* a **moved portfolio context** (the holding became oversized, cash was freed for a higher-conviction name, an overlap emerged) — so an action can change with the intrinsic verdict unchanged, and that reads honestly rather than as a mystery re-grade.

A self-correction is the **flagged exception** to firm-output continuity — it swings the verdict with no new facts — so it is labeled as such rather than dressed as an external move, and repeated self-corrections on one holding are a calibration signal. The line also carries the **position delta** (new / increased / decreased / unchanged since last run; an unseen holding reads as `new`).

The fixed action ladder and grade vocabulary are load-bearing: like the report's fixed regime labels ([storage.md](storage.md)), they keep verdicts comparable across runs and prevent the model from retreating into hedged, non-comparable language. The grade is **intrinsic** to the holding; the **action** additionally reflects the investor profile (horizon, risk tolerance, objective, tax sensitivity), available cash, and the whole-book context the construction stage supplies.

## Portfolio roll-up and construction

After the per-holding pass produces every **intrinsic verdict** and standalone action lean, a final stage takes the whole book in view and does the two things the per-holding loop structurally cannot — because that loop decides each holding before any other holding's verdict exists. It runs in two parts.

First, **deterministic whole-book aggregates** (application layer): concentration and sector / factor exposure (with fund exposure folded in at the **sector / country level** from each held fund's weightings — single-name look-through into fund constituents is off-plan with `etf/holdings`, so a name held both directly and inside an ETF aggregates at the **exposure** level, not per-name, unless SEC N-PORT supplies constituents), correlation / overlap clusters, the cash / buying-power position, **the risk / exposure contribution of any material not-rated positions** (fixed-income duration / credit weight, options notional / delta — graded nowhere but real exposure), and, per holding, the **action-sizing spine inputs** ([§The holding verdict](#the-holding-verdict)) — existing weight, concentration headroom, overlap contribution, upside/downside, the capital-efficiency / dead-money read, unrealized P/L (a harvestable loss or a taxable gain, by sign), risk tier, tax. These are computed, not modeled, and they are the inputs the next part is bounded by.

Then a **portfolio-construction synthesis** (122B) reconciles each holding's intrinsic standalone lean against those aggregates into its **final action and target weight**, and produces the portfolio-level view: overall risk posture, a cash / deployment stance (what to trim to fund which adds), the concentration and exposure read, and **positions closed since the last run** (the exited names from the holdings diff — graded nowhere, but acknowledged here rather than silently dropped) — all read against both the report's house view and the investor profile. This is the only stage that can say "trim the A-grade because it is 11% of the book and correlated with two other holdings, and route the proceeds to the B+ that diversifies it." The per-holding **action half of the what-changed audit** is written here, where the portfolio context exists, and is app-validated the same way the intrinsic half is — a "became oversized" claim must map to a real aggregate, not a model assertion. Beyond the sum of individual holdings, this is where the job answers "what does the portfolio as a whole look like, how should it be shaped, and how does it sit relative to the current market thesis."

## Starting parameters (calibratable)

The deterministic engine runs on concrete constants, gathered as a single **calibration surface** at the top of `src-tauri/src/portfolio/engine.rs` (with the durable plan-time parameters in `portfolio/mod.rs`). Every value below is an **MVP starting point chosen to be shadow-tuned against live runs, not frozen** — the same stance the report's thresholds take. They are recorded here so the spec matches the engine, not to pin them.

**As-built (the narrow single-equity slice).**

- **Composite grade.** Sub-score weights **quality 0.30 / valuation 0.25 / momentum 0.20 / risk 0.25** (fixed, summing to 1.0); letter cutoffs over the 0–100 composite — **A ≥ 85, B ≥ 70, C ≥ 55, D ≥ 40**, else F.
- **Sub-scores.** Each a clamped linear map to 0–100 (higher = better): **quality** from net and gross margin, **valuation** from inverted P/E · P/S · P/B (a negative P/E scores low, never "cheap"), **momentum** from trailing price return, **risk** (higher = safer) from realized volatility and debt/equity. A missing sub-score is **imputed to the neutral midpoint (50)** and the composite divides by the full weight sum — so a missing input pulls the composite toward neutral rather than being dropped (not a renormalization over the present sub-scores); the evidence floor guarantees at least two sub-scores are real.
- **Evidence floor.** Fewer than **2** computable sub-scores, or no current price, abstains as `insufficient-evidence` rather than grading on too little.
- **Scenario targets.** Expected annual drift = `clamp(0.5 × revenue-growth + 0.02, −0.25, +0.35)`; end-of-year base = spot × (1 + drift), end-of-month = spot × (1 + drift ⁄ 12); bull/bear bands scale off realized volatility with fixed fallbacks (±15% EOY, ±5% EOM).
- **Verdict vocabularies.** Conviction is categorical **High / Medium / Low** (model-authored); the horizon outlook is **short ≈ 1 month / mid ≈ 1 year / long ≈ 3–5 years**, each a {bullish, neutral, bearish} read.
- **Action sizing.** Per-rung target-weight multipliers around the position's current weight — **trim 0.4–0.7, hold 0.9–1.1, add 1.2–1.6, add-aggressively 1.6–2.2** — under a single-position **concentration cap of 25%** of the account. A buy is additionally bounded by the profile's cash cap **when one is set**, but the fixed preset encodes cash as **unconstrained** (`available_cash: None` — [configuration.md §Investor Profile](configuration.md#investor-profile)), so under it adds are never gated on observed Schwab cash.
- **Retention.** Newest **10** Portfolio runs (independent of the 30-report window and of Trade Opportunities); the house view loads the **3** most recent reports.

**Genuinely open (drafted starting values, landing with the slice that needs them).**

- **Dead-money hurdle.** `hurdle = run-level risk-free rate + 5% risk premium` (the premium calibratable, or replaced by the live Damodaran total equity-risk-premium already fetched for the report). A holding is **dead money** when its base-case forward annual return (the engine's expected-drift `annual`) falls below the hurdle → its standalone lean tilts to **exit, on its own merits** ([§The holding verdict](#the-holding-verdict)).
- **Action feasible-set bounding.** Before the model picks a rung, the engine narrows the offered set from **engine-known inputs only** (grade, dead-money, concentration, cash — conviction is model-authored, so it can't pre-gate): drop *add* / *add-aggressively* when the holding is dead money and grade ≤ C, when grade = F, or when current weight ≥ the 25% cap; offer *add-aggressively* only for an A/B grade with headroom. The prompt states the allowed set; the model chooses within it ([§Portfolio action](#portfolio-action)). (The cash bound is **profile-gated**; the fixed preset sets cash unconstrained — `available_cash: None` — so it isn't a feasible-set gate here.)
- **Material not-rated threshold.** A not-rated position (option / bond / cash) at **≥ 5%** of the account contributes its risk / exposure to the whole-book aggregates ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)); below that it is recorded but does not drive the roll-up.
- **Conviction-layer caps** (need the paid-FMP revision feed + `financial-scores`). Narrative-vs-reality caps conviction when multiple expansion outruns estimate revisions by more than **~1.5×** with no leading-metric anchor; the forensic flags cap conviction — **Altman Z < 1.8**, **Piotroski ≤ 3**, **net income > 1.3× operating cash flow**, **receivables / inventory growth > 1.5× revenue growth** — with a restatement / auditor-change the hard one. These move conviction and the risk read, **never the letter grade**.
- **Forward-conviction raise (bidirectional conviction).** Research may **raise** conviction by at most **one band** (Low → Medium → High), and only when it validates an *engine-unscored*, countable / dated / third-party leading indicator (a `research_metric`-class signal, carried as the typed **`validated_leading_indicator`** distillation emits — [portfolio-workflow.md §Step 6d](portfolio-workflow.md#step-6d-distillation)) confirming a thesis-ledger **key driver** *ahead of* the trailing financials; the raise must cite that field and is logged and continuity-validated ([portfolio-workflow.md §Step 6g](portfolio-workflow.md#step-6g-continuity-check-and-checkpoint)). Price action and narrative can never raise it, and a metric the engine already scored cannot (no double-count). The **one-band cap** and the **leading-indicator gate** are the calibratable values.
- **Whole-book reads.** Correlation / overlap starts from a **90-day Stooq return correlation**, co-sizing names with **|ρ| > 0.7**; a dedicated **factor-exposure model is deferred** (the sector / country exposure tilt already specified at construction carries the load until one is justified).

## Continuity and isolation

The job retains its most recent N runs. The prior run feeds the next (continuity above), and run results are embedded into the **Portfolio Analysis memory partition only** — its learnings are never read by the report or by Trade Opportunities, and it never reads theirs (see [local-models.md §Run history and continuity](local-models.md#run-history-and-continuity)). This keeps holding-grading calibration cumulative within the job and uncontaminated by other jobs' context.

## Storage and display

Each run persists its per-holding verdicts, the per-holding **thesis ledger** (carried forward to seed the next run's continuity check), and the portfolio roll-up, together with an **audit record** that makes the run traceable and reviewable: the holdings snapshot it ran against, the report(s) and sources it used (with retrieval timestamps), the distilled findings and the **computed financial metrics, the derived reads (revision / narrative-vs-reality, forensic flags, positioning, the capital-efficiency / dead-money read), the conviction decomposition (`base_conviction` + any `conviction_raise` with its cited `validated_leading_indicator` + the app-computed `final_conviction`, so an honored-or-dropped raise is reconstructable — [portfolio-workflow.md §Step 6g](portfolio-workflow.md#step-6g-continuity-check-and-checkpoint)), the input delta and the per-value what-changed attribution, and the price-target methodology — including its discount-rate assumption and any research-sourced forward assumption applied in the post-distillation refinement (with its source) —** behind each verdict, the model ids and quantizations, the prompt/schema version, and any degraded-input flags. (Source URLs, timestamps, and the distilled findings make a verdict's basis inspectable; full page snapshots are not stored, so a run is traceable rather than bit-for-bit reproducible.) Retention keeps the last N runs (parallel to report retention — [storage.md](storage.md)). The **Portfolio page** renders each holding's verdict — the **intrinsic** read (the **backward grade** and sub-scores paired *side by side* with the **forward outlook** — horizon reads and scenario targets — so a grade/outlook divergence is legible at a glance, plus conviction and the bear/base/bull monitor) beside the **portfolio action** (action, target weight, sizing rationale), **anchored by the thesis ledger's current standing thesis** as the card's *why we hold this view* (rendered straight from the ledger the continuity check carries forward and rewrites — [§The position thesis ledger](#the-position-thesis-ledger) — **not** a separately authored summary, so there is one continuity-validated source of truth and no drift between what the job carries forward and what the user reads; the held-position analog of each Trade Opportunity leading with its directional thesis), with financials and the what-changed line — alongside the portfolio roll-up, and shows not-rated positions with their reason (see [interface.md](interface.md)).

**Holdings sorting.** Above the holding-card stack a compact **sort bar** reorders the cards **in place** — no separate table view, so every card keeps its full intrinsic-verdict / portfolio-action / thesis-anchor content. It offers four deterministic, **position-level** keys, all engine-computed (never the model's grade or conviction — sorting by position size or P/L is orthogonal to the analysis read) from two **Schwab-reported per-position values** that already account for option contract multipliers, bond conventions, and cash balances — the position's **market value** and its **cost basis** — rather than a naïve `quote × shares` that would misprice options, fixed income, and cash: **overall value** (market value), **dollar gain** (market value − cost basis, the same unrealized P/L the action-sizing spine derives — [§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)), **percentage gain** (that gain over cost basis), and **total cash invested** (cost basis). A position Schwab reports **no cost basis** for (cash, typically) has an **undefined gain and cash-invested**, so it sorts **last** under the dollar-gain, percentage-gain, and cash-invested keys (the holdings analog of a debut Trade Opportunity's null since-flagged read), while still sorting normally by overall value; a zero or missing denominator never produces a divide-by-zero rank. The default is **overall value, descending** (biggest positions first), tie-broken alphabetically by ticker; the last-used key is remembered as **ephemeral frontend state** in webview `localStorage` — the same store the Light/Dark appearance preference uses (pure presentation, no backend consumer), not the SQLite config store. **Not-rated** positions (options / fixed income / cash) carry no grade but a real market value (and cost basis where Schwab reports one — cash typically has none), so they sort within the value / gain / cost keys alongside graded holdings, kept visually tagged as not-rated; **exited** positions (closed since last run) live only in the roll-up ([§Holdings change tracking](#holdings-change-tracking)), never in the sortable stack. The control is **display-only** — it reorders already-computed fields and touches no engine, schema, or workflow — and follows the project's frontend-craft requirements (keyboard-operable **toggle buttons** exposing the active key via `aria-pressed` plus a screen-reader-announced current-sort state (the active button's **accessible name carries the key *and* its sort direction** — e.g. "Sort by value, descending" — since the `▾`/`▴` glyph is decorative / `aria-hidden` and can't be relied on to announce direction) — *not* `aria-sort`, which is reserved for grid/table column headers such as Trade Opportunities' List view, not this card-stack toolbar; a stable sort; and a reduced-motion-respecting reflow). The sort bar is a small **design-system extension**: it ports the `.ana-grid` sortable-header vocabulary (the neutral `▾` / `▴` glyph) to a non-table toolbar of tracked-caps triggers, since the holdings surface is cards rather than a data grid — no off-system dropdown or pill control ([interface.md](interface.md)).

## Failure posture

The execution gate requires the local model daemon and roster ([local-models.md](local-models.md)) **and a connected Schwab account** ([schwab-integration.md](schwab-integration.md)) — holdings and the options signal both come from Schwab, so a missing or lapsed connection blocks the job (manual-import holdings do not satisfy the gate). Because holdings are independent, the job **checkpoints per holding** — each holding's completed stages persist, so a cancellation or a single model failure resumes the unfinished holdings rather than restarting the (potentially hours-long) run, and recent research is cached within a freshness window. Within a run, web research is fail-soft (thinner evidence, lower conviction), while a hard model or persistence failure fails the run, recorded like any other job ([scheduling.md](scheduling.md)).
