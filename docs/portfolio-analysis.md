# Portfolio Analysis

Portfolio Analysis is a **local, on-demand job** that grades every gradable holding in the user's portfolio and recommends an action for each — a letter and price targets where the vehicle class is priceable, a typed role / risk read where it isn't ([§Asset eligibility](#asset-eligibility)) — grounded in the holding's fundamentals, fresh web research, and the current Market Signal house view.
It runs entirely on local models (see [local-models.md](local-models.md)) and sources holdings — and the per-stock options signal — from a **connected Charles Schwab account, which is required to run the job** (manual import supplements the holdings but does not replace the connection; see [schwab-integration.md](schwab-integration.md)).

It is deliberately **prescriptive** — it issues buy/trim/hold/sell-style actions and, on priced verdicts, explicit price targets — which is the opposite of the Market Signal Report's no-buy/sell stance ([report-structure.md](report-structure.md)).
The report forms the *house view*; this job applies that view to the specific positions the user holds.

## Triggering

The job is manual, with three independent controls rather than a sequenced flow:

- **Run analysis** is **one-touch and always starts from a fresh holdings pull**: it retrieves holdings from the connected Schwab account as its first step and runs against that pull ([portfolio-workflow.md §Step 2](portfolio-workflow.md#step-2-load-holdings-and-fetch-option-chains)) — never a prior standalone pull (**a resumed run is the one exception**: it reopens its own interrupted run's pinned pull rather than re-pulling — [§Failure posture](#failure-posture)).
  Every holding that enters the full per-holding loop recomputes its engine reads; cached *research*, by contrast, may be reused within the ~4-week window ([§The per-holding pipeline](#the-per-holding-pipeline) Step 3).
  With no cards selected it runs the whole book; with a **selection** active (per-card, with select-all / deselect-all — the same selection-gated pattern as Trade Opportunities' Audit) it becomes a **selective re-analysis**: the initial work-list is the selected holdings plus every holding new since the last run (which has no verdict to carry), then the safety sweep below expands that list with any required force-inclusions.
  Holdings left outside the resulting work-list carry their prior intrinsic verdict forward **vintage-stamped**, and the roll-up re-runs over the mixed-vintage verdicts at current weights ([portfolio-workflow.md §Step 6](portfolio-workflow.md#step-6-per-holding-analysis-loop)) — so a handful of flagged holdings can be re-analyzed in minutes rather than committing to the hours-long full run.
  Mixed vintages are made **safe, not just visible**, by three rules: the selective run first executes the **quick check over its unselected tail** (engine-only, cheap — [§The quick check](#the-quick-check-engine-only)) and **force-includes any holding it flags, any holding whose sweep result is `unknown`** (a required signal family's retrieval failed, so the sweep cannot vouch for the carried verdict — the degraded-sweep rule in [§The quick check](#the-quick-check-engine-only)), **any holding whose position side reversed** (a long↔short flip since its verdict — thesis-changing by construction, so no carried verdict survives it — [§Asset eligibility](#asset-eligibility)), **or one that carries an unexamined evidence event** since its last full pass (the deterministic **evidence-event list**, every leg detectable by the sweep's own retrieval — [§Starting parameters](#starting-parameters-calibratable): the ledger's *anticipated* conditions can't be the only tripwire when new evidence is precisely what nobody anticipated) — alongside the new-holding force-include and the over-age-exit force-include below; every remaining unselected holding's final action is bound by an explicit **carried-action transition rule** — *toward hold only*: the roll-up may re-affirm the carried action or move it stepwise **toward *hold***, never away from it on either side of the bidirectional ladder (a fresh *add* is never synthesized from stale research, and a carried *trim* is never escalated to *sell all* — increasing an action's magnitude in either direction is an intrinsic-conviction call that needs a fresh pass), with one carve-out: fresh whole-book aggregates may move a carried *hold* or add-family action to ***trim*** — never to *sell all*, a full exit being an intrinsic call — when the action half's attribution maps to a real recomputed aggregate (a concentration-cap breach, an overlap cluster — the portfolio context is recomputed fresh every run, so a context-driven trim is not a stale-research call); and an **over-age** unselected holding — its last full pass older than the ~4-week window — is resolved **per action family**: a carried **add-family action is rule-demoted to *hold*** at the roll-up — a labeled, rule-based weaken, stamped **`action_source: rule-demoted`** so it never enters the outcome ledger as an analytical hold ([§Outcome learning](#outcome-learning-calibration)) — while a carried **exit-family action (*trim* / *sell all*) is force-included into the run's loop** like a flagged holding (an exit can neither be safely rule-demoted — softening risk-reducing advice on stale evidence raises risk — nor stand indefinitely as current advice for an irreversible act, so re-analysis is the only honest resolution; over-age *holds* stand: hold is the ladder's neutral rung, and anything actually wrong with one reaches the loop through the flag, evidence-event, and degraded-sweep force-includes), so a stale strong action can never stand as current advice on either side of the ladder, with its stale badge riding the card.
- **Quick check** is **engine-only** — no model call, no web research, no Schwab call: it re-prices the last run's book and evaluates every thesis ledger's machine-checkable conditions, raising per-card **attention flags** without rewriting anything ([§The quick check](#the-quick-check-engine-only)).
  Because it makes no model call it runs even when the daemon is configured-but-down — the same run-gate relaxation as Trade Opportunities' Quick Audit ([trade-opportunities.md §Failure posture](trade-opportunities.md#failure-posture)).
- **Pull holdings** is **view-only**: it fetches the current positions, persists them as the latest holdings snapshot, and displays them — it never triggers analysis, and it **never becomes the holdings-diff baseline** (the diff always compares the run-time pull against the prior *run's* snapshot — [§Holdings change tracking](#holdings-change-tracking) — so pulling twice between runs can't eat the what-did-the-user-do delta).
  Because it makes no model call, it gates only on the Schwab connection, not on the local-model configuration — the one Portfolio control usable before local models are set up.

Only one run happens at a time across the whole app — a single global run slot serializes the report, both local jobs, the quick check, and the pull (see [scheduling.md §Concurrent Job Protection](scheduling.md#concurrent-job-protection)).
While the analysis runs it streams into the run tracker with **per-holding progress**, the same observability seam the report uses ([run-tracking.md](run-tracking.md)).

## Asset eligibility

A Schwab portfolio holds more than equities — ETFs, mutual funds, options, fixed income, and cash.
The equity-centric pipeline (FMP fundamentals, company financials) applies cleanly only to individual stocks, and in reduced form to funds.
Each position is classified by asset type before analysis:

- **Stocks** — the full pipeline and verdict, **behind a deterministic listing-resolution guard**: Schwab's asset type alone says *equity*, not *US-listed equity* — an account can hold foreign ordinaries and other non-US listings — while the suite's data plan is US-only ([data-sources.md §FMP — current paid-plan tier audit](data-sources.md#fmp--current-paid-plan-tier-audit)).
  So at dossier assembly ([portfolio-workflow.md §Step 6a](portfolio-workflow.md#step-6a-dossier-assembly)) the position's Schwab instrument identity (symbol, CUSIP, description) must resolve to a **canonical FMP symbol** whose `profile` identity (issuer name, exchange) cross-checks against Schwab's — a loop-time refinement for the same reason the fund strategy classification is one (Step 3 has only Schwab's own instrument identity).
  Three outcomes: a **resolved, matching US listing** — US-listed ADRs included, being US listings with SEC filings — takes the full pipeline; a symbol with **no FMP resolution or a non-US primary listing** (a foreign ordinary held at Schwab) re-classifies to **not-rated — unsupported listing, with that reason** (a structural can't-grade, stable across runs, so the same abstention is never re-spent rediscovering); a symbol that **resolves but whose identity conflicts** (the FMP profile disagreeing with Schwab's issuer identity — an OTC ticker collision) is the evidence floor's **conflicting-identity leg made concrete** → `insufficient-evidence` at the floor check (a data problem, possibly transient — [§Evidence floor](#evidence-floor)), so a wrong-issuer mapping can never grade the wrong company's financials.
  Manual-import positions ride the same guard — they populate the same holdings model.
- **ETFs / funds** — a reduced verdict (no single-company financials; graded on strategy / **exposure**, valuation, and the house view).
  On the current FMP plan ([data-sources.md §FMP — current paid-plan tier audit](data-sources.md#fmp--current-paid-plan-tier-audit)) **`etf/holdings`, `etf/asset-exposure`, and the `funds/disclosure*` filings are off-plan**, so constituent-level look-through is lost.
  ETFs are sourced from the **on-plan** `etf/info` (expense ratio, AUM, NAV, asset class, mandate) + `etf/sector-weightings` / `etf/country-weightings` — an **exposure-tilt** read (sector / country, not single names).
  Single-name **concentration** and direct-vs-fund **overlap** degrade to that exposure level unless reconstructed from **SEC N-PORT** (keyless but heavy, ~60-day lag — an optional enhancement, never floor-bearing).
  A future **issuer-holdings adapter** — major ETF sponsors publish current constituent files fresher than N-PORT's lag, at the cost of per-issuer, messier parsing — is the practical way to buy back single-name look-through without a higher FMP tier; it is **planned, not built**, and like N-PORT would only enrich the fund path, never gate it.
  **Mutual funds degrade hardest**: FMP's `etf/info` and country-allocation contracts include mutual funds, but the sector / country weighting coverage required for the exposure-priced valuation is not guaranteed for a given fund; a mutual fund without that usable weighting set is not lettered at all — it takes the typed **`role_risk_only`** verdict on **quote / NAV + available profile / allocation data + the house view** (the classification below) unless SEC N-PORT supplies the missing exposure detail.
  `etf/asset-exposure` (reverse "which ETFs hold this name" lookup) is off-plan and dropped — it was only an optional per-direct-equity cross-check.
- **Options, fixed income, cash, and unsupported types** — marked **not rated**, with a short reason, and excluded from grading.
  Cash and buying power still feed the investor profile and the portfolio roll-up — and a **material** not-rated position (a sizable fixed-income sleeve, a large options position) still contributes its **risk and exposure** to the whole-book aggregates ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)), so the portfolio's risk posture reflects the real book rather than only the rated sleeve; it simply carries no per-position grade.
  An option position **on a held underlying** is additionally linked into that holding's dossier by symbol, so the equity's own verdict sees its overlay ([§The per-holding pipeline](#the-per-holding-pipeline) Step 1).

A not-rated position never receives a fabricated grade; the eligibility decision is explicit and shown in the UI.

**Net-short equity positions are not rated.**
The normalized holdings snapshot carries one signed net **book-level** quantity per symbol — same-symbol rows across accounts and manual supplements net at snapshot assembly ([schwab-integration.md §What is pulled](schwab-integration.md#what-is-pulled)) — and a position held net short is a direction the prescriptive layer doesn't model: the ladder's verbs, the sizing multipliers, and the outcome labels all read long ([§Portfolio action](#portfolio-action), [§Outcome learning](#outcome-learning-calibration)).
Rather than grade a short with long-side semantics, a net-short equity takes the **not-rated treatment with a short-position reason** — no grade, no ladder action, no outcome episode — while its **signed (negative) market value and exposure still feed the whole-book aggregates** like any material not-rated position, so the book's risk posture reflects the hedge or the bet rather than ignoring it.
A **long↔short reversal** is thesis-changing by construction — the diff never reads it as `unchanged` ([§Holdings change tracking](#holdings-change-tracking)) — and it **force-includes the holding into a selective run's work list** ([§Triggering](#triggering)), where the position re-enters as what it now is: a not-rated short after a long→short flip, a fresh full-pipeline candidate after a short→long flip — a long-side verdict is never carried across a reversal.

For a **fund** holding the engine runs a **reduced computation** in place of the equity pipeline: **expense drag** (the expense ratio as an annual return headwind), **exposure tilt** (sector / country from `etf/sector-weightings` / `etf/country-weightings`), and a **fund-level valuation** — with constituent look-through off-plan, valuation is an **exposure-priced read** computed as an **exposure-weighted earnings yield**, `Σ(weight ÷ sector P/E)` over the per-sector aggregate P/E (`sector-pe-snapshot`, exchange-tagged — the composite's per-sector read is a **defined blend of the exchange prints**: the NYSE and NASDAQ sector earnings yields averaged per sector, drafted, with the same convention applied to the snapshot and its history so the vs-own-history comparison is apples-to-apples — the level is a proxy, the time-series read is the signal), inverted only for a P/E presentation — displayed as a **proxy composite**, never as the fund's own P/E — with the weight in loss-making or P/E-unavailable sectors **reported separately, never averaged in** (P/E is nonlinear, so a weighted-average P/E is not a sound aggregate — the harmonic form is how index-level multiples are actually built) — the composite computed **over the covered weight only and renormalized by it** (`Σ(wᵢ ÷ PEᵢ) ÷ Σ(wᵢ)` across the sectors with a usable P/E), so uncovered weight neither silently reads as zero earnings nor lets a small priced slice extrapolate across the whole fund, under a **minimum-coverage guard** — drafted **≥ 70% of fund weight in P/E-usable sectors** ([§Starting parameters](#starting-parameters-calibratable)) — below which the valuation is **recorded as a gap rather than lettered off a sliver**, dropping the fund below the evidence floor's two-real-sub-scores bar into `insufficient-evidence` (genuinely deficient data, not an unpriceable class — [§Evidence floor](#evidence-floor)), with the same renormalization-and-coverage convention applied to the snapshot and its history so the vs-own-history read compares like to like — read against that composite's own trailing history (`historical-sector-pe`): *what the fund's sector mix costs now versus what it has cost* — a **constant-current-mix read** (today's weights over the historical sector multiples; the mix is held constant because historical fund weights aren't sourced) — never a constituent P/E the plan can't see ([data-sources.md §Portfolio Analysis — endpoint surface](data-sources.md#portfolio-analysis--endpoint-surface)); a closed-end fund additionally carries the **price-vs-NAV** read (applied only where the vehicle makes it meaningful — an open-end ETF's create / redeem mechanism keeps its premium small and transient in normal markets, unlike a CEF's structural discount, so the read carries signal only on the closed-end form), and a mutual fund without weightings records valuation as a gap, per its degrade above.
All of it is read against the house view.
**Constituent concentration** (top-N weight, Herfindahl) is **off-plan by default** — `etf/holdings` is blocked — so it is computed only when **SEC N-PORT** is available and otherwise omitted (recorded as a degraded-input flag, never a fabricated number).
These feed a **reduced sub-score set** — **valuation** plus a **risk** score driven by exposure (and concentration where N-PORT supplies it), with **no company-quality score** (there is no single issuer to grade) — for an **equity** fund the composite can honestly price; a class it can't returns the typed **`role_risk_only`** verdict instead (the classification below).
On the priced fund path the action, conviction, and continuity machinery stay identical to the equity path (a `role_risk_only` fund keeps the continuity machinery but runs the **reduced action spine** and carries **no decomposed conviction** — [§Intrinsic verdict](#intrinsic-verdict), [§Portfolio action](#portfolio-action)); the **evidence floor uses the fund analog** above, not the company-statements floor (see [§Evidence floor](#evidence-floor)).
One question is deliberately **left open for the fund slice's plan**: what a *priced* fund's scenario price targets — and therefore its dead-money read — derive from, since the as-built drift runs on revenue growth, which a fund doesn't have; the letter and the valuation read don't depend on the answer, but the priced branch's target fields do, so the fund slice names the fund-form target methodology before those fields are computed — a **blocking input to that slice's plan** (the targets drive the dead-money read, the upside / downside, and action sizing), not a during-implementation detail.

**Priced-fund grade contract.**
A priced equity fund supplies two real sub-scores — **valuation** and **risk** — and no company-quality score.
Its letter uses the shared composite's existing missing-sub-score rule ([§Starting parameters](#starting-parameters-calibratable)): the absent quality axis is **neutral-imputed to 50**, the composite keeps the full weight sum, and the card carries the visible low-confidence marker beside the letter.
That neutral placeholder is not presented or persisted as a fund-quality assessment.
The fund evidence floor still requires both real fund sub-scores, so a missing valuation cannot leave the neutral placeholder plus risk to manufacture a grade.
This closes the priced branch without introducing fund-specific weights; the fund-form target methodology remains the slice's only named blocking input.

**Funds are classified by strategy before the reduced computation runs** — a **loop-time routing decision**, made at dossier / engine time (workflow Steps 6a–6b) once `etf/info` is in hand: Step 3's asset-type eligibility needs only Schwab's own instrument identity, and the strategy class can't be read before its endpoint is retrieved.
The asset class routes the computation because one generic fund valuation cannot grade every vehicle: **equity funds** take the full exposure-valuation path above, *when the composite can honestly represent them* — the exposure-priced valuation applies only where the fund's exposure is substantially in the composite's market (drafted: **≥ 70% US** by `etf/country-weightings`), since an exchange-tagged US sector P/E is not an honest read on an international fund merely listed in the US; **bond and commodity funds** take a further-reduced path — expense drag + exposure / risk + house-view fit, with **valuation recorded as a gap** (the on-plan surface carries no duration, credit-quality, or curve data to price them honestly, and pretending sector P/E applies would fabricate a number); **leveraged / inverse and option-overlay vehicles** carry a deterministic **structurally-path-dependent flag** (from the profile / name — the same screen the report's movers list applies), since a buy-and-hold read of a daily-reset product is structurally unsound; and a **closed-end fund** adds the NAV read above.
Every class the pipeline is **structurally unable to price** — a bond or commodity fund, an ex-US-exposure fund below the guard, a leveraged / inverse vehicle, a mutual fund without weightings — returns a typed **`role_risk_only`** intrinsic verdict: **no A–F letter and no price targets** (a letter over a single computable sub-score would be a fabricated grade, and `insufficient-evidence` would misname it — the evidence isn't deficient, the vehicle class is unpriceable to this pipeline), while the **portfolio action machinery still applies** — construction can still trim an oversized sleeve from the fresh whole-book aggregates, with the action's rationale limited to portfolio role and risk.
The classification is deterministic, shown on the card, and a class the engine can't price honestly degrades — never a fabricated number.

## Evidence floor

Eligibility decides whether a position *can* be graded; the evidence floor decides whether it *should* be — and the engine's inputs split by criticality.
**Floor-bearing inputs** — the company's **financial statements** (FMP / SEC) and a **current price** — are the irreducible basis for a grade: an eligible holding missing these, or carrying stale prices, **fewer than the minimum real (non-imputed) sub-scores** (the two-real-sub-scores bar — [§Starting parameters](#starting-parameters-calibratable)), or conflicting identity data, returns **`insufficient-evidence`** — an explicit abstention — rather than a low-conviction grade.
Web-source tiers and extraction quality are **never floor inputs** — they move conviction and degraded-input flags only, with the explicit `deny` list the sole categorical web exclusion ([web-research.md §Source quality and evidence weighting](web-research.md#source-quality-and-evidence-weighting)).
A **fund** holding has no company statements, so it floors on the **fund analog**: a current **quote / NAV**, `etf/info`, the expense ratio, and — on the **exposure-priced (equity-fund) branch** — enough **sector / country weighting** coverage to read exposure — with the exposure-priced valuation under its own drafted **minimum P/E-coverage guard** (≥ 70% of fund weight in P/E-usable sectors — [§Starting parameters](#starting-parameters-calibratable)); missing those, or coverage below the guard, abstains the fund the same way.
The floor governs **evidence adequacy only**: a vehicle class the pipeline is *structurally* unable to price (a bond or commodity fund, an ex-US-exposure fund, a leveraged / inverse vehicle, a weightless mutual fund) is **not** an abstention — it takes the typed **`role_risk_only`** verdict ([§Asset eligibility](#asset-eligibility)), keeping `insufficient-evidence` reserved for genuinely missing or conflicting data.
Constituent **concentration** is **enriching, not floor-bearing** (it needs the off-plan `etf/holdings`, recoverable only via SEC N-PORT), so its absence lowers conviction rather than forcing abstention.
Every **other** input (`financial-scores`, peers, the revision / surprise reads, positioning, dividends, float, beta, and all analyst data) is **enriching**: its absence **lowers conviction** and is recorded as a degraded-input flag, but never on its own forces abstention.
"Firm" means committing when the evidence supports it, not manufacturing a verdict when it doesn't — the per-holding analog of the report's Step-3 coverage floor.

**Execution semantics.**
The floor is a **deterministic engine check, evaluated once at the end of the per-holding engine stage** ([portfolio-workflow.md §Step 6b](portfolio-workflow.md#step-6b-deterministic-financial-analysis)) — every floor-bearing input is structured-input presence, all in hand by then, and research cannot repair a missing statement, price, or weighting — so a below-floor holding **exits the loop there**, spending no research, distillation, target-refinement, or interpretation call (workflow Steps 6c–6f are skipped).
The exit persists the typed **`insufficient-evidence`** disposition with **named gap reasons** (which floor-bearing inputs were missing, stale, or conflicting) and **checkpoints the holding as completed**, so a resumed run does not redo it.
What survives the exit: the **standing thesis ledger is retained unrewritten** — only an interpretation call rewrites a ledger, and the quick check keeps evaluating it between runs ([§The quick check](#the-quick-check-engine-only)); any **attention flag is retained** — clearing requires the holding's next successful full pass, which an abstention is not; the holding still **contributes its real risk / exposure to the whole-book aggregates** (like a material not-rated position — graded nowhere, real exposure; [§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)) but receives **no per-holding action this run** (acting on an abstention would manufacture a prescription from the very evidence the floor found wanting; the card shows the abstention and its reasons instead); and **no new outcome episode is created** — there is no recommendation state to key one to, while prior episodes mature normally ([§Outcome learning](#outcome-learning-calibration)).

## Holdings change tracking

Each run persists the **holdings snapshot it ran against** (see [§Storage and display](#storage-and-display)), so the next run diffs the current pull against the prior run's snapshot deterministically — in the application layer, before any model stage.
Both sides of the diff are the **normalized book-level rows** — one netted position per symbol across granted accounts and manual supplements ([schwab-integration.md §What is pulled](schwab-integration.md#what-is-pulled)) — so a same-symbol row in a second account can never read as a separate position or silently overwrite the first.
Every current position is classified **by quantity** as **new**, **increased**, **decreased**, or **unchanged** — by *position size*: **absolute quantity for a same-side move, the signed swing on a sign flip** — so a short reads correctly (adding to a short is an increase, buying it back a decrease) and a net long↔short reversal is never mistaken for no change even at equal magnitude (a reversal additionally force-includes the holding in a selective run — [§Triggering](#triggering)).
Cost basis is **corroborating context for a quantity move, not a second classification axis** — it is surfaced alongside an **increase** or **decrease** (paid-up vs averaged-down), while a cost-basis-only move at unchanged quantity is a non-trade accounting event (a return-of-capital distribution, a wash-sale adjustment), not something the user did, so it neither retags the position nor surfaces as a change.
That delta rides into the holding's dossier so the verdict reasons over what the user actually did — added to, trimmed, or left a position — rather than re-grading it in a vacuum.
A symbol present last run but **absent now** is an **exited** position: it earns no per-holding verdict (there is nothing left to grade), but it is surfaced in the portfolio roll-up as closed-since-last-run, so a sold-out name is acknowledged rather than silently vanishing from the run.
The diff is the app's, not the model's — the same compute-don't-guess boundary the rest of the pipeline holds.

## The position thesis ledger

Where holdings-change tracking remembers *what the user did* with a position, the **thesis ledger** remembers *why the job holds a view on it* — a compact, persisted, per-holding record carried forward run to run, the Portfolio analog of Trade Opportunities' opportunity graph ([trade-opportunities.md §Discovery memory](trade-opportunities.md#discovery-memory-the-opportunity-graph)) but **downstream and holding-specific**: it tracks an owned position rather than discovering a new one.
Without it, each run re-derives its conviction from scratch and continuity is only a value-by-value diff; with it, a run tests the *standing thesis* against fresh evidence and says precisely which part broke.

Each gradable holding carries a ledger with:

- **Thesis** — the original thesis (when the position first entered an analysis) and the current thesis, so drift is legible.
- **Key drivers** — the few variables the thesis actually depends on (e.g. data-center revenue growth, gross-margin trajectory, net retention), each tied where possible to an engine-tracked metric so the next run can read whether it moved.
- **Bear / base / bull monitor** — three compact scenarios, each with the conditions that define it, the engine's scenario price target for it, and a rough probability lean.
  The monitor names, per scenario, **what must improve** to migrate toward the bull case, **what must not break** to stay in the base case, and the conditions that would force a downgrade or license an upgrade — so the next run's continuity check has explicit goalposts rather than a gestalt.
- **Key falsifiers** — monitorable conditions that would invalidate the thesis (the Portfolio analog of an opportunity's key falsifiers), each typed as **quantitative** (engine-checkable against the input delta — e.g. "operating margin falls below 30%") or **qualitative** (research/model-checkable — e.g. "a credible second source ships at scale").
  Each quantitative falsifier — and each quantitative action trigger below — carries the suite's shared **persistence and condition-identity contract**, stated once — contract and drafted constants — at [trade-opportunities.md §The opportunity](trade-opportunities.md#the-opportunity): the materiality margin and consecutive-observation count, the observation-identity-keyed evaluation state that advances only on a new distinct print or filing, the acknowledgment transition that keeps a cleared flag from re-raising off an already-examined observation, and the stable app-assigned **`condition_id`** whose state survives a rewrite when the machine-evaluable core is unchanged and is superseded — fresh streak, `supersedes` link — when it isn't. Portfolio applies that contract at its own seams: **the app — not the model — decides what carries when the interpretation stage rewrites the ledger** ([§The per-holding pipeline](#the-per-holding-pipeline) Step 5), and the continuity check rejects carried-state claims that don't structurally match, like any other unvalidated rewrite (Step 6).
  And each condition is **tagged by its series' cadence** (market-data vs filing), so the quick check evaluates market-data conditions on every pass and filing-cadence conditions when a fresh observation of their series has landed — a new EDGAR filing for a stock condition, a changed `etf/info` print for a fund-ledger condition ([§The quick check](#the-quick-check-engine-only)).
  A **third-party technology event** that threatens a holding (a competitor's or supplier's product / standard announcement) is a first-class **qualitative falsifier class** here: when one fires, it is recorded with the same sized **`technology_read`** the Trade Opportunities lens uses (substitute / complement / mix-shift, the *actually-exposed* revenue / profit pool, deployment timeline, switching costs), so "is this a real impairment or a panic-priced threat?" is tracked as a monitorable condition rather than a vague worry ([§The per-holding pipeline](#the-per-holding-pipeline) Step 3; [trade-opportunities.md §The event-impact / value-chain repricing lens](trade-opportunities.md#the-event-impact--value-chain-repricing-lens)).
- **Action triggers** — explicit **add / trim / sell triggers** and a **target-weight range**: the pre-committed conditions under which the standing view would change the position (e.g. "trim above 8% portfolio weight," "add on a pullback to the base-case downside with the thesis intact").
  The triggers are *intrinsic* conditions on the holding; the portfolio-construction stage ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)) still owns whether a fired trigger actually sizes, given the rest of the book.

A **fund** holding carries a ledger of the same sections but a different flavor of content, matching its reduced engine path ([§Asset eligibility](#asset-eligibility)): its **drivers** are exposure tilt, expense drag, mandate / tracking fidelity, role in the portfolio, and house-view fit — not company operating metrics — and its falsifiers and triggers are written against those (e.g. "mandate drift," "expense ratio rises above the category," "the exposure this fund supplies is now better held directly").
**The ledger is additionally typed by verdict branch** ([§Intrinsic verdict](#intrinsic-verdict)): a **`priced`** holding — stock or fund — carries the full shape above, while a **`role_risk_only`** holding's ledger keeps the same sections with two reductions — its **monitor scenarios are condition-only** (each keeps its defining conditions and probability lean but carries no engine scenario price target, because no target exists on that branch) and its **triggers are trim / sell only** (an add trigger would pre-commit to an action its {sell all, trim, hold} feasible set never offers — [§Portfolio action](#portfolio-action)).

The ledger is **read at dossier assembly and re-evaluated and rewritten each run**, across the pipeline's deterministic and model stages ([§The per-holding pipeline](#the-per-holding-pipeline)): the **engine** (as part of its input delta) tests which **quantitative** falsifiers and triggers crossed this run — under their persistence semantics, so a crossing is a first-breach note until confirmed; the **interpretation stage** reads those crossings, judges the **qualitative** conditions from research, and rewrites the thesis, re-weights the scenarios, and re-sets the triggers; and the **continuity check** validates the rewrite.
A confirmed falsifier breach or fired trigger is a first-class continuity signal held to the same app-validation as the what-changed audit — it must resolve to a real input-delta crossing or a source-backed finding, so the ledger cannot be quietly rewritten to fit a new verdict.
The ledger is persisted with the run and survives within the job's retention like the rest of a holding's continuity (see [§Storage and display](#storage-and-display)).

## The quick check (engine-only)

The thesis ledger names monitorable conditions, but a full run is the only thing that tests them — and a full run is potentially hours long, so verdicts would otherwise age silently between runs.
The **quick check** closes that gap with Trade Opportunities' warn-don't-decide split ([trade-opportunities.md §The two jobs](trade-opportunities.md#the-two-jobs)): a cheap, engine-only pass that keeps the ledger *live* between full runs, whose outputs are deliberately **non-destructive** — the attention flag, the quiet evidence-event badge, and updated condition evaluation state; never a rewritten verdict or any model-authored content.

It runs over the **last run's holdings snapshot and ledgers** (no Schwab pull — it tests theses, not the book) and retrieves **by asset type**: prices for every holding (the job-time FMP `quote` plus the shared Stooq cache) and the run-level **`DGS2` print** (one FRED call on the **shared FRED credential** — [configuration.md](configuration.md); a presence precondition of the local-suite gate ([portfolio-workflow.md §Step 1](portfolio-workflow.md#step-1-job-start-and-gate)), so always configured where Portfolio runs at all, and a failed call **fail-softs to the last run's cached print** with its as-of date); per **stock**, the SEC EDGAR filing check (CIK-gated — [data-sources.md §SEC EDGAR](data-sources.md#sec-edgar)), the lightweight **revision preflight** (an `analyst-estimates` snapshot), an **`earnings` re-pull**, a **statement-and-`dividends` re-pull where the EDGAR check surfaced a new filing** (the income / balance-sheet / cash-flow series plus the dividend record — so a filing-cadence condition's fresh observation arrives with the *value* the condition reads, not just the fact of the filing, and the hurdle's payout legs — dividend and buyback yield both — refresh on filing cadence, where they mostly move: declarations typically ride quarterly earnings, and a mid-quarter change *often* files as an 8-K — often, not always, since the 8-K item is optional, so a payout change with no accompanying filing waits for the next full pass rather than being guaranteed-fresh here), and a **`news/stock` pull for holdings carrying a standing technology-class falsifier** (the qualifying-news-seed leg) — the calls that make the evidence-event list's legs deterministically detectable (the estimate and earnings series are company-scoped, so those legs are equity legs — [§Starting parameters](#starting-parameters-calibratable)); and per **fund**, a light **fund refresh** — `etf/info` for every fund, plus sector / country weightings per **equity** fund on either verdict branch (the US-guard crossing flips the branch in **both** directions — a priced fund can fall below it, a role-only international equity fund can rise above it — and a material sector shift is an evidence event wherever exposure is load-bearing) and, for a non-equity fund, where a ledger condition reads exposure — so fund-ledger conditions like an expense-ratio rise, and the fund evidence-event legs, are evaluable.
It then evaluates deterministically, per holding:

- every **quantitative falsifier and action trigger** in the ledger, under the persistence semantics ([§The position thesis ledger](#the-position-thesis-ledger)) and at each condition's cadence — market-data conditions on every pass, filing-cadence conditions when a fresh observation of their series lands (a stock's new EDGAR filing; a fund's changed `etf/info` print) — a **confirmed breach** or a **fired trigger** flags the holding, while a lone noisy print logs a quiet first-breach note;
- the **capital-efficiency / dead-money read** — **priced verdicts only** (the `role_risk_only` branch carries no hurdle read) — re-derived on a **stated basis**: the fresh price and `DGS2` print against payout and growth legs refreshed only when a new filing lands (the statement-and-dividends re-pull above — the cadence those inputs actually move on); a holding newly crossing into the hurdle's **fails** state flags (an indeterminate read never flags);
- the **scenario bands** — likewise **priced only** (a `role_risk_only` ledger carries no band) — the live price crossing outside the ledger's bear–bull band flags (the scenario read is stale in a way worth a fresh look).

A flagged holding surfaces an amber, actionable **attention flag** on its card — the Portfolio analog of Trade Opportunities' *Consider Deep Audit* warning — suggesting a selective re-analysis.
The flag (and which trigger raised it) persists with the holding until the next **successful full pass over it** (a full run, or a selective run that includes it — [storage.md §Local Analysis Suite Storage](storage.md#local-analysis-suite-storage)); that pass consumes the triggering observation in interpretation / continuity, records it in the verdict and ledger audit, clears the flag on completion, and stamps the **acknowledging observation** into each confirmed condition's evaluation state.
A full pass therefore **never creates a fresh persisted attention flag from the same engine evaluation it just interpreted**: the condition is already part of that pass's authored result.
The acknowledged breach cannot re-raise on the next quick check — it re-raises only on a breach confirmed against a *later* observation (the shared contract's acknowledgment transition — [§The position thesis ledger](#the-position-thesis-ledger)).
Separately from the amber flag, the quick check surfaces an **unexamined evidence event** its sweeps see, with no full pass since ([§Starting parameters](#starting-parameters-calibratable) defines the list), as a **quiet, informational badge** in the *Research stale* family, never the amber action color: every holding reports quarterly, so folding this into the attention flag would blanket the book each earnings season and dilute the flag reserved for actual problems.
The quick check **never rewrites a verdict, grade, or any model-authored ledger content** — with one deliberate carve-out: it **updates each condition's evaluation state** (the observation-identity-keyed breach streak, first-breach and confirmation timestamps — [§The position thesis ledger](#the-position-thesis-ledger)), which is engine state, not authored content, exactly as Trade Opportunities' cheap re-derivation freezes the authored fields while running the evaluations.
Streaks advance **only on a distinct new observation** (a new trading day's print, a new filing), so repeated quick checks against the same quote or filing can never confirm a breach.

**A failed retrieval is typed, never a silent clear.**
Each holding's sweep concludes in one of three states per required signal family — **`fresh_clear`** (a new observation was evaluated and nothing fired), **`flagged`**, or **`unknown`** (the family's retrieval failed and the cache held no newer observation, so the sweep could not vouch either way: a fail-soft to already-evaluated cached data produces no new observation, which is a non-detection, not a clear).
The `unknown` state is load-bearing for selective runs: the holding is **force-included into the work list exactly like a flagged one** ([§Triggering](#triggering)) — the sweep's job there is to vouch for the carried verdicts, and a verdict it couldn't check must never stand on the sweep's silence — and the card carries a quiet **degraded-sweep note** naming the families that couldn't be checked (the *Research stale* badge family, never the amber action color — [interface.md §Connection status](interface.md#connection-status-local-suite)).
For a stock, the filing family additionally reads `unknown` when the symbol's **CIK is unresolved** ([data-sources.md §SEC EDGAR](data-sources.md#sec-edgar)), so a missing filing feed surfaces as unverifiable rather than reading as no-new-filings.
It holds the single global run slot and runs in the run tracker like any job; because it makes **no model call** it skips the daemon-connectivity check and runs configured-but-down, and because it does **no web research** it triggers no pre-run SearXNG notice ([interface.md §Connection status](interface.md#connection-status-local-suite)).
The **qualitative** falsifiers stay a full pass's job — they are research-checkable only.

## The per-holding pipeline

This section covers the per-holding pipeline's **design**; the full job's **control flow** — every step, its Type (Computed / API retrieval / Local-model call), and each local-model-call contract — is the canonical spec in [portfolio-workflow.md](portfolio-workflow.md) (the per-holding loop is its Step 6; the six numbered stages below map to workflow sub-steps **6a–6g**, with stage 5 here spanning the workflow's separate target-refinement [6e] and interpretation [6f] steps).

Each holding is processed through a chain of distilled, schema-validated hand-offs (see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)):

1. **Dossier assembly (application layer — deterministic, with a non-reasoning embedding lookup for continuity retrieval).**
   The app — not the model — assembles the holding's evidence packet, holding the pure-stage boundary (the only model touch is the fixed embedder vectorizing the continuity query — no reasoning; see [portfolio-workflow.md §Step 6a](portfolio-workflow.md#step-6a-dossier-assembly)): the position and its prior-run **position delta** ([§Holdings change tracking](#holdings-change-tracking)); any **same-underlying option positions** from the same Schwab pull, linked deterministically by underlying symbol and carried as a **typed overlay** — `{contract, direction, quantity, strike, expiry, delta, coverage ratio}`, classified **covered-call / protective-put / collar / other** (a naked short call must never read as covered; an unrecognized multi-leg reads `other` with its net delta) — because the overlay changes what the right action is, so the verdict and the sizing spine see it rather than only the roll-up's book-level options aggregate; the per-symbol **fundamentals / revision / positioning / price** surface (equity holdings draw the full FMP + SEC + Stooq surface, funds the reduced ETF surface — [§Asset eligibility](#asset-eligibility); every endpoint, by source and cardinality, is in [data-sources.md §Portfolio Analysis — endpoint surface](data-sources.md#portfolio-analysis--endpoint-surface) and [portfolio-workflow.md §Step 6a](portfolio-workflow.md#step-6a-dossier-assembly)), with **SEC EDGAR** the authoritative cross-check behind FMP's working feed; the Market Signal **house view**, loaded *deterministically* (the report's own sections + summaries, never vector-searched — [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)) and **freshness-gated** (dropped past one week — [portfolio-workflow.md §Step 5](portfolio-workflow.md#step-5-load-shared-context)); and the **prior run's verdict and thesis ledger** ([§The position thesis ledger](#the-position-thesis-ledger)).
   Three market inputs — the **risk-free rate**, **commodity prices**, and **CFTC positioning** — are fetched **once per run and shared across every holding** rather than re-requested per symbol, the load-relief that keeps the per-holding budget small.
2. **Deterministic financial analysis (application layer).**
   The shared Rust engine (also Trade Opportunities' — [trade-opportunities.md](trade-opportunities.md)) computes every number in three layers, the load-bearing decision being *which layer touches the grade*: **(a) the grade core** — for a stock, fundamentals and valuation multiples → the real **quality / valuation / risk** sub-scores; for a priced equity fund, the reduced engine → real **valuation / risk** plus the neutral-imputed absent quality axis under the [fund-grade contract](#asset-eligibility) — and scenario **price targets** (discounted off the run-level risk-free rate); **(b) a conviction layer** — revision velocity, rating drift, earnings surprise, the engine's **forensic flags**, and the **momentum / market-setup read** (price trend and setup, computed alongside the grade core but held **out of the letter** — context that can temper or corroborate conviction, never raise it) → the **narrative-vs-reality ratio** (revision pace vs multiple — *justified-expensive* vs *hype*; falling back to the **operating-reality-vs-price** form where analyst coverage is too thin to read revisions — [trade-opportunities.md §The two non-negotiables](trade-opportunities.md#the-two-non-negotiables)) and the **implied-expectations read** — the growth / margin trajectory the current price *already assumes*, solved by inverting the engine's scenario math at the live price and expressed as a **range under stated assumptions**, never one number (the shared primitive Trade Opportunities computes at Step 5c) — the priced-in anchor the trim-a-winner judgment leans on; and **(c) positioning context** — insider / congressional / FINRA short interest / the Schwab options signal.
   Layers (b) and (c) are **computed but held *out* of the sub-scores** (positioning until the outcome-learning scorecard proves it adds value — [§Outcome learning](#outcome-learning-calibration)), so new signals sharpen *conviction and risk* without moving the letter grade — the three-layer engine's whole point.
   From the targets the engine derives the **capital-efficiency / dead-money read** — the base-case forward **total return** (expected price drift **plus shareholder yield**: trailing dividends + net buybacks) against a DGS2-anchored, risk-tier-scaled hurdle, returning the three-state **clears / indeterminate / fails** verdict (only *fails* is dead money — [§Starting parameters](#starting-parameters-calibratable)) — also kept out of the grade and fed to the action-sizing spine ([§The holding verdict](#the-holding-verdict)); the targets and this read are **provisional**, recomputed if research refines the base case (Step 5), while the backward-looking sub-scores are final.
   The engine also diffs this run against the prior run's stored values into a deterministic **input delta** — the evidence the continuity audit (Step 6) holds every verdict move to, and the same delta that evaluates the thesis ledger's **quantitative** falsifiers and triggers under their persistence semantics, and that sets the **technology-event pre-flag** when the holding's sector-relative move exceeds the volatility-scaled threshold ([§The position thesis ledger](#the-position-thesis-ledger), [§Starting parameters](#starting-parameters-calibratable)).
   The dossier's **absolute street opinions** (consensus target *level*, buy / hold / sell rating, FMP's ratings snapshot) are deliberately **not** engine inputs — they ride into interpretation (Step 5) as *evidence the model weighs against the engine's own read*, never numbers to adopt.
   These computed values, not the model, are the source of every number in the verdict ([local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)).
3. **Bounded web research** — the 122B reasoner (thinking mode) over the bounded per-topic research loop ([web-research.md](web-research.md)), fail-soft, grounded by the deterministic SEC/FMP financials so research *fills the gaps the numbers leave* rather than substituting a story.
   Its agenda: competitive / business position, recent results and estimate revisions, catalysts and risks; **management quality and capital allocation** (guidance *delivered* vs *promised*, candor in bad quarters, buyback / dividend behavior — the held-name form of Trade Opportunities' investor-judgment lens, [trade-opportunities.md §The lenses](trade-opportunities.md#the-lenses)); the **market narrative and sentiment** as a *sourced qualitative* read (how much of the price reflects emotion about what *might* come versus present fundamentals — distinct from the quantitative options / momentum proxies); and **forward opportunity and thematic fit** (TAM, optionality, theme exposure).
   A **fund** holding's agenda swaps these company-centric topics for fund-flavored ones matched to its ledger's driver set ([§The position thesis ledger](#the-position-thesis-ledger)): mandate / strategy and manager changes, expense and structure versus its category, how well the exposure it supplies fits the house view — and whether it would be better held directly — and, for a closed-end fund, the discount and distribution coverage; the conditional technology-event topic below is an **equity topic** (its pre-flag and `technology_read` size a single company's exposure, not a wrapper's).
   **Conditionally**, when the holding moved on a third-party **technology event** (a competitor's or supplier's announcement that repriced the name, the same materiality gate Trade Opportunities applies), the agenda adds a **technology-event impact assessment** — reading the actual technology and *sizing the holding's real exposure* into a typed **`technology_read`** ([trade-opportunities.md §The event-impact / value-chain repricing lens](trade-opportunities.md#the-event-impact--value-chain-repricing-lens)) — which is what separates a **panic drop from a genuine impairment** (and, symmetrically, overstated-benefit euphoria on the upside); it feeds the conviction read and a thesis-ledger falsifier ([§The position thesis ledger](#the-position-thesis-ledger)), dormant otherwise.
   **The trigger is defined, not assumed** — Portfolio runs none of the discovery news routes that fire Trade Opportunities' materiality gate, so the conditional topic is added by any of: the engine's deterministic **event pre-flag** in the input delta (a sector-relative move exceeding the volatility-scaled threshold — [§Starting parameters](#starting-parameters-calibratable)); a **technology-class falsifier already standing in the ledger**; the symbol-scoped structured **news-feed seed** surfacing a qualifying announcement ([data-sources.md §Portfolio Analysis — endpoint surface](data-sources.md#portfolio-analysis--endpoint-surface); a seed is a lead, never evidence — [web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management)); or **mid-loop escalation**, where another topic's findings surface the event and the model proposes the follow-up, which the orchestrator approves like any follow-up pass.
   Research effort is also **graduated, not uniform**: a holding qualifies for **research reuse** — the loop and distillation skip, its cached distilled findings and still-credited forward assumption carried into interpretation — when its research is younger than the **~4-week freshness window** *and* nothing moved (position unchanged, no attention-trigger condition, no event pre-flag, and no new evidence event — any leg of the [§Starting parameters](#starting-parameters-calibratable) evidence-event list invalidates the cache); anything moving, or research aging past the window (the built-in max-age), earns the full agenda, and every reuse decision is logged in the audit record ([§Starting parameters](#starting-parameters-calibratable)).
4. **Distillation** — the reasoner in **non-thinking mode** consolidates each topic's **complete** findings into the compact object interpretation reads (never a re-distillation of in-loop summaries), single-pass or hierarchical by the deterministic shared rule ([web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management)); Portfolio has no cross-lens contradiction check, so the reduce here is pure consolidation.
   Running it on the same resident 122B avoids per-holding model swaps — the 35B fast tier is a benchmark-gated option ([local-models.md §The model roster and per-task routing](local-models.md#the-model-roster-and-per-task-routing)).
5. **Interpretation and grading.**
   First a **deterministic target refinement**: if distillation surfaced a typed **`research_forward_assumption`** (a sourced forward fact the structured feeds lacked), the **engine — not the model —** recomputes the affected scenario target with it as an explicit, logged assumption and refreshes the capital-efficiency / dead-money read, so the forward view reflects research while the number stays engine-computed (the refinement contract is [portfolio-workflow.md §Step 6e](portfolio-workflow.md#step-6e-deterministic-target-refinement); the backward-looking sub-scores are untouched; a `role_risk_only` holding **skips the refinement** — it has no scenario targets for an assumption to move).
   Then the 122B in thinking mode interprets the refined analysis and the distilled research into the **intrinsic verdict** — grade, conviction, horizon, the justified base-case target, and a **standalone action lean** (what it would do if the holding stood alone, before portfolio context; for a `role_risk_only` holding the same call authors the union's other branch — the role / risk assessment and its rewritten fund ledger, [§Intrinsic verdict](#intrinsic-verdict)) — **reading every number from the engine, never inventing one**, with the *final* portfolio action deferred to the construction stage that has the whole book in view ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)).
   This stage also **rewrites the thesis ledger** — thesis, bear/base/bull monitor, falsifiers, and triggers — in light of what the run learned.
6. **Continuity check.**
   An **app-layer validator**, not just a recorder: every verdict move interpretation labels **external** (market data, company information, or research / narrative) must resolve to a concrete **input-delta** entry, a source-backed research finding, or the logged forward assumption — otherwise it is **downgraded to a labeled self-correction** (or fails schema validation), so a no-new-facts swing can't be laundered as "the market changed."
   Output stays firm run-to-run, the self-correction the earned, flagged exception ([thesis-continuity.md](thesis-continuity.md)).
   The same rule **validates the thesis-ledger rewrite** — every falsifier marked tripped and every trigger marked fired must map to the engine's deterministic crossing (quantitative) or a source-backed finding (qualitative), so the ledger can't be quietly rewritten to fit the verdict ([§The position thesis ledger](#the-position-thesis-ledger)).

## The holding verdict

Each holding's output is a structured, schema-validated record that deliberately keeps two things apart — the **intrinsic verdict** (what the holding is worth on its own merits) and the **portfolio action** (what to do about it given the rest of the book).
A holding can be an A-grade business and a *trim* because it is oversized, crowded, or duplicative; a B-grade business can be an *add* because it diversifies the book at better risk-adjusted upside.
Collapsing the two hides that reasoning; separating them — and auditing each on its own — is what keeps an "A-grade" from silently becoming "add aggressively."

Inside a **`priced`** intrinsic verdict, two reads are likewise kept **orthogonal** — the **grade** (a *backward / current* quality anchor, computed deterministically from the trailing financials and valuation) and the **forward outlook** (a *forward* scenario read — the horizon outlooks and scenario price targets, refined by research).
They answer different questions: *what the business is* versus *what the market is likely to pay for it next*.
A low grade with a strong forward outlook is a **legitimate, first-class state, not a contradiction** — the market routinely prices the future, so a C- or D-graded business mid-inflection can carry a constructive outlook; the card shows the two side by side and never collapses the forward read into the letter (or the letter into the forward read).
This is the per-holding form of the suite's **four-part model** — **grade** (backward anchor) · **forward outlook** (scenario, research-informed) · **conviction** (the evidence-weighted synthesis, which moves *both* up and down) · **action** (the portfolio-aware decision over all three).
One **governing invariant** binds it: research may lift the forward outlook and conviction, but a conviction *raise* must trace to a *countable, dated, third-party leading indicator the engine's structured feeds did not already score* — **never** to price action or narrative, and never to a metric the engine already counted (the anti-reflexivity / no-double-count guard the suite holds throughout — [trade-opportunities.md §Outcome learning](trade-opportunities.md#outcome-learning-calibration)).

### Intrinsic verdict

Decided in the per-holding loop with **no whole-book context** — it is a property of the holding alone, **and of no investor**: every intrinsic field — the disposition, grade, sub-scores, conviction triple, targets, horizon reads, dead-money state, and the standalone lean — is **profile-independent**, so the same security and evidence produce the same intrinsic record for any investor (what keeps outcome learning, whose intrinsic calibration keys on the standalone lean, and cross-run comparison honest).
The investor profile enters at portfolio construction only ([§Portfolio action](#portfolio-action)) — the intrinsic loop never sees it.
What a holding's analysis resolves to is first an **outer three-arm disposition** — kept distinct so the UI and the roll-up can tell a holding that *was* analyzed from one that *can't* be (wrong asset class) and one that *shouldn't* be (too little evidence): **`not-rated`** (an ineligible class — [§Asset eligibility](#asset-eligibility)), **`insufficient-evidence`** (eligible but below the evidence floor — the explicit abstention of [§Evidence floor](#evidence-floor), never a low-conviction guess and never a conviction *value*), or an **analyzed verdict** (the as-built `VerdictDisposition` in `portfolio/mod.rs` already carries these three arms — its `graded` arm predating the two-branch union below).
The analyzed verdict is then a **discriminated union of two branches**, and every downstream schema branches on the type — the sizing spine, the outcome episodes, and the calibration reads.
The default **`priced`** branch is the full record below.
A holding whose vehicle class is structurally unpriceable ([§Asset eligibility](#asset-eligibility)) returns the **`role_risk_only`** branch instead: the **role** read (the vehicle's mandate and the exposure it exists to supply, read **in isolation** like every intrinsic field — what it does for *this* book is the construction stage's whole-book judgment), the **exposure tilt**, the observable **risk** read, expense drag, the **structural flag and evidence gaps**, and the rewritten fund ledger — with **no letter, no price targets, no implied-expectations read, no dead-money state, no standalone lean, and no decomposed conviction** (none of the return-derived machinery has an input, and the conviction triple's caps and raises are defined over revisions, targets, and thesis drivers this branch doesn't produce — its confidence surface is the **evidence gaps and degraded-input flags**, never a fabricated High / Medium / Low; its action is set wholly at construction from the reduced spine — [§Portfolio action](#portfolio-action)).
The `priced` branch:

- **Composite grade** (A–F) over the branch's applicable deterministic sub-scores: a stock supplies **quality**, **valuation**, and **risk**; a priced equity fund supplies real **valuation** and **risk**, with the structurally absent quality axis represented only by the shared neutral-50 imputation and visible low-confidence marker defined in [§Asset eligibility](#asset-eligibility).
  **Momentum is computed alongside but lives outside the letter**, surfaced as the market-setup read in the conviction / positioning context: a price move is not a business fundamental, and the letter is presented as *what the business is* ([§Starting parameters](#starting-parameters-calibratable) — as-built the letter still carries momentum, with the settled re-weight pending).
- **Conviction** — the verdict's confidence, and the one read that moves in **both** directions.
  It is **lowered** when evidence is thin or degraded and **capped** by the **narrative-vs-reality** read (a price driven by multiple-expansion the estimates don't support caps conviction even on a strong company — falling back to the operating-reality-vs-price form where coverage is too thin to read revisions, so the cap neither vanishes nor fires on noise for a thinly-covered holding), and below the evidence floor the holding abstains entirely — the outer `insufficient-evidence` disposition above, never a conviction value (see [§Evidence floor](#evidence-floor)).
  It is **raised** only when research validates a *countable, dated, third-party leading indicator the engine's structured feeds did not already score* — a confirmed thesis-ledger **key driver** moving *ahead of* the trailing financials (the held-name case for the very C-/D-graded-but-inflecting business the grade alone misreads).
  A raise is **bounded (≤ one band), cited to that indicator, and logged**, and is held to the same continuity validation as any other verdict move ([portfolio-workflow.md §Step 6g](portfolio-workflow.md#step-6g-continuity-check-and-checkpoint)).
  Price action or narrative on its own can **never** raise it — the governing invariant above.
- **Forward outlook** — the holding's *forward* read, kept distinct from the backward grade and surfaced beside it: separate short-, mid-, and long-term directional reads, together with the scenario price targets below — read against the engine's **implied-expectations range** (what the current price already assumes — [§The per-holding pipeline](#the-per-holding-pipeline) Step 2), so "strong outlook, already paid for" is a computed contrast, not a vibe.
  A grade/outlook divergence — a weak business with an improving setup, or a strong one the market has already paid for — is shown as such rather than averaged into the letter ([§The holding verdict](#the-holding-verdict) intro).
- **Price targets** — **one-month and twelve-month**, each a rolling window from the run date (the settled rename of the as-built end-of-month / end-of-year fields — [§Starting parameters](#starting-parameters-calibratable)), computed by the financial-analysis engine as scenario outputs with their **methodology and assumptions exposed** (including any research-sourced forward assumption folded in post-research — see [§The per-holding pipeline](#the-per-holding-pipeline) Step 5), aligned to the ledger's bear/base/bull monitor; the model selects and justifies the base case rather than inventing the number.
- **Standalone action lean** — what the action *would* be if the holding stood alone (a pure read of grade × conviction × upside/downside × risk — plus the engine's **capital-efficiency / dead-money** read — on the same ladder), set **before** portfolio constraints.
  A holding whose total-return hurdle read is **fails** (the three-state read — even the bull case misses; an *indeterminate* read tilts nothing) leans toward **exit here, on its own merits**; the **possible-tax-benefit and redeployment** counterweights that firm up that exit are applied with the whole book in view at construction ([§Portfolio action](#portfolio-action)).
  It is the input the construction stage reconciles, and showing it beside the final action is what makes the intrinsic-vs-portfolio split visible.
- **Financial analysis** — a concise read of the company's financial health.

### Portfolio action

Decided **after** the roll-up has the whole book in view (see [§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)), so the action reflects the portfolio, not just the holding:

- **Action**, on a fixed ladder: **sell all → trim → hold → add → add aggressively**, made concrete with a **target portfolio-weight range and an estimated share/dollar adjustment**.
  No orders are ever placed.
- **The action-sizing spine.**
  The action and its size are reconciled from an explicit, mostly-deterministic input set rather than a model's gestalt: the **grade** and **conviction**, the **upside / downside** the targets imply against the current price, the engine's **capital-efficiency read** (the holding's forward base-case **total return** — price drift plus shareholder yield — against a hurdle: the run-level risk-free rate plus a **risk-tier-scaled** premium — flagging a position as **dead money** only on the hurdle's **fails** state, when even the bull case misses it; an indeterminate read flags nothing), the **risk tier**, the **existing weight**, the **concentration headroom** against the profile's limits, the **correlation / exposure overlap** with the rest of the book (a name held directly and inside a fund, or several names sharing one factor, size down together), the holding's typed **same-underlying option overlay** (a covered call caps the upside the targets imply; a protective put trims the downside; the classification and coverage ratio ride the dossier from the Step-2 pull), and the **cash / buying-power and tax** constraints.
  The engine computes these and **bounds the feasible action set** — *add aggressively* is offered only when buying power and concentration headroom support it; a *sell all* sitting on a large taxable gain carries that cost into the call — and the model chooses within those bounds and justifies it.
  (The buying-power bound is **profile-controlled**: under the current fixed preset, cash is treated as **unconstrained**, so that bound is **inert** and only concentration headroom binds *add aggressively* — see [configuration.md §Investor Profile](configuration.md#investor-profile); a future configurable profile can re-enable it.)
  A **`role_risk_only`** holding ([§Intrinsic verdict](#intrinsic-verdict)) runs a **reduced spine** — existing weight, concentration headroom, overlap / exposure contribution, observable risk, the option overlay, unrealized P/L, and tax; none of the return-derived inputs exists — and its feasible action set is **{sell all, trim, hold}**: the add family requires return evidence, which this branch has none of by construction ("add to the sleeve" is an asset-allocation call outside the job's mandate), so its actions carry only portfolio-role and risk rationale.
  The spine is what prevents a strong grade from sizing past what the portfolio can bear.

  **Capital efficiency and the sunk-cost guard.**
  A holding flagged **dead money** is where the spine deliberately overrides the disposition reflex.
  The instinct to hold a deep loser "until it comes back" — or because selling "wouldn't free much cash" — treats the unrealized loss as a reason to stay; the spine treats it as the opposite.
  When the hurdle read is **fails** — even the bull case misses over the holding's horizon — **realizing the position (some or all) is leaned toward, not away from**, and two *generic* counterweights support that exit beyond the weak forward read: the **possible tax benefit** of booking the loss, and the **redeployment optionality** of the freed cash — worthwhile even when the proceeds are modest, since dead capital earns nothing.
  These are weighed **high-level**: the job models neither the user's account type and tax rate nor a specific replacement name (naming the next idea stays Trade Opportunities' job, kept isolated — [trade-opportunities.md](trade-opportunities.md)); it **assumes the user acts on the specifics appropriately**.
  Concretely, the tax counterweight enters the action's **rationale text as a flagged user consideration — it never moves the chosen rung on its own** (with account type, lots, and rates unmodeled, it has no standing as an analytical vote).
  Two disciplines keep this from becoming a license to dump every red position: the unrealized loss **never moves the grade or sub-scores** — those stay cost-basis-agnostic, a read of the business, never of the position's P/L ([§Intrinsic verdict](#intrinsic-verdict)), so it informs only the *action and its rationale* — and the lean applies **only once the forward prospects are independently judged poor**.
  A loser with an intact, improving thesis is still a hold (or an add); the lean to realize-and-redeploy is reserved for a position the forward read has already written off.

### What changed

A structured **audit of the prior verdict**, not a bare diff, **split to match the separation above**:

- The **intrinsic** half attributes every moved value (grade, any sub-score, conviction, target, horizon, a re-weighted scenario, a tripped falsifier or fired trigger — for a `role_risk_only` holding, its role / exposure / risk reads and ledger conditions) old → new to an **external change** — **market data** (price / momentum / positioning), **company information** (earnings, guidance, a new filing, a fundamentals shift), or **research / narrative** (a catalyst or narrative change), tied to the specific engine input-delta entry or source-backed finding that moved — or to a **self-correction**, where the inputs did *not* materially change and the model is revising its own prior read.
- The **action** half attributes a changed action or target weight to either a **moved intrinsic verdict** *or* a **moved portfolio context** (the holding became oversized, cash was freed for a higher-conviction name, an overlap emerged) — so an action can change with the intrinsic verdict unchanged, and that reads honestly rather than as a mystery re-grade.

A self-correction is the **flagged exception** to firm-output continuity — it swings the verdict with no new facts — so it is labeled as such rather than dressed as an external move, and repeated self-corrections on one holding are a calibration signal.
The line also carries the **position delta** (new / increased / decreased / unchanged since last run; an unseen holding reads as `new`).

The fixed action ladder and grade vocabulary are load-bearing: like the report's fixed regime labels ([storage.md](storage.md)), they keep verdicts comparable across runs and prevent the model from retreating into hedged, non-comparable language.
The grade is **intrinsic** to the holding; the **action** additionally reflects the investor profile (horizon, risk tolerance, objective, tax sensitivity), available cash, and the whole-book context the construction stage supplies.

## Portfolio roll-up and construction

After the per-holding pass produces every **intrinsic verdict** and standalone action lean (none for a `role_risk_only` verdict — its action arises wholly at construction), a final stage takes the whole book in view and does the two things the per-holding loop structurally cannot — because that loop decides each holding before any other holding's verdict exists.
It runs in two parts.

First, **deterministic whole-book aggregates** (application layer): concentration and sector / factor exposure (with fund exposure folded in at the **sector / country level** from each held fund's weightings — single-name look-through into fund constituents is off-plan with `etf/holdings`, so a name held both directly and inside an ETF aggregates at the **exposure** level, not per-name, unless SEC N-PORT supplies constituents), correlation / overlap clusters, the cash / buying-power position, **the risk / exposure contribution of any material not-rated positions** — computed from what the Schwab position payload actually carries: **market value and signed notional** (an option's notional derives from quantity × strike × contract multiplier via its OCC symbol), with **fixed-income duration / credit and a standalone option's delta recorded as typed gaps**, never computed numbers (no on-plan source carries them — the same honesty as the bond-fund valuation gap, [§Asset eligibility](#asset-eligibility); the one exception is a held underlier's **overlay** delta, read from that holding's own chain), graded nowhere but real exposure — and, per holding, the **action-sizing spine inputs** ([§The holding verdict](#the-holding-verdict)) — existing weight, concentration headroom, overlap contribution, upside/downside, the capital-efficiency / dead-money read, unrealized P/L (a harvestable loss or a taxable gain, by sign), risk tier, tax (a `role_risk_only` holding supplies the reduced spine, without the return-derived inputs — [§Portfolio action](#portfolio-action)).
These are computed, not modeled, and they are the inputs the next part is bounded by.

Then a **portfolio-construction synthesis** (122B) reconciles each holding's intrinsic standalone lean against those aggregates into its **final action and target weight**, and produces the portfolio-level view: overall risk posture, a cash / deployment stance (what to trim to fund which adds — and, under the unconstrained-cash profile, the **external funding the plan implies**: net new dollars = total buys − total disposition proceeds (trims **and** sell-alls), a line the joint-feasibility solve already computes — a negative value is **net cash raised**, not funding, and reads as such — so the profile's outside-cash assumption is visible on the output rather than silent), the concentration and exposure read, and **positions closed since the last run** (the exited names from the holdings diff — graded nowhere, but acknowledged here rather than silently dropped) — all read against both the report's house view and the investor profile.
This is the only stage that can say "trim the A-grade because it is 11% of the book and correlated with two other holdings, and route the proceeds to the B+ that diversifies it."
The per-holding **action half of the what-changed audit** is written here, where the portfolio context exists, and is app-validated the same way the intrinsic half is — a "became oversized" claim must map to a real aggregate, not a model assertion.
The proposed sizing is additionally held to a deterministic **joint-feasibility check**: per-holding target weights are chosen in one synthesis but must hold *simultaneously*, so the app solves the implied post-action book (every proposed share/dollar adjustment applied at once, holding the **book identity** — final weights plus the cash residual account to 100% of the implied book, with **deployable capital an explicit line** whenever the profile constrains cash; under the unconstrained preset, external funding is the profile's *stated* assumption, not an accounting hole) and validates that each final weight lands inside its stated target range, that no post-action weight breaches the concentration cap, that — when the profile constrains cash — the buys are funded by the trims plus available cash, and that — in a selective run — **every unselected holding's final action obeys the carried-action transition rule** (toward *hold* only, save the aggregate-validated context trim) **and every over-age carry was resolved per family** (add-family demoted to *hold*; exit-family force-included, so none remains carried) ([§Triggering](#triggering)).
A jointly infeasible set fails validation and the synthesis re-runs once with the violation named (the same model-proposes / app-validates seam as the continuity check); a persisting infeasibility fails the run like any hard model failure ([portfolio-workflow.md §Step 7b](portfolio-workflow.md#step-7b-portfolio-construction)).

One stronger alternative was weighed here and **deferred, not adopted**: a deterministic allocation optimizer — the app solving every target weight, the model only explaining the result.
Adopted now, it would put the engine's least-trustworthy numbers (the expected returns) in sole charge of sizing while demoting the model's thesis-level judgment out of it entirely, and the estimation-error evidence — constrained and even naive allocations beating optimized ones out of sample — cuts against optimizing at this data quality.
The evidence says *not now*, not *never*: the named condition for revisiting is the outcome ledger showing the return and risk inputs have matured enough to bear it — the target-calibration read is exactly that gauge ([§Outcome learning](#outcome-learning-calibration)).
Until then the bounded-choice spine stands; what was adopted instead is the joint-feasibility check above and the absolute add-rung floors ([§Starting parameters](#starting-parameters-calibratable)).
Beyond the sum of individual holdings, this is where the job answers "what does the portfolio as a whole look like, how should it be shaped, and how does it sit relative to the current market thesis."

## Starting parameters (calibratable)

The deterministic engine runs on concrete constants, gathered as a single **calibration surface** at the top of `src-tauri/src/portfolio/engine.rs` (with the durable plan-time parameters in `portfolio/mod.rs`).
Every value below is an **MVP starting point chosen to be shadow-tuned against live runs, not frozen** — the same stance the report's thresholds take.
They are recorded here so the spec matches the engine, not to pin them.

**As-built (the narrow single-equity slice).**

- **Composite grade.**
  Sub-score weights **quality 0.30 / valuation 0.25 / momentum 0.20 / risk 0.25** (fixed, summing to 1.0); letter cutoffs over the 0–100 composite — **A ≥ 85, B ≥ 70, C ≥ 55, D ≥ 40**, else F. **Settled (engine update pending): momentum leaves the letter** — the letter re-weights over **quality 0.40 / valuation 0.30 / risk 0.30** (drafted) and momentum re-homes to the **market-setup read** in the conviction / positioning layer, because a price move is the one grade input that isn't a business fundamental and the letter is described to the user as *what the business is*; keeping it in would also force two grades (a display letter and a gating composite), which the move dissolves.
  Until that engine change lands, the as-built composite above stands and the **feasible-set rules read the ex-momentum composite** as the interim guard.
- **Sub-scores.**
  Each a clamped linear map to 0–100 (higher = better): **quality** from net and gross margin, **valuation** from inverted P/E · P/S · P/B (a negative P/E scores low, never "cheap"), **momentum** from trailing price return (computed alongside; outside the letter once the settled re-weight lands — see Composite grade), **risk** (higher = safer) from realized volatility and debt/equity.
  A missing sub-score is **imputed to the neutral midpoint (50)** and the composite divides by the full weight sum — so a missing input pulls the composite toward neutral rather than being dropped (not a renormalization over the present sub-scores); the evidence floor guarantees at least two sub-scores are real, and a grade resting on an imputed sub-score carries a visible **low-confidence marker beside the letter** on the card — the degraded input surfaced at the grade itself, not only inside conviction.
- **Evidence floor.**
  Fewer than **2** computable sub-scores, or no current price, abstains as `insufficient-evidence` rather than grading on too little.
- **Scenario targets.**
  Expected annual drift = `clamp(0.5 × revenue-growth + 0.02, −0.25, +0.35)`; twelve-month base = spot × (1 + drift), one-month = spot × (1 + drift ⁄ 12); bull/bear bands scale off realized volatility with fixed fallbacks (±15% twelve-month, ±5% one-month).
  **Settled (naming fix pending): the two horizons are rolling one-month / twelve-month windows, not calendar ends** — the as-built names (`end_of_month` / `end_of_year`, and the EOM/EOY methodology strings) misname them: outside January, calendar year-end is not twelve months away, the drift applied is always a full year's, and calibration scores these targets against the 1- and 12-month labels ([§Outcome learning](#outcome-learning-calibration)), so the rolling-window name is the honest one; the fields rename with the next engine touch (Trade Opportunities' docs read the same shared window as its twelve-month target window).
  The drift formula is the engine's **named structural upgrade target** — the dead-money bullet's per-share decomposition replaces its *shape*, not just its constants, since this number now drives targets, exit leans, and add eligibility.
- **Verdict vocabularies.**
  Conviction is categorical **High / Medium / Low** (model-authored); the horizon outlook is **short ≈ 1 month / mid ≈ 1 year / long ≈ 3–5 years**, each a {bullish, neutral, bearish} read.
- **Action sizing.**
  Per-rung target-weight multipliers around the position's current weight — **trim 0.4–0.7, hold 0.9–1.1, add 1.2–1.6, add-aggressively 1.6–2.2** — under a single-position **concentration cap of 25%** of the account.
  A buy is additionally bounded by the profile's cash cap **when one is set**, but the fixed preset encodes cash as **unconstrained** (`available_cash: None` — [configuration.md §Investor Profile](configuration.md#investor-profile)), so under it adds are never gated on observed Schwab cash.
  **Settled (engine update pending): the add-family rungs gain absolute target floors** — pure current-weight multipliers perpetuate historical accidents (an A-grade 0.5% position could only ever reach ~0.8%), so *add* targets at least a **~1.5%** book weight and *add-aggressively* at least **~3%** (drafted; the multiplier band or the floor, whichever is higher, with the 25% cap always binding above), while *trim* / *hold* stay relative — trimming is inherently relative to what is held.
- **Retention.**
  Newest **10** Portfolio runs (independent of the 30-report window and of Trade Opportunities); the house view loads the **3** most recent reports.

**Genuinely open (drafted starting values, landing with the slice that needs them).**

- **Dead-money hurdle.**
  `hurdle = rf + risk premium`, where **rf = the run-level `DGS2`** (the suite's short-end anchor, mirroring Trade Opportunities' entry-threshold anchor — a two-year yield read against an annual window, the already-fetched short rate chosen for cross-suite consistency rather than an exact duration match; scenario *discounting* keeps `DGS10`) and the premium is **risk-tier-scaled** — drafted **+3pts (Low) / +5pts (Medium) / +8pts (High)** — in place of a flat 5% (a high-risk holding held to a utility's premium misprices risk with the tier already computed in the spine).
  The read the hurdle tests is the holding's **base-case forward annual *total* return**, whose target decomposition is `expected total return = per-share earnings / FCF growth + forward dividend yield + valuation change` — with **share-count change appearing exactly once** (today's as-built drift uses *company-level* revenue growth, so a separate net-buyback-yield term adds correctly; the moment the growth term moves per-share, the buyback term folds into it and must not also be added) and the payout terms **forward / sustainable**, never a raw trailing print (a one-off special dividend or buyback doesn't set the yield).
  Until that decomposition lands, the read is the as-built expected-drift `annual` **plus shareholder yield** (trailing dividend yield from `dividends` plus net buyback yield from the cash-flow statement, both already fetched) — price drift alone systematically mislabels the low-growth, high-cash-return holding (a 3%-growth staple returning 7% in dividends and buybacks is not dead money at a ~9% hurdle).
  The hurdle verdict is **three-state, not binary**, with one exact deterministic rule over the scenario bands the engine already computes: **clears** when even the *bear-case* total return clears the hurdle; **fails** when even the *bull-case* total return misses it; **indeterminate** otherwise (a point estimate missing the hurdle inside its own scenario dispersion proves nothing).
  Only **fails** makes a holding **dead money** → its standalone lean tilts to **exit, on its own merits** ([§The holding verdict](#the-holding-verdict)); **indeterminate neither tilts the lean nor creates dead money** — the scenario-dispersion tolerance is *exit-side* hysteresis; the add family runs its own admission test in the feasible-set rule below.
  The bars sit deliberately **below** Trade Opportunities' entry thresholds (rf + 8/16/30 — [trade-opportunities.md §Starting parameters](trade-opportunities.md#starting-parameters-calibratable)): a holding may continue below the bar a fresh entry would need (hold-side hysteresis, so the book doesn't churn at the boundary); the *add* side — new dollars, which are an entry decision — is governed by the feasible-set rule below.
- **Action feasible-set bounding.**
  Before the model picks a rung, the engine narrows the offered set from **engine-known inputs only** (grade, dead-money, concentration, cash — conviction is model-authored, so it can't pre-gate): offer *add* / *add-aggressively* only when the holding passes the **new-money admission test** — the **base-case forward total return clears the tier-scaled hurdle as a point test** (new dollars are an entry decision, and the three-state read's dispersion tolerance is *exit-side* hysteresis — it protects held capital, never licenses new capital; an indeterminate read whose base case clears still admits, and the test deliberately stops short of Trade Opportunities' full entry asymmetry — an add to an owned, already-diligenced name is not a cold entry, the same rationale that sets these bars below TO's); dead money (the hurdle's **fails** state) drops the family a fortiori at **any** grade (the former B-grade carve-out is removed, and *hold* stays available, the hysteresis above), as does grade = F or current weight ≥ the 25% cap; offer *add-aggressively* only for an A/B grade with headroom.
  **Every grade test in these rules reads the momentum-free letter** (the settled re-weight above; until that engine change lands, the interim ex-momentum composite serves the same role), so a price crash alone — a momentum swing with the business unchanged — can never bar the contrarian add.
  A **`role_risk_only`** holding's offered set is **{sell all, trim, hold}** — the add family is never offered without return evidence ([§Portfolio action](#portfolio-action)).
  The prompt states the allowed set; the model chooses within it ([§Portfolio action](#portfolio-action)).
  (The cash bound is **profile-gated**; the fixed preset sets cash unconstrained — `available_cash: None` — so it isn't a feasible-set gate here.)
- **Material not-rated threshold.**
  A not-rated position (option / bond / short / cash) at **≥ 5%** of the account contributes its risk / exposure to the whole-book aggregates ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)) — computed where sourceable (market value + signed notional), the unsourceable analytics (duration / credit / standalone delta) riding as typed gaps per that section's contract; below that it is recorded but does not drive the roll-up.
- **Conviction-layer caps** (need the paid-FMP revision feed + `financial-scores`).
  Narrative-vs-reality caps conviction when multiple expansion outruns estimate revisions by more than **~1.5×** with no leading-metric anchor; the forensic flags cap conviction — **Altman Z < 1.8**, **Piotroski ≤ 3**, **net income > 1.3× operating cash flow**, **receivables / inventory growth > 1.5× revenue growth** — with a restatement / auditor-change the hard one.
  These move conviction and the risk read, **never the letter grade**.
  For a **thinly-covered holding** (estimates absent or stale), the narrative-vs-reality read falls back to the **operating-reality-vs-price** form — the company's own reported operating series against the move in price and multiple ([trade-opportunities.md §The two non-negotiables](trade-opportunities.md#the-two-non-negotiables)) — so the cap neither silently vanishes nor fires on noise exactly where coverage is thinnest.
- **Implied-expectations read.**
  The engine inverts its scenario math at the live price into the growth / margin trajectory the price *already assumes* — expressed as a **range under stated assumptions**, never one solved number (the shared primitive Trade Opportunities computes at Step 5c — [trade-opportunities-workflow.md §Step 5c](trade-opportunities-workflow.md#step-5c-deterministic-analysis-archetype-weighted-engine)) — computed per holding and fed to interpretation beside the forward outlook.
  It is the priced-in anchor for the job's most common real decision — trimming or keeping a winner; it informs the conviction and action read and is never itself a gate.
- **Falsifier / trigger persistence.**
  The required consecutive-observation counts (filing-cadence vs high-frequency series) and the per-condition **materiality margin** are **shared suite constants, drafted once with the contract itself** at [trade-opportunities.md §The opportunity](trade-opportunities.md#the-opportunity) — listed here as a pointer, not re-drafted; Portfolio's seams are in [§The position thesis ledger](#the-position-thesis-ledger).
- **Fund valuation coverage.**
  A fund letter requires **≥ 70% of fund weight in P/E-usable sectors** (drafted); below the bar the valuation sub-score records as a gap and the fund abstains under the evidence floor's fund analog ([§Evidence floor](#evidence-floor)).
  The composite's mechanics — covered-weight renormalization, the uncovered share reported beside the read, never averaged in, and the same exchange-blend / coverage convention on the snapshot and its history — are specified once in [§Asset eligibility](#asset-eligibility).
- **Quick-check attention triggers.**
  The four deterministic flag conditions, their priced-only scoping, and the clear-on-full-pass semantics are specified once at [§The quick check](#the-quick-check-engine-only) — listed here as a pointer, not re-drafted: they carry no constants of their own beyond the shared persistence contract above.
- **Technology-event pre-flag.**
  The input delta flags a possible third-party repricing event — an **equity-holding read** — when the holding's **sector-relative move since the prior run** exceeds **2 × its interval-scaled realized volatility** (drafted; √-of-time scaling, the suite's cadence-honest convention), read against the same Stooq sector benchmark series the outcome labels use.
  The pre-flag only adds the conditional research topic — it asserts nothing about the cause ([§The per-holding pipeline](#the-per-holding-pipeline) Step 3).
- **Evidence events.**
  The **evidence-event list** — referenced by the research-cache invalidation, the selective-run force-include, and the quick check's informational badge — is defined once: a **new earnings actual** (a fresh `earnings` surprise row since the last full pass — re-pulled per stock by the quick check, so the leg is checkable between full runs), a **material filing** (a new 8-K / 10-Q / 10-K on the EDGAR sweep — CIK-gated: a symbol whose CIK doesn't resolve reads the filing family `unknown`, which force-includes through the degraded-sweep rule rather than silently skipping the leg — [data-sources.md §SEC EDGAR](data-sources.md#sec-edgar), [§The quick check](#the-quick-check-engine-only)), or a **large revision move** — drafted: the current-FY consensus EPS moving **more than ~5%**, the percentage read only where the consensus is **positive and at least ~$0.10/share** (the denominator floor; a negative or below-floor consensus tests an **absolute move of ~$0.10/share or more** instead — both constants drafted — so the ratio can't explode into a noisy trigger), or estimate-revision breadth flipping sign, since the last full pass — read from the quick check's lightweight **revision preflight** (a per-stock `analyst-estimates` snapshot compared against the stored prior, the same series the engine already snapshots for revision velocity), or a **qualifying structured-news seed** — a fresh symbol-scoped `news/stock` item arriving while the holding carries a **standing technology-class falsifier** (a known threat axis plus new news on it: the deterministic conjunction, deliberately **high-recall and not topic-matched** — any fresh news on a tech-flagged holding qualifies, since deterministic headline-to-threat-axis matching would be fabricated precision, and the cost of an occasional spurious force-include is bounded by how few holdings carry a standing technology falsifier — the quick check pulls `news/stock` only for those holdings; an announcement with **no price impact and no standing falsifier** is deliberately outside the list, because a headline's materiality can't be classified without a model and a fabricated gate would be worse than the honest boundary — the next full pass's fresh seeds catch it, bounded by the ~4-week max-age) — so every leg of the list is deterministically detectable by the pass that promises it.
  Those legs are **equity legs** — the estimate, earnings, and news-seed series are company-scoped.
  A **fund** holding carries its own deterministic legs, detected from the quick check's **fund refresh**: a **material `etf/info` change** — the reported expense ratio moving, or the asset-class / mandate fields that route its strategy classification changing (a structural-flag reclassification, a vehicle newly reading leveraged / inverse, counts here) — for every fund, and, per **equity** fund on either verdict branch (the guard crossing flips the branch in **both** directions — a demotion below it or a promotion above it), an **exposure shift**: the US share crossing the ≥ 70% US-exposure guard, or a top sector weight moving **~10 points or more** (drafted).
  Fund legs invalidate research reuse and force-include in a selective run exactly as the equity legs do — alongside, not instead of, the model-authored fund-ledger conditions, which reach a selective run through the ordinary flag-based force-include.
- **Research reuse window.**
  A holding **reuses its cached research** (the loop and distillation skip; interpretation reads the cached distilled findings) when that research is younger than **~4 weeks** — aligned with the suite's research-freshness / forward-assumption credit window ([trade-opportunities.md §Starting parameters](trade-opportunities.md#starting-parameters-calibratable)) — *and* nothing moved: position unchanged, no attention-trigger condition, no event pre-flag, **and no new evidence event since the research date** — any leg of the evidence-event list above (all visible to 6a/6b even on a reuse pass) **invalidates the cache**, since price and trigger movement alone would miss exactly the events research exists to read.
  Past the window the full agenda is mandatory, so the window doubles as the max-age service level; every reuse decision is logged in the audit record.
- **Resume window.**
  An interrupted run offers **resume** only while its pinned holdings pull is younger than **~48 hours** (drafted); past the bound the checkpoints are stale against a book that may have moved, so Run analysis starts a new run ([§Failure posture](#failure-posture)).
- **Outcome learning (constants).**
  The full contract — episode creation and vintage stamping, branch typing, `action_source`, label mechanics, and the derived reads — lives in [§Outcome learning](#outcome-learning-calibration); its calibratable constants: the **matured archive's cap** — drafted **5,000** rows, evicted oldest-first (the active store carries no cap: an episode still accruing labels is never evicted; it is **age-bounded rather than count-capped** — an episode matures out into the archive once its 12-month labels record, normally right at 12 months (one held **pending** by a price-coverage gap stays active until the label can honestly record — never matured from stale bars, per the label contract in [§Outcome learning](#outcome-learning-calibration)), so the active set holds roughly a year of decision changes on the book); the **proposal eligibility bar** — **≥ 30 unique holdings with matured windows**, clustered by holding, never raw episode counts (each proposal's uncertainty interval clusters by **holding × decision date**), a slice-specific proposal needing its own cohort clearing the same bar; and the bear–bull band's declared **nominal coverage of 80%** (drafted), scored with an interval score on the price-only label.
- **Grade normalization & inputs (full Portfolio).**
  When the full slice lands, the sub-score inputs inherit the shared engine's Trade-Opportunities-side machinery rather than staying raw clamped maps: **quality** lifts the value-creation reads — **gross profitability (GP / assets)**, the **ROIC-vs-cost-of-capital spread**, and **FCF conversion** — rather than resting on margins alone, and every factor normalizes against **sector-adjusted absolute bands + the name's own history** (the Step-5c basis) with **sector-appropriate metric selection** — P/B-family valuation for financials, FFO-style economics for REITs, debt/equity dropped where it is structurally meaningless (banks) — because universal margin / P-E / D-E bands misgrade banks, REITs, cyclicals, and negative-equity names; the weights and letter cutoffs recalibrate then.
- **Forward-conviction raise (bidirectional conviction).**
  Research may **raise** conviction by at most **one band** (Low → Medium → High), and only when it validates an *engine-unscored*, countable / dated / third-party leading indicator (a `research`-class signal, carried as the typed **`validated_leading_indicator`** distillation emits — [portfolio-workflow.md §Step 6d](portfolio-workflow.md#step-6d-distillation)) confirming a thesis-ledger **key driver** *ahead of* the trailing financials; the raise must cite that field and is logged and continuity-validated ([portfolio-workflow.md §Step 6g](portfolio-workflow.md#step-6g-continuity-check-and-checkpoint)).
  Price action and narrative can never raise it, and a metric the engine already scored cannot (no double-count).
  The **one-band cap** and the **leading-indicator gate** are the calibratable values.
- **Whole-book reads.**
  Correlation / overlap starts from a **90-day Stooq return correlation**, co-sizing names with **|ρ| > 0.7**; a dedicated **factor-exposure model is deferred** (the sector / country exposure tilt already specified at construction carries the load until one is justified).
  That deferred factor model is also the **named home for portfolio stress reads** — rate / credit / commodity / dollar shock scenarios and marginal risk contribution, the tail co-movement a 90-day correlation can't see — deferred with it, not separately scheduled.

## Continuity and isolation

The job retains its most recent N runs.
The prior run feeds the next (continuity above), and run results are embedded into the **Portfolio Analysis memory partition only** — its learnings are never read by the report or by Trade Opportunities, and it never reads theirs (see [local-models.md §Run history and continuity](local-models.md#run-history-and-continuity)).
This keeps holding-grading calibration cumulative within the job and uncontaminated by other jobs' context.

## Outcome learning (calibration)

The calibration stance below ([§Starting parameters](#starting-parameters-calibratable)) — every constant an MVP starting point, shadow-tuned against live runs — needs something to tune *against*: a recorded, engine-computed read of how the job's own past calls actually resolved.
Trade Opportunities carries this machinery already ([trade-opportunities.md §Outcome learning](trade-opportunities.md#outcome-learning-calibration)); Portfolio's analog is structurally simpler — the universe is the holdings list, so there is no shadow ledger of turned-away names — but adds one confound of its own: **the user acts on the book between runs**, so what happened to a position reflects both the job's call and the user's response.

**A decision episode is created when a holding's recommendation state changes on either half of its verdict** — the **intrinsic read** (the verdict branch, the standalone lean, or the standing thesis) or the **construction read** (the final action or target-weight range) — anchored on the run date.
The standing thesis counts as *changed* only when the run's what-changed audit records an attributed intrinsic move or a labeled self-correction for the holding — a wording-only rewrite over an empty audit line is a re-affirmation, so model rephrasing can't mint episodes; **input movement alone never opens an episode** (rates, prices, and sub-scores — and the scenario targets recomputed off them — move every run: a re-affirmed decision over moved inputs is still one decision, the movement recorded as an observation on the active episode), and a run that merely **re-affirms** *extends* the holding's active episode (the re-affirmation noted, no new anchor), so weekly runs can't manufacture overlapping pseudo-observations of a single decision.
The extension rule applies only **while an episode is active**: an episode is a bounded twelve-month measurement instrument, not a live-state mirror, so once a holding's episode matures, its unchanged recommendation legally carries **no active episode** — a post-maturity re-affirmation records nothing (there is no open forecast left to observe against), and the next genuine state change opens a fresh episode under the creation rule above — the Portfolio form of Trade Opportunities' one-open-episode-per-live-lifecycle bound ([trade-opportunities.md §Starting parameters](trade-opportunities.md#starting-parameters-calibratable)).
The two halves keep **separate clocks inside the one episode stream**: besides its anchor date, each episode stamps its **intrinsic vintage** — the run that actually *authored* the intrinsic fields it records.
On a full pass the two coincide; in a **selective run** a construction-only change over a carried, vintage-stamped verdict ([§Triggering](#triggering)) opens an episode whose intrinsic fields keep their **older vintage**, and the **intrinsic calibration reads — the lean-keyed cohorts and target calibration — consume only episodes whose intrinsic vintage equals their anchor** (a fresh forecast, scored from the date it was made; the carried fields' forecast is already scored by the episode that authored them), while a construction-only episode still feeds the final-action diagnostic strata and the alignment tag.
Symmetrically, an intrinsic-state change the construction stage doesn't act on — a moved lean, or a `priced ↔ role_risk_only` branch flip, with the final action unchanged — still opens a fresh episode: the intrinsic forecast changed, so the old one stops accruing and the new one anchors on its own date, which is what keeps every calibration label anchored to the date its forecast was actually made.
Episodes are **typed by verdict branch, with an explicit schema per branch**: a **`priced`** episode records the final action and target weight, the standalone lean **and the divergence-from-lean rationale** (whether the final action matched the lean and, where it diverged, the action half's attribution category — moved portfolio context vs moved intrinsic verdict — so the final-action strata are per-episode facts, not a join against an audit record that ages out), the grade, the decomposed conviction, the scenario bands and base-case target, the dead-money state, the position delta, the degraded-input flags, and a compact **calibration-feature snapshot** — the decision-time engine inputs a future parameter proposal needs for counterfactual re-testing: the sub-scores and composite, the risk tier, the scenario total-return distribution, the hurdle inputs (the DGS2 print, tier premium, and three-state read), the cap signals in force (narrative-vs-reality state, forensic flags), and the **parameter version** the decision ran under — because the run's own audit record can age out of the 10-run retention before a 12-month label matures, so the episode carries what re-testing needs rather than pointing at a run that may no longer exist; a **`role_risk_only`** episode ([§Intrinsic verdict](#intrinsic-verdict)) records only the final action and target weight, the position delta, the degraded-input flags, and the parameter version — its snapshot's reduced analog (no lean, grade, conviction, band, target, or dead-money field exists on its verdict to record), is **excluded from target calibration and every grade-linked read**, and is counted **in its own class, never pooled** with priced episodes in the action-cohort spreads.
Each episode also carries an **`action_source`** field — `model-chosen` or `rule-demoted` — because a stale-action demotion ([§Triggering](#triggering)) is not an analytical decision: rule-demoted actions stay out of the pooled cohort reads (their own class), so the hold cohort measures only holds a model actually chose.
The next run's deterministic holdings diff then tags the holding's active episode — where one exists — with **`observed_net_alignment`** — **aligned / contrary / partial / unknown / reversed** — from the net quantity move relative to the recommended direction (`hold` reads aligned when unchanged; a net **long↔short reversal always tags `reversed`**, never `aligned` — crossing zero lies outside any long-side recommendation's direction, so the tag is its own class, excluded from the aligned / contrary cohort slices, while the episode's labels keep maturing on the name's path exactly as after an exit — [§Asset eligibility](#asset-eligibility)).
The name is deliberate: a net diff cannot see round trips, transfers, partial execution, or a user acting twice between runs, so the tag claims only what the diff can observe — never that advice was "followed."
Episodes persist in their own store, **independent of the 10-run retention** (a count-based retention can age out before a 12-month label matures); a matured episode is **frozen into a compact matured archive** once its 12-month labels record — not discarded at maturation, because matured episode-level rows are what let a future parameter proposal be tested against realized history and let uncertainty intervals be recomputed; an embedding or aggregate summary cannot recreate them ([storage.md §Local Analysis Suite Storage](storage.md#local-analysis-suite-storage)).

**Confirmed falsifier events.**
A priced episode also records each confirmed falsifier that attaches while the episode carries the condition as `{condition_id, confirmed_at, confirmation_observation_id}`.
The event lives with the episode rather than only in a run audit that can age out before the twelve-month read matures; a falsifier that confirms **after the holding's episode has matured** (the recommendation unchanged, so no active episode to attach to — [the episode boundary above](#outcome-learning-calibration)) records on the holding's persistent **thesis ledger** entry instead — retained as context for the next episode, though it feeds no lead-time read, that read being bounded to the matured episode's own window; `role_risk_only` episodes retain their ordinary condition-evaluation state but carry no price-derived lead-time record.

**The labels are engine-computed, never model-judged**, under the measurement contract Trade Opportunities pinned: forward return at **1 / 3 / 6 / 12 months** — absolute and vs sector / market — plus maximum drawdown, reconstructed statelessly from Stooq daily bars.
**Before a window is marked matured**, the outcome pass refreshes that episode's symbol-scoped Stooq series through the window end via the shared price-bar cache — independently of the current holdings work-list, so an exited name refreshes too.
A failed refresh may use the cache only when its latest bar reaches the window end; otherwise the label stays **pending** with a price-coverage gap rather than maturing from stale bars (the sector / market benchmark legs follow the same coverage rule).
The entry reference is the **next session's close** after the run (a consistent evaluation anchor, deliberately not called an executable price); returns are split-adjusted; and terminal events are **typed and scored per type** — an acquisition realizes at the final trading price, a bankruptcy scores to zero, an ambiguous delisting is `terminal-unscorable`, excluded from spreads but counted and logged.
One deliberate Portfolio divergence from that contract: labels are computed on **two declared bases, with total return primary** — the decision rules being calibrated (the dead-money hurdle, the action leans) are themselves total-return rules, and scoring them price-only would mismeasure exactly the dividend-heavy names the total-return fix exists for — while the **price-only read is retained as the cross-entry common basis and diagnostic** (derivable for every entry; any comparison with a missing total-return leg quotes price-only, a labeled mix still being a mix).
The two bases have **defined mechanics**: the price leg reconstructs from the split-adjusted Stooq bars alone (the shared cache is price-only — [storage.md §Local Analysis Suite Storage](storage.md#local-analysis-suite-storage)); the total-return leg adds the window's cash dividends from a **label-time `dividends` re-pull** on the episode's symbol — symbol-scoped, so an exited name is as computable as a held one — summed **without reinvestment** over the anchor price (a declared simplification, immaterial at these windows; a failed re-pull records that episode's total-return leg unavailable, taking the labeled price-only fallback above).
And the **relative legs — vs sector / market — run on the price-only common basis on both sides**: the Stooq benchmark series carry no distributions, so a total-return holding leg against a price-only benchmark would bias every spread by roughly the dividend yield; **total-return-primary is the absolute label**, the one the hurdle and action-lean rules calibrate against.
A user's exit does **not** stop measurement: the label tracks the *name's* path from the decision date whether or not the position is still held — that is exactly what makes a sell call scorable — with the alignment tag carrying the confound into the reads.

Four derived reads; where a read scores returns, it is computed per matured window on **unique-holding counts**, sliceable by the alignment tag, and on an **explicitly declared return basis** (the basis follows the rule being scored, never a blanket default):

- **Action-cohort spreads** — did the add-cohort outperform the hold-cohort, and the hold-cohort the trim / sell-cohort?
  Cohorts form on **two declared layers, matched to what each calibrates**: for **intrinsic** parameters (sub-score weights, the hurdle, the caps) the cohort key is the **standalone lean** — the pure intrinsic read, before portfolio context, read from **vintage-fresh episodes only** ([the episode boundary above](#outcome-learning-calibration)) — because a final action deliberately reflects concentration, overlap, tax, and overlay context, and a concentration-driven trim of a winner that keeps winning would otherwise teach the intrinsic engine a lesson about expected returns it never asserted; **final-action cohorts are read separately**, stratified by each episode's **divergence-from-lean rationale** — and read **diagnostically only**: raw return ordering cannot score a risk override (a concentration trim is judged by the risk it removed, not the return it forwent), so a portfolio-level risk / drawdown evaluation of construction is deferred with the factor model ([§Starting parameters](#starting-parameters-calibratable)) (`role_risk_only` episodes, having no lean, stay in their own class as before).
  Computed on the **absolute total-return labels as the primary ordering read** — the ladder orders total-return expectations, so the read's basis must match the rule being scored (and the benchmark's missing dividend leg is largely common to the compared cohorts at matched windows, so it nets out of a spread anyway) — with the **sector / market-relative price-only spreads retained as the regime-controlled diagnostic** (absolute cross-cohort comparisons are confounded by *when* the decisions were made, which is exactly what the relative leg controls for).
  The Portfolio form of Trade Opportunities' picked-vs-rejected spread: the cleanest read of whether the action ladder is ordering names correctly.
- **Target calibration** — band coverage against its **declared nominal level** (drafted **80%** — [§Starting parameters](#starting-parameters-calibratable)), scored **on the price-only label** (the bands and base case are *price* targets, so the price path is the only honest comparison — a total-return read would systematically overshoot the band for dividend payers) with an **interval score** that evaluates calibration and sharpness together (raw hit-rate rewards an arbitrarily wide band; a proper scoring rule can't be gamed by width) — each band scored at its **matching window**: the one-month band against the 1-month label, the twelve-month band against the 12-month label (the 3- and 6-month labels serve the cohort reads, not band scoring), each read from a **vintage-fresh episode's recorded band** — a band an extending run merely recomputes rides as an observation, not a new forecast — so every scored band anchors on the date it was authored — plus the signed error of the base case: a systematic-bias read on the scenario engine (and, indirectly, on the drift formula the dead-money read consumes).
- **Falsifier lead-time** — **priced episodes only**, using fields already frozen at the episode's intrinsic vintage: the recorded **twelve-month bear-case target is the material-drawdown line**, and that target's rolling **twelve-month window is the thesis-realization window** for this read.
  A confirmed falsifier event attaches to the episode that carried the condition as `{condition_id, confirmed_at, confirmation_observation_id}`; the engine records the **signed trading-day distance** from `confirmed_at` to the first within-window close below the bear-case line — positive when the falsifier confirmed before the breach, zero on the same session, negative when the line had already broken before confirmation — or explicit **`no-material-drawdown`** when no such close occurs by maturity.
  This makes “early enough to matter” deterministic rather than interpretive; `role_risk_only` episodes are excluded because they carry no price band.
- **Self-correction accumulation** — the per-holding self-correction counts the what-changed audit already labels ([§What changed](#what-changed)), made cumulative: repeated self-corrections on one holding are the calibration signal that clause names, now actually counted.

Two disciplines keep the reads honest.
**Attribution stays at the cohort level** — a single trim that preceded a rally was a risk decision, not an error, so no per-decision P&L verdict is ever assigned; only cohort spreads and calibration rates feed anything.
And the loop is **propose-only, on the same stance as Trade Opportunities**: realized outcomes *propose* sub-score-weight / hurdle / cap changes for the user's review, never auto-apply — intrinsic-parameter proposals reading the **lean-keyed cohorts**, never the final-action strata ([the spreads read above](#outcome-learning-calibration)) — and no proposal is made below the eligibility bar (drafted in [§Starting parameters](#starting-parameters-calibratable): **≥ 30 unique holdings with matured windows**, clustered by holding, never raw episode counts — each proposal reporting its effect size and an uncertainty interval clustered by **holding *and* decision date**, since names judged in the same market regime share shocks).
The bar is a *minimum*: a **slice-specific** proposal — per-sector, per-action-rung, per-parameter — needs its own cohort clearing the same bar, not a share of the pooled one.
On a single book of a few dozen names all of this accrues slowly — deliberately: a small sample proposes nothing rather than proposing noise.

This scorecard is also the **named graduation path for the held-out signals**: the positioning layer and the Schwab options-activity signal enter the sub-scores only when these reads show they add discriminating power — "held out until shadow-mode calibration proves it adds value" now points here, not at an unspecified future judgment.
Matured labels and the derived reads persist with the run's audit record and embed into the **Portfolio Analysis memory partition** as durable learnings ([§Continuity and isolation](#continuity-and-isolation)); whether the scorecard gets its own UI surface is deferred alongside Trade Opportunities' matching display decision.

## Storage and display

Each run persists its per-holding verdicts, the per-holding **thesis ledger** (carried forward to seed the next run's continuity check), and the portfolio roll-up, together with the **run audit record** that makes the run traceable and reviewable — the holdings snapshot it ran against, sources and retrieval timestamps, distilled findings, computed metrics and derived reads, the conviction decomposition, the input delta and per-value what-changed attribution, the price-target methodology, model and prompt/schema versions, degraded-input flags, each holding's **research-reuse decision** ([§The per-holding pipeline](#the-per-holding-pipeline) Step 3), and this run's **outcome-learning records** ([§Outcome learning](#outcome-learning-calibration)) — with the field set, including which fields each verdict branch records, specified once in [storage.md §Local Analysis Suite Storage](storage.md#local-analysis-suite-storage).
(Source URLs, timestamps, and the distilled findings make a verdict's basis inspectable; full page snapshots are not stored, so a run is traceable rather than bit-for-bit reproducible.)
Retention keeps the last N runs (parallel to report retention — [storage.md](storage.md)).
The **Portfolio page** renders each holding's verdict — the **intrinsic** read (the **backward grade** and sub-scores paired *side by side* with the **forward outlook** — horizon reads and scenario targets — so a grade/outlook divergence is legible at a glance, plus conviction and the bear/base/bull monitor) beside the **portfolio action** (action, target weight, sizing rationale) — a **`role_risk_only`** verdict rendering its own explicit card branch instead: the role read, exposure tilt, observable risk, expense drag, structural flag, and evidence gaps beside the action, a designed state with **no empty priced-field placeholders** ([§Intrinsic verdict](#intrinsic-verdict)) — **anchored by the thesis ledger's current standing thesis** as the card's *why we hold this view* (rendered straight from the ledger the continuity check carries forward and rewrites — [§The position thesis ledger](#the-position-thesis-ledger) — **not** a separately authored summary, so there is one continuity-validated source of truth and no drift between what the job carries forward and what the user reads; the held-position analog of each Trade Opportunity leading with its directional thesis), with financials and the what-changed line — plus the amber, actionable **attention flag** when the quick check has flagged the holding, retained until the next successful full pass over it clears and acknowledges the trigger ([§The quick check](#the-quick-check-engine-only)), a per-card **selection control** (with select-all / deselect-all) driving selective re-analysis ([§Triggering](#triggering)), and — after a selective run — a per-card **analysis-vintage stamp**, so mixed-vintage verdicts read as labeled vintages rather than a blend — alongside the portfolio roll-up, and shows not-rated and insufficient-evidence positions with their reason (see [interface.md](interface.md)).

**The current-holdings view.**
The page also carries the latest standalone **Pull holdings** snapshot ([§Triggering](#triggering)) as a compact, position-level view — symbol, quantity, the per-unit price as the source reported it (absent where it carries none; for option contracts it is the per-contract value the source's market value implies, not a per-share quote), market value, cost basis, the percentage gain on cost (the card sort bar's same per-position-totals ratio, rendered as a directional value), and a pulled-at stamp.
Every column head sorts the rows **in place** — symbol alphabetically (opens ascending); quantity, price, market value, cost basis, and percentage gain numerically (open descending), a position missing a key's value (no reported price or cost basis) sorting last under that key in either direction, the card sort bar's convention — defaulting to the account's as-pulled order, with the last-used column remembered in webview `localStorage` like the card sort.
Being a real data grid, its heads carry **`aria-sort`** through keyboard-operable head buttons — the pattern the card-stack sort bar deliberately reserves for tables.
**Before any run exists it *is* the page body** (holdings viewable with no analysis — and no local-model configuration — at all, framed as *pulled, not yet analyzed* with Run analysis as the call to action); **once runs exist it appears only when the pull is fresher than the last run**, as a stamped section *above* the verdict cards — **never replacing them** — so fresh quantities and the run's verdicts read as two labeled vintages (*holdings as of…* / *analysis from…*) rather than a blend.
Position churn between the two vintages is tagged by **symbol presence only**: a pulled symbol absent from the run's snapshot reads *new · not in last analysis*, and a verdict card whose symbol has left the pull reads *no longer held* (the card itself stays, untouched — run output is never mutated or hidden by a pull).
Quantity-move classification is deliberately **not** re-derived here — that stays the run-time engine diff's job, with its short-position and reversal handling ([§Holdings change tracking](#holdings-change-tracking)) — and the *no-longer-held* tag is provisional until the next run formalizes the exit in the roll-up's closed-positions record.

**Holdings sorting.**
Above the holding-card stack a compact **sort bar** reorders the cards **in place** — no separate table view, so every card keeps its full intrinsic-verdict / portfolio-action / thesis-anchor content.
It offers four deterministic, **position-level** keys, all engine-computed (never the model's grade or conviction — sorting by position size or P/L is orthogonal to the analysis read) from two **per-position account-currency totals** that already account for option contract multipliers, bond conventions, and cash balances — the position's **market value** (Schwab-reported) and its **cost basis** (the app-derived signed total — [schwab-integration.md §What is pulled](schwab-integration.md#what-is-pulled)) — rather than a naïve `quote × shares` that would misprice options, fixed income, and cash: **overall value** (market value), **dollar gain** (market value − cost basis, the same unrealized P/L the action-sizing spine derives — [§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)), **percentage gain** (that gain over cost basis), and **total cash invested** (cost basis).
A position Schwab reports **no cost basis** for (cash, typically) has an **undefined gain and cash-invested**, so it sorts **last** under the dollar-gain, percentage-gain, and cash-invested keys (the holdings analog of a debut Trade Opportunity's null since-flagged read), while still sorting normally by overall value.
A **zero-or-negative netted basis** (a net-short or zero-net book, where the short side's proceeds offset the basis — [schwab-integration.md §What is pulled](schwab-integration.md#what-is-pulled)) keeps its **dollar gain** — signed totals hold market value − cost basis equal to the book's aggregate unrealized P/L even at zero net quantity — but has **no honest percentage gain or cash-invested**, so both render **undefined** and sort last under those two keys; a zero, negative, or missing denominator never produces a divide-by-zero rank.
The default is **overall value, descending** (biggest positions first), tie-broken alphabetically by ticker; the last-used key is remembered as **ephemeral frontend state** in webview `localStorage` — the same store the Light/Dark appearance preference uses (pure presentation, no backend consumer), not the SQLite config store.
**Not-rated** positions (options / fixed income / cash) carry no grade but a real market value (and cost basis where Schwab reports one — cash typically has none), so they sort within the value / gain / cost keys alongside graded holdings, kept visually tagged as not-rated; **exited** positions (closed since last run) live only in the roll-up ([§Holdings change tracking](#holdings-change-tracking)), never in the sortable stack.
The control is **display-only** — it reorders already-computed fields and touches no engine, schema, or workflow — and follows the project's frontend-craft requirements (keyboard-operable **toggle buttons** exposing the active key via `aria-pressed` plus a screen-reader-announced current-sort state (the active button's **accessible name carries the key *and* its sort direction** — e.g. "Sort by value, descending" — since the `▾`/`▴` glyph is decorative / `aria-hidden` and can't be relied on to announce direction) — *not* `aria-sort`, which is reserved for grid/table column headers such as Trade Opportunities' List view, not this card-stack toolbar; a stable sort; and a reduced-motion-respecting reflow).
The sort bar is a small **design-system extension**: it ports the `.ana-grid` sortable-header vocabulary (the neutral `▾` / `▴` glyph) to a non-table toolbar of tracked-caps triggers, since the holdings surface is cards rather than a data grid — no off-system dropdown or pill control ([interface.md](interface.md)).

## Failure posture

The execution gate requires the local model daemon and roster ([local-models.md](local-models.md)), **a connected Schwab account** ([schwab-integration.md](schwab-integration.md)) — holdings and the options signal both come from Schwab, so a missing or lapsed connection blocks the job (manual-import holdings do not satisfy the gate) — and the shared **FMP / FRED credential presence** ([portfolio-workflow.md §Step 1](portfolio-workflow.md#step-1-job-start-and-gate)).
Because holdings are independent, the job **checkpoints per holding** — each holding's completed stages persist, so a cancellation or a single model failure resumes the unfinished holdings rather than restarting the (potentially hours-long) run, and recent research is cached within the ~4-week freshness window (the same cache the graduated research-depth reuse reads — [§The per-holding pipeline](#the-per-holding-pipeline) Step 3); a selective re-analysis checkpoints identically.
**Resume is its own entry path, distinct from a new run** — the checkpoint contract is only coherent against one holdings snapshot.
A **resume** — offered from the failed / cancelled run's tracker state, only while its checkpoints exist and its pinned pull is younger than the drafted resume window ([§Starting parameters](#starting-parameters-calibratable)) — **reopens the interrupted run's ID and pins everything from it**: the original holdings pull and option-chain snapshot, the Step-5 shared context, and the model / prompt / schema versions, with **no new pull** — so completed and resumed holdings alike are computed against one coherent snapshot, and the finished run is honestly stamped with that pull's as-of time.
A **new run** — what the Run analysis control always starts — takes a fresh pull under a new run ID and **discards any interrupted run's checkpoints** (its partial verdicts never became a persisted run — the run-is-never-a-report analog); the per-holding **research cache survives** independently under its ~4-week window, so a discard re-spends interpretation calls, not the web research.
Within a run, web research is fail-soft (thinner evidence, lower conviction), while a hard model or persistence failure fails the run, recorded like any other job ([scheduling.md](scheduling.md)).
The complete split, in one place — the failure matrix [scheduling.md](scheduling.md) delegates here: **hard, fails the run** — the Step-2 Schwab positions pull ([schwab-integration.md §Failure posture](schwab-integration.md#failure-posture)), a per-holding model-call failure (the run fails but resumes from its checkpoints — above), a Step-7b joint infeasibility that persists after its one re-run, and the run-persistence write; **fail-soft, degrades** — web research and every enriching feed (FINRA, CBOE, the commodity context, CFTC positioning, option chains), each to its typed gap or degraded-input flag; **per-holding, never run-level** — missing floor-bearing inputs exit that holding as `insufficient-evidence` at Step 6b ([§Evidence floor](#evidence-floor)); and the engine-only **quick check** fail-softs to the last cached data with its as-of date — each failed signal family typed **`unknown`**, never a silent clear, force-including the holding in a selective run ([§The quick check](#the-quick-check-engine-only)).
The **quick check** is engine-only — no model call (it runs even configured-but-down, like Trade Opportunities' Quick Audit), no web research, no Schwab call — so its failure surface is the local store and its refresh retrievals ([§The quick check](#the-quick-check-engine-only)), fail-soft to the last cached data with its as-of date — surfaced through the typed per-family sweep states, so a failed retrieval is visible and selective-run-safe rather than indistinguishable from a clean pass.
