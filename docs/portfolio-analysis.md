# Portfolio Analysis

Portfolio Analysis is a **local, on-demand job** that grades every holding in the user's portfolio and recommends an action for each, grounded in the holding's fundamentals, fresh web research, and the current Market Signal house view. It runs entirely on local models (see [local-models.md](local-models.md)) and sources holdings — and the per-stock options signal — from a **connected Charles Schwab account, which is required to run the job** (manual import supplements the holdings but does not replace the connection; see [schwab-integration.md](schwab-integration.md)).

It is deliberately **prescriptive** — it issues buy/trim/hold/sell-style actions and explicit price targets — which is the opposite of the Market Signal Report's no-buy/sell stance ([report-structure.md](report-structure.md)). The report forms the *house view*; this job applies that view to the specific positions the user holds.

## Triggering

The job is manual and runs in two user-controlled steps: **pull holdings**, then **run analysis**. Only one run happens at a time across the whole app — a single global run slot serializes the report and both local jobs (see [local-models.md §Failure posture](local-models.md#failure-posture)). While it runs it streams into the run tracker with **per-holding progress**, the same observability seam the report uses ([run-tracking.md](run-tracking.md)).

## Asset eligibility

A Schwab portfolio holds more than equities — ETFs, mutual funds, options, fixed income, and cash. The equity-centric pipeline (FMP fundamentals, company financials) applies cleanly only to individual stocks, and in reduced form to funds. Each position is classified by asset type before analysis:

- **Stocks** — the full pipeline and verdict.
- **ETFs / funds** — a reduced verdict (no single-company financials; graded on strategy / holdings, valuation, and the house view), sourced from FMP's **ETF & mutual-fund** endpoints: `etf/info` (expense ratio, AUM, NAV, asset class, mandate), `etf/holdings` (constituents + weights for look-through and concentration), and `etf/sector-weightings` / `etf/country-weightings` (exposure breakdown); mutual funds add the `funds/disclosure*` holdings filings. Portfolio-level **look-through concentration** is computed locally from each held fund's `etf/holdings` (not a per-fund fetch); FMP's `etf/asset-exposure` — a reverse lookup of which ETFs hold a single-name — is an optional per-direct-equity cross-check, not part of fund grading.
- **Options, fixed income, cash, and unsupported types** — marked **not rated**, with a short reason, and excluded from grading. Cash and buying power still feed the investor profile and the portfolio roll-up.

A not-rated position never receives a fabricated grade; the eligibility decision is explicit and shown in the UI.

For a **fund** holding the engine runs a **reduced computation** in place of the equity pipeline: **expense drag** (the expense ratio as an annual return headwind), **concentration** (top-N holding weight and a Herfindahl index from `etf/holdings`), **look-through exposure** (sector / factor / country, aggregated from the holdings and `etf/sector-weightings` / `etf/country-weightings`), and a **fund-level valuation** (the holdings-weighted multiple read against the house view). These feed a **reduced sub-score set** — **valuation** plus a **risk** score driven by concentration and exposure, with **no company-quality score** (there is no single issuer to grade) — while the action, conviction, and continuity machinery stay identical to the equity path; the **evidence floor uses the fund analog** above, not the company-statements floor (see [§Evidence floor](#evidence-floor)).

## Evidence floor

Eligibility decides whether a position *can* be graded; the evidence floor decides whether it *should* be — and the engine's inputs split by criticality. **Floor-bearing inputs** — the company's **financial statements** (FMP / SEC) and a **current price** — are the irreducible basis for a grade: an eligible holding missing these, or carrying stale prices, too few quality sources, or conflicting identity data, returns **`insufficient-evidence`** — an explicit abstention — rather than a low-conviction grade. A **fund** holding has no company statements, so it floors on the **fund analog**: a current **quote / NAV**, `etf/info`, sufficiently fresh **holdings / disclosure**, the expense ratio, and enough sector / country / top-holding coverage to compute concentration and exposure — missing those abstains the fund the same way. Every **other** input (`financial-scores`, peers, the revision / surprise reads, positioning, dividends, float, beta, and all analyst data) is **enriching**: its absence **lowers conviction** and is recorded as a degraded-input flag, but never on its own forces abstention. "Firm" means committing when the evidence supports it, not manufacturing a verdict when it doesn't — the per-holding analog of the report's Step-3 coverage floor.

## Holdings change tracking

Each run persists the **holdings snapshot it ran against** (see [§Storage and display](#storage-and-display)), so the next run diffs the current Schwab pull against the prior run's snapshot deterministically — in the application layer, before any model stage. Every current position is classified by quantity and cost basis as **new**, **increased**, **decreased**, or **unchanged**, and that delta rides into the holding's dossier so the verdict reasons over what the user actually did — added to, trimmed, or left a position — rather than re-grading it in a vacuum. A symbol present last run but **absent now** is an **exited** position: it earns no per-holding verdict (there is nothing left to grade), but it is surfaced in the portfolio roll-up as closed-since-last-run, so a sold-out name is acknowledged rather than silently vanishing from the run. The diff is the app's, not the model's — the same compute-don't-guess boundary the rest of the pipeline holds.

## The per-holding pipeline

This section covers the per-holding pipeline's **design**; the full job's **control flow** — every step, its Type (Computed / API retrieval / Local-model call), and each local-model-call contract — is the canonical spec in [portfolio-workflow.md](portfolio-workflow.md) (the per-holding loop is its Step 6).

Each holding is processed through a chain of distilled, schema-validated hand-offs (see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)):

1. **Dossier assembly (deterministic, application layer).** The app builds the holding's evidence packet: the position itself (quantity, cost basis, market value, P/L); for an **equity** holding, the FMP per-symbol surface on the paid key — fundamentals (financial statements / ratios / key metrics, `financial-scores` for Altman Z + Piotroski, owner earnings, enterprise values, DCF, **product / geographic revenue segments**), the **analyst / revision signal** (forward estimates, price-target consensus + summary trend, the `grades-historical` rating distribution + consensus, upgrades / downgrades, earnings surprises), **symbol-keyed positioning** (insider, 13F institutional, Senate / House congressional, **FINRA short interest** — level / trend / days-to-cover), **stock peers** for relative valuation, **dividends** (yield, frequency, schedule), the **next earnings date** (`earnings?symbol=`) as a near-term catalyst, **earnings-call transcripts** (`earning-call-transcript`) for management commentary on guidance / margins / demand, **M&A involvement** (`mergers-acquisitions-search` — acquirer or target), **share float / liquidity**, and the company **profile** (sector, industry, beta) — joined with **SEC EDGAR** (10-K/Q/8-K + XBRL company facts) as the authoritative cross-check; for a **fund** holding, the reduced FMP **ETF / mutual-fund** surface instead (see [§Asset eligibility](#asset-eligibility)); price history (Stooq) and a live quote (FMP `quote`); the full endpoint surface — every path, by source, with its per-holding / per-fund / run-level cardinality — is tabulated in [data-sources.md §Portfolio Analysis — endpoint surface](data-sources.md#portfolio-analysis--endpoint-surface); the **investor profile** (risk tolerance, time horizon, tax sensitivity, and available cash / buying power — see [configuration.md](configuration.md)); the Market Signal house view, loaded deterministically as the latest report's **Thesis, Investment Strategy, and Forward Outlook** sections plus recent report summaries (`thesis_stance`, `forward_outlook_themes`, `key_risks`); vector-retrieved continuity from **this job's own prior runs** for this holding; the **prior run's verdict for this holding**; and the **position delta** against the prior run's holdings snapshot (new / increased / decreased / unchanged since last run — see [§Holdings change tracking](#holdings-change-tracking)). Three market inputs are fetched **once per run and shared across every holding**, not re-requested per symbol — the **risk-free rate** (a FRED `DGS10` / `DGS2` read, the series the report already gathers) that anchors the valuation engine's discounting, **cyclical commodity prices** (FRED) for commodity-linked holdings, and **CFTC Commitments-of-Traders positioning** on the bellwether contracts (the same keyless pull the report makes), which a commodity / macro **fund** holding (e.g. a gold or oil ETF) maps onto for an underlying-positioning read where its underlying is among those contracts. Report context is loaded directly, not vector-searched (see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)).
2. **Deterministic financial analysis (application layer).** A Rust financial-analysis engine (shared with Trade Opportunities — see [trade-opportunities.md](trade-opportunities.md)) computes the quantitative picture from the FMP/SEC data for an **equity** holding (a fund holding takes the reduced computation in [§Asset eligibility](#asset-eligibility)), in three layers. **(a) The grade core** — margins, growth, leverage, free cash flow, and valuation multiples (the last including **EV multiples** and an **owner-earnings yield**, cross-checked against FMP's pre-computed **DCF**); **revenue-segment concentration** (product / geographic, into the quality and thematic read); **momentum and volatility** from price history, plus **beta** and share **float / liquidity**; and the **forensic flags** the shared engine derives (margin compression while revenue accelerates, net-income-vs-operating-cash-flow divergence, receivable / inventory build outpacing sales, restatement / auditor-change history). These roll up into the **quality / valuation / momentum / risk sub-scores** — `financial-scores` (Altman Z + Piotroski) and the forensic flags folded into the **risk / quality** read, `stock-peers` multiples into **valuation** — and into scenario-based **end-of-month and end-of-year price targets** discounted off the run-level risk-free rate, each with its methodology and assumptions recorded. (The forward targets are a **provisional scenario menu**: a material forward fact from research can refine them at Step 5; the backward-looking sub-scores computed here are final.) **(b) A conviction layer**, computed but kept *out of the sub-scores*: estimate-revision velocity (`analyst-estimates`), analyst-rating drift (`grades-historical`), and the **earnings-surprise** magnitude, combined into a **narrative-vs-reality ratio** (revision pace vs multiple change — *justified-expensive* when estimates outrun the multiple, *hype* when the multiple outruns flat / declining estimates) that sharpens **conviction** and the valuation read without moving the letter grade. **(c) Positioning context** — insider net, institutional (13F) and congressional flow, **short-interest level / days-to-cover (FINRA)**, and the **options-activity signal** from Schwab chains (put/call by volume and open interest, IV/skew — see [schwab-integration.md](schwab-integration.md)) — likewise computed but **held out of the grade sub-scores until shadow-mode calibration shows it adds value**, grounding the narrative/sentiment read only. **Dividend yield and payout** feed the **total-return** read for income-oriented profiles. These computed values — not the model — are the source of every number in the verdict (see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)). The engine also diffs this run's computed picture (metrics, sub-scores, positioning, price) against the **prior run's stored values** to produce a deterministic **input delta** — the evidence the continuity audit (Step 6) attributes each verdict move to, so "the market changed" can only be claimed against an input that actually did. The dossier's **absolute street opinions** — the consensus price-target *level*, the current buy / hold / sell consensus, and FMP's ratings snapshot — are deliberately **not** engine inputs (distinct from the computed revision *pace* in layer b): the engine's grade and targets come only from the fundamentals, and those opinions ride into interpretation (Step 5) as *external evidence the model weighs against the engine's own read* (a consensus far from the engine target is a flag to explain, not a number to adopt).
3. **Bounded web research** — the 122B reasoner (thinking mode) plus the web tool, running the bounded research loop ([web-research.md](web-research.md)), fail-soft. The agenda is worked **one topic at a time** — each topic its own focused **research pass** over a clean context (itself a bounded multi-turn tool loop), with up to two app-governed follow-up passes (depth ≤2, so ≤3 *passes*, not LLM turns, per topic) when a pass surfaces a sub-thread worth pursuing — and all topics share a **per-holding research budget** (web-fetch count and wall-clock) that binds first, bounding the raw turns and fetches, and is spent in topic-priority order, so an exhausted budget drops the lowest-priority topic fail-soft rather than overrunning the run. Grounded by the deterministic SEC/FMP financials and the holding's latest **earnings-call transcript** (so research fills the gaps the numbers and management's own commentary don't), the agenda covers: competitive / business position, recent results and estimate revisions, catalysts and risks; **market narrative and sentiment** — the prevailing story driving the stock and how much of its price reflects emotion about *what might come* versus what the company does today, weighed against the fundamentals (narrative vs reality) rather than chased — a **sourced qualitative read** (every claim cited via the research loop), distinct from the quantitative sentiment proxies (options activity, momentum, volatility); and **forward opportunity and thematic fit** — the future opportunity set (TAM, optionality, pipeline) and how the company maps onto dominant market themes (e.g. the AI buildout) that can carry the stock regardless of near-term results.
4. **Distillation** — the reasoner in **non-thinking mode** consolidates each topic's **full findings response** into the compact object the interpretation stage reads — a **single** consolidation over the complete per-topic outputs, not a re-distillation of in-loop summaries (research is never planned over already-distilled notes; see [web-research.md](web-research.md)) — so interpretation reasons over a clean synthesis rather than the research transcript. This runs on the same resident 122B by default — one model across research, distill, and interpret avoids per-holding model swaps; a dedicated fast 35B model is a benchmark-gated option, used only if distillation wall-clock proves a measured bottleneck *and* it co-resides cleanly (see [local-models.md §The model roster and per-task routing](local-models.md#the-model-roster-and-per-task-routing)).
5. **Interpretation and grading** — first, a **deterministic target refinement** runs if distillation produced a typed **`research_forward_assumption`** — a material forward fact the structured feeds lacked (a guidance figure, a signed-contract value, a commodity / ASP turn) carried as a strict object (fact type, numeric value, units, period / as-of date, source URL, confidence, the target assumption it affects, and conflict handling): the **engine — not the model —** recomputes the affected scenario target with it as an explicit, logged assumption, so the number stays engine-computed (compute-don't-guess) while the forward view reflects what research learned. A malformed, unsourced, or non-numeric claim is rejected (it cannot move a target), and a fact conflicting with a structured feed is resolved by the assumption's declared conflict rule. The backward-looking sub-scores are untouched, and absent a valid assumption the Step-2 targets stand. Then the 122B reasoner in thinking mode interprets the (refined) analysis and the distilled research into the verdict: it sets the grade and action and explains the reasoning, **selecting and justifying the base-case target**, but reads every number (sub-scores, metrics, targets) from the deterministic engine rather than inventing it.
6. **Continuity check.** The verdict is **audited** against the prior run's, using the engine's deterministic **input delta** (Step 2). Interpretation (Step 5) attributes every verdict move to an input that actually moved — an **external** change (market data, company information, or research / narrative) — or, where the inputs sat still, to a **labeled self-correction** of the model's own prior read. The app then **validates** the attribution — every "external" label must resolve to a concrete input-delta entry, a source-backed research finding, or the logged forward assumption, or it is **downgraded to self-correction** (or fails schema validation) — so the distinction can't be gamed. Output stays firm and does not swing run-to-run absent hard supporting data, so a self-correction is the earned, flagged exception, not a quiet re-grade (see [thesis-continuity.md](thesis-continuity.md)).

## The holding verdict

Each holding's output is a structured, schema-validated record:

- **Composite grade** (A–F) over the engine's deterministically-computed sub-scores — **quality**, **valuation**, **momentum**, and **risk** — so the letter rolls up from real metrics, not a model's gestalt.
- **Action**, on a fixed ladder: **sell all → trim → hold → add → add aggressively**, made concrete with a **target portfolio-weight range and an estimated share/dollar adjustment** (bounded by concentration limits, available cash, and tax sensitivity from the investor profile). No orders are ever placed.
- **Conviction** — the verdict's confidence, lowered when evidence is thin or degraded and shaped by the **narrative-vs-reality** read (a price driven by multiple-expansion the estimates don't support caps conviction even on a strong company); below the evidence floor it abstains entirely (see [§Evidence floor](#evidence-floor)).
- **Horizon outlook** — separate short-, mid-, and long-term reads.
- **Price targets** — end-of-month and end-of-year, computed by the financial-analysis engine as scenario outputs with their **methodology and assumptions exposed** (including any research-sourced forward assumption folded in post-research — see [§The per-holding pipeline](#the-per-holding-pipeline) Step 5); the model selects and justifies the base case rather than inventing the number.
- **Financial analysis** — a concise read of the company's financial health.
- **What changed** — a structured **audit of the prior verdict**, not a bare diff. Every moved value (grade, any sub-score, action, conviction, target, horizon) is shown old → new with its **cause attributed**: an **external change** — **market data** (price / momentum / positioning), **company information** (earnings, guidance, a new filing, a fundamentals shift), or **research / narrative** (a catalyst or narrative change) — tied to the specific input that moved; or a **self-correction**, where the inputs did *not* materially change and the model is revising its own prior read (fixing a prior error or refining its understanding of the same facts). A self-correction is the **flagged exception** to firm-output continuity — it swings the verdict with no new facts — so it is labeled as such rather than dressed as an external move, and repeated self-corrections on one holding are a calibration signal. The line also carries the **position delta** (new / increased / decreased / unchanged since last run; an unseen holding reads as `new`).

The fixed action ladder and grade vocabulary are load-bearing: like the report's fixed regime labels ([storage.md](storage.md)), they keep verdicts comparable across runs and prevent the model from retreating into hedged, non-comparable language. The grade is intrinsic to the holding; the **action** additionally reflects the investor profile (horizon, risk tolerance, tax sensitivity) and available cash — for example, *add aggressively* is offered only when buying power supports it.

## Portfolio roll-up

After the per-holding pass, a synthesis stage (122B) produces a **portfolio-level view**: concentration and sector/factor exposure (with **fund holdings looked through** — computed locally from each held fund's `etf/holdings` — so a name held both directly and inside an ETF aggregates rather than hiding), overall risk posture, a cash/deployment stance, and **positions closed since the last run** (the exited names from the holdings diff — graded nowhere, but acknowledged here rather than silently dropped) — read against both the report's house view and the investor profile. This is where the job answers "what does the portfolio as a whole look like, and how does it sit relative to the current market thesis," beyond the sum of individual holdings.

## Continuity and isolation

The job retains its most recent N runs. The prior run feeds the next (continuity above), and run results are embedded into the **Portfolio Analysis memory partition only** — its learnings are never read by the report or by Trade Opportunities, and it never reads theirs (see [local-models.md §Run history and continuity](local-models.md#run-history-and-continuity)). This keeps holding-grading calibration cumulative within the job and uncontaminated by other jobs' context.

## Storage and display

Each run persists its per-holding verdicts and the portfolio roll-up, together with an **audit record** that makes the run traceable and reviewable: the holdings snapshot it ran against, the report(s) and sources it used (with retrieval timestamps), the distilled findings and the **computed financial metrics, the derived reads (revision / narrative-vs-reality, forensic flags, positioning), the input delta and the per-value what-changed attribution, and the price-target methodology — including its discount-rate assumption and any research-sourced forward assumption applied in the post-distillation refinement (with its source) —** behind each verdict, the model ids and quantizations, the prompt/schema version, and any degraded-input flags. (Source URLs, timestamps, and the distilled findings make a verdict's basis inspectable; full page snapshots are not stored, so a run is traceable rather than bit-for-bit reproducible.) Retention keeps the last N runs (parallel to report retention — [storage.md](storage.md)). The **Portfolio page** renders each holding's verdict — grade and sub-scores, conviction, action, outlook, targets, financials, and the what-changed line — alongside the portfolio roll-up, and shows not-rated positions with their reason (see [interface.md](interface.md)).

## Failure posture

The execution gate requires the local model daemon and roster ([local-models.md](local-models.md)) **and a connected Schwab account** ([schwab-integration.md](schwab-integration.md)) — holdings and the options signal both come from Schwab, so a missing or lapsed connection blocks the job (manual-import holdings do not satisfy the gate). Because holdings are independent, the job **checkpoints per holding** — each holding's completed stages persist, so a cancellation or a single model failure resumes the unfinished holdings rather than restarting the (potentially hours-long) run, and recent research is cached within a freshness window. Within a run, web research is fail-soft (thinner evidence, lower conviction), while a hard model or persistence failure fails the run, recorded like any other job ([scheduling.md](scheduling.md)).
