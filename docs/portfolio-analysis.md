# Portfolio Analysis

Portfolio Analysis is a **local, on-demand job** that grades every holding in the user's portfolio and recommends an action for each, grounded in the holding's fundamentals, fresh web research, and the current Market Signal house view. It runs entirely on local models (see [local-models.md](local-models.md)) and sources holdings — and the per-stock options signal — from a **connected Charles Schwab account, which is required to run the job** (manual import supplements the holdings but does not replace the connection; see [schwab-integration.md](schwab-integration.md)).

It is deliberately **prescriptive** — it issues buy/trim/hold/sell-style actions and explicit price targets — which is the opposite of the Market Signal Report's no-buy/sell stance ([report-structure.md](report-structure.md)). The report forms the *house view*; this job applies that view to the specific positions the user holds.

## Triggering

The job is manual and runs in two user-controlled steps: **pull holdings**, then **run analysis**. Only one run happens at a time across the whole app — a single global run slot serializes the report and both local jobs (see [local-models.md §Failure posture](local-models.md#failure-posture)). While it runs it streams into the run tracker with **per-holding progress**, the same observability seam the report uses ([run-tracking.md](run-tracking.md)).

## Asset eligibility

A Schwab portfolio holds more than equities — ETFs, mutual funds, options, fixed income, and cash. The equity-centric pipeline (FMP fundamentals, company financials) applies cleanly only to individual stocks, and in reduced form to funds. Each position is classified by asset type before analysis:

- **Stocks** — the full pipeline and verdict.
- **ETFs / funds** — a reduced verdict (no single-company financials; graded on strategy / **exposure**, valuation, and the house view). On the current FMP plan ([data-sources.md §FMP — current paid-plan tier audit](data-sources.md#fmp--current-paid-plan-tier-audit)) **`etf/holdings`, `etf/asset-exposure`, and the `funds/disclosure*` filings are off-plan**, so constituent-level look-through is lost. ETFs are sourced from the **on-plan** `etf/info` (expense ratio, AUM, NAV, asset class, mandate) + `etf/sector-weightings` / `etf/country-weightings` — an **exposure-tilt** read (sector / country, not single names). Single-name **concentration** and direct-vs-fund **overlap** degrade to that exposure level unless reconstructed from **SEC N-PORT** (keyless but heavy, ~60-day lag — an optional enhancement, never floor-bearing). A future **issuer-holdings adapter** — major ETF sponsors publish current constituent files fresher than N-PORT's lag, at the cost of per-issuer, messier parsing — is the practical way to buy back single-name look-through without a higher FMP tier; it is **planned, not built**, and like N-PORT would only enrich the fund path, never gate it. **Mutual funds degrade hardest**: the `etf/*` weighting endpoints don't cover them, so a mutual fund grades on **quote / NAV + profile + the house view** only (or SEC N-PORT). `etf/asset-exposure` (reverse "which ETFs hold this name" lookup) is off-plan and dropped — it was only an optional per-direct-equity cross-check.
- **Options, fixed income, cash, and unsupported types** — marked **not rated**, with a short reason, and excluded from grading. Cash and buying power still feed the investor profile and the portfolio roll-up — and a **material** not-rated position (a sizable fixed-income sleeve, a large options position) still contributes its **risk and exposure** to the whole-book aggregates ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)), so the portfolio's risk posture reflects the real book rather than only the rated sleeve; it simply carries no per-position grade.

A not-rated position never receives a fabricated grade; the eligibility decision is explicit and shown in the UI.

For a **fund** holding the engine runs a **reduced computation** in place of the equity pipeline: **expense drag** (the expense ratio as an annual return headwind), **exposure tilt** (sector / country from `etf/sector-weightings` / `etf/country-weightings`), and a **fund-level valuation** (read against the house view). **Constituent concentration** (top-N weight, Herfindahl) is **off-plan by default** — `etf/holdings` is blocked — so it is computed only when **SEC N-PORT** is available and otherwise omitted (recorded as a degraded-input flag, never a fabricated number). These feed a **reduced sub-score set** — **valuation** plus a **risk** score driven by exposure (and concentration where N-PORT supplies it), with **no company-quality score** (there is no single issuer to grade) — while the action, conviction, and continuity machinery stay identical to the equity path; the **evidence floor uses the fund analog** above, not the company-statements floor (see [§Evidence floor](#evidence-floor)).

## Evidence floor

Eligibility decides whether a position *can* be graded; the evidence floor decides whether it *should* be — and the engine's inputs split by criticality. **Floor-bearing inputs** — the company's **financial statements** (FMP / SEC) and a **current price** — are the irreducible basis for a grade: an eligible holding missing these, or carrying stale prices, too few quality sources, or conflicting identity data, returns **`insufficient-evidence`** — an explicit abstention — rather than a low-conviction grade. A **fund** holding has no company statements, so it floors on the **fund analog**: a current **quote / NAV**, `etf/info`, the expense ratio, and (for ETFs) enough **sector / country weighting** coverage to read exposure — missing those abstains the fund the same way. Constituent **concentration** is **enriching, not floor-bearing** (it needs the off-plan `etf/holdings`, recoverable only via SEC N-PORT), so its absence lowers conviction rather than forcing abstention. Every **other** input (`financial-scores`, peers, the revision / surprise reads, positioning, dividends, float, beta, and all analyst data) is **enriching**: its absence **lowers conviction** and is recorded as a degraded-input flag, but never on its own forces abstention. "Firm" means committing when the evidence supports it, not manufacturing a verdict when it doesn't — the per-holding analog of the report's Step-3 coverage floor.

## Holdings change tracking

Each run persists the **holdings snapshot it ran against** (see [§Storage and display](#storage-and-display)), so the next run diffs the current Schwab pull against the prior run's snapshot deterministically — in the application layer, before any model stage. Every current position is classified by quantity and cost basis as **new**, **increased**, **decreased**, or **unchanged**, and that delta rides into the holding's dossier so the verdict reasons over what the user actually did — added to, trimmed, or left a position — rather than re-grading it in a vacuum. A symbol present last run but **absent now** is an **exited** position: it earns no per-holding verdict (there is nothing left to grade), but it is surfaced in the portfolio roll-up as closed-since-last-run, so a sold-out name is acknowledged rather than silently vanishing from the run. The diff is the app's, not the model's — the same compute-don't-guess boundary the rest of the pipeline holds.

## The position thesis ledger

Where holdings-change tracking remembers *what the user did* with a position, the **thesis ledger** remembers *why the job holds a view on it* — a compact, persisted, per-holding record carried forward run to run, the Portfolio analog of Trade Opportunities' opportunity graph ([trade-opportunities.md §Discovery memory](trade-opportunities.md#discovery-memory-the-opportunity-graph)) but **downstream and holding-specific**: it tracks an owned position rather than discovering a new one. Without it, each run re-derives its conviction from scratch and continuity is only a value-by-value diff; with it, a run tests the *standing thesis* against fresh evidence and says precisely which part broke.

Each gradable holding carries a ledger with:

- **Thesis** — the original thesis (when the position first entered an analysis) and the current thesis, so drift is legible.
- **Key drivers** — the few variables the thesis actually depends on (e.g. data-center revenue growth, gross-margin trajectory, net retention), each tied where possible to an engine-tracked metric so the next run can read whether it moved.
- **Bear / base / bull monitor** — three compact scenarios, each with the conditions that define it, the engine's scenario price target for it, and a rough probability lean. The monitor names, per scenario, **what must improve** to migrate toward the bull case, **what must not break** to stay in the base case, and the conditions that would force a downgrade or license an upgrade — so the next run's continuity check has explicit goalposts rather than a gestalt.
- **Key falsifiers** — monitorable conditions that would invalidate the thesis (the Portfolio analog of an opportunity's key falsifiers), each typed as **quantitative** (engine-checkable against the input delta — e.g. "operating margin falls below 30%") or **qualitative** (research/model-checkable — e.g. "a credible second source ships at scale"). A **third-party technology event** that threatens a holding (a competitor's or supplier's product / standard announcement) is a first-class **qualitative falsifier class** here: when one fires, it is recorded with the same sized **`technology_read`** the Trade Opportunities lens uses (substitute / complement / mix-shift, the *actually-exposed* revenue / profit pool, deployment timeline, switching costs), so "is this a real impairment or a panic-priced threat?" is tracked as a monitorable condition rather than a vague worry ([§The per-holding pipeline](#the-per-holding-pipeline) Step 3; [trade-opportunities.md §The event-impact / value-chain repricing lens](trade-opportunities.md#the-event-impact--value-chain-repricing-lens)).
- **Action triggers** — explicit **add / trim / sell triggers** and a **target-weight range**: the pre-committed conditions under which the standing view would change the position (e.g. "trim above 8% portfolio weight," "add on a pullback to the base-case downside with the thesis intact"). The triggers are *intrinsic* conditions on the holding; the portfolio-construction stage ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)) still owns whether a fired trigger actually sizes, given the rest of the book.

A **fund** holding carries a ledger of the same shape but a different flavor of content, matching its reduced engine path ([§Asset eligibility](#asset-eligibility)): its **drivers** are exposure tilt, expense drag, mandate / tracking fidelity, role in the portfolio, and house-view fit — not company operating metrics — and its falsifiers and triggers are written against those (e.g. "mandate drift," "expense ratio rises above the category," "the exposure this fund supplies is now better held directly").

The ledger is **read at dossier assembly and re-evaluated and rewritten each run**, across the pipeline's deterministic and model stages ([§The per-holding pipeline](#the-per-holding-pipeline)): the **engine** (as part of its input delta) tests which **quantitative** falsifiers and triggers crossed this run; the **interpretation stage** reads those crossings, judges the **qualitative** conditions from research, and rewrites the thesis, re-weights the scenarios, and re-sets the triggers; and the **continuity check** validates the rewrite. A tripped falsifier or fired trigger is a first-class continuity signal held to the same app-validation as the what-changed audit — it must resolve to a real input-delta crossing or a source-backed finding, so the ledger cannot be quietly rewritten to fit a new verdict. The ledger is persisted with the run and survives within the job's retention like the rest of a holding's continuity (see [§Storage and display](#storage-and-display)).

## The per-holding pipeline

This section covers the per-holding pipeline's **design**; the full job's **control flow** — every step, its Type (Computed / API retrieval / Local-model call), and each local-model-call contract — is the canonical spec in [portfolio-workflow.md](portfolio-workflow.md) (the per-holding loop is its Step 6).

Each holding is processed through a chain of distilled, schema-validated hand-offs (see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)):

1. **Dossier assembly (deterministic, application layer).** The app builds the holding's evidence packet: the position itself (quantity, cost basis, market value, P/L); for an **equity** holding, the FMP per-symbol surface on the paid key — fundamentals (financial statements / ratios / key metrics, `financial-scores` for Altman Z + Piotroski, owner earnings, enterprise values, DCF, **product / geographic revenue segments**), the **analyst / revision signal** (forward estimates, price-target consensus + summary trend, the `grades-historical` rating distribution + consensus, upgrades / downgrades, earnings surprises), **symbol-keyed positioning** (insider, Senate / House congressional, **FINRA short interest** — level / trend / days-to-cover; FMP **13F institutional is off-plan** → SEC EDGAR or omitted, [data-sources.md §FMP — current paid-plan tier audit](data-sources.md#fmp--current-paid-plan-tier-audit)), **stock peers** for relative valuation, **dividends** (yield, frequency, schedule), the **next earnings date** (`earnings?symbol=`) as a near-term catalyst, **management commentary** on guidance / margins / demand (**earnings-call transcripts are off-plan**, so this rides the Step-3 web-research loop, not an FMP pull), **M&A involvement** (the per-symbol `mergers-acquisitions-search` is off-plan → the market-wide `mergers-acquisitions-latest` + 8-K), **share float / liquidity**, and the company **profile** (sector, industry, beta) — joined with **SEC EDGAR** (10-K/Q/8-K + XBRL company facts) as the authoritative cross-check; for a **fund** holding, the reduced FMP **ETF / mutual-fund** surface instead (see [§Asset eligibility](#asset-eligibility)); price history (Stooq) and a live quote (FMP `quote`); the full endpoint surface — every path, by source, with its per-holding / per-fund / run-level cardinality — is tabulated in [data-sources.md §Portfolio Analysis — endpoint surface](data-sources.md#portfolio-analysis--endpoint-surface); the **investor profile** (risk tolerance, time horizon, tax sensitivity, and available cash / buying power — see [configuration.md](configuration.md)); the Market Signal house view, loaded deterministically as the latest report's **Thesis, Investment Strategy, and Forward Outlook** sections plus recent report summaries (`thesis_stance`, `forward_outlook_themes`, `key_risks`); vector-retrieved continuity from **this job's own prior runs** for this holding; the **prior run's verdict and thesis ledger for this holding** ([§The position thesis ledger](#the-position-thesis-ledger)); and the **position delta** against the prior run's holdings snapshot (new / increased / decreased / unchanged since last run — see [§Holdings change tracking](#holdings-change-tracking)). Three market inputs are fetched **once per run and shared across every holding**, not re-requested per symbol — the **risk-free rate** (a FRED `DGS10` / `DGS2` read, the series the report already gathers) that anchors the valuation engine's discounting, **cyclical commodity prices** (FRED) for commodity-linked holdings, and **CFTC Commitments-of-Traders positioning** on the bellwether contracts (the same keyless pull the report makes), which a commodity / macro **fund** holding (e.g. a gold or oil ETF) maps onto for an underlying-positioning read where its underlying is among those contracts. Report context is loaded directly, not vector-searched (see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)).
2. **Deterministic financial analysis (application layer).** A Rust financial-analysis engine (shared with Trade Opportunities — see [trade-opportunities.md](trade-opportunities.md)) computes the quantitative picture from the FMP/SEC data for an **equity** holding (a fund holding takes the reduced computation in [§Asset eligibility](#asset-eligibility)), in three layers. **(a) The grade core** — margins, growth, leverage, free cash flow, and valuation multiples (the last including **EV multiples** and an **owner-earnings yield**, cross-checked against FMP's pre-computed **DCF**); **revenue-segment concentration** (product / geographic, into the quality and thematic read); **momentum and volatility** from price history, plus **beta** and share **float / liquidity**; and the **forensic flags** the shared engine derives (margin compression while revenue accelerates, net-income-vs-operating-cash-flow divergence, receivable / inventory build outpacing sales, restatement / auditor-change history). These roll up into the **quality / valuation / momentum / risk sub-scores** — `financial-scores` (Altman Z + Piotroski) and the forensic flags folded into the **risk / quality** read, `stock-peers` multiples into **valuation** — and into scenario-based **end-of-month and end-of-year price targets** discounted off the run-level risk-free rate, each with its methodology and assumptions recorded. (The forward targets are a **provisional scenario menu**: a material forward fact from research can refine them at Step 5; the backward-looking sub-scores computed here are final.) **(b) A conviction layer**, computed but kept *out of the sub-scores*: estimate-revision velocity (`analyst-estimates`), analyst-rating drift (`grades-historical`), and the **earnings-surprise** magnitude, combined into a **narrative-vs-reality ratio** (revision pace vs multiple change — *justified-expensive* when estimates outrun the multiple, *hype* when the multiple outruns flat / declining estimates) that sharpens **conviction** and the valuation read without moving the letter grade. **(c) Positioning context** — insider net and congressional flow (FMP **13F institutional is off-plan** → SEC EDGAR or omitted), **short-interest level / days-to-cover (FINRA)**, and the **options-activity signal** from Schwab chains (put/call by volume and open interest, IV/skew — see [schwab-integration.md](schwab-integration.md)) — likewise computed but **held out of the grade sub-scores until shadow-mode calibration shows it adds value**, grounding the narrative/sentiment read only. **Dividend yield and payout** feed the **total-return** read for income-oriented profiles. These computed values — not the model — are the source of every number in the verdict (see [local-models.md §Context-memory discipline](local-models.md#context-memory-discipline)). The engine also diffs this run's computed picture (metrics, sub-scores, positioning, price) against the **prior run's stored values** to produce a deterministic **input delta** — the evidence the continuity audit (Step 6) attributes each verdict move to, so "the market changed" can only be claimed against an input that actually did. The same input delta drives the deterministic evaluation of the prior thesis ledger's **quantitative** falsifiers and triggers — which conditions crossed this run — for the interpretation stage to read ([§The position thesis ledger](#the-position-thesis-ledger)). The dossier's **absolute street opinions** — the consensus price-target *level*, the current buy / hold / sell consensus, and FMP's ratings snapshot — are deliberately **not** engine inputs (distinct from the computed revision *pace* in layer b): the engine's grade and targets come only from the fundamentals, and those opinions ride into interpretation (Step 5) as *external evidence the model weighs against the engine's own read* (a consensus far from the engine target is a flag to explain, not a number to adopt).
3. **Bounded web research** — the 122B reasoner (thinking mode) plus the web tool, running the bounded research loop ([web-research.md](web-research.md)), fail-soft. The agenda is worked **one topic at a time** — each topic its own focused **research pass** over a clean context (itself a bounded multi-turn tool loop), with up to two app-governed follow-up passes (depth ≤2, so ≤3 *passes*, not LLM turns, per topic) when a pass surfaces a sub-thread worth pursuing — and all topics share a **per-holding research budget** (web-fetch count and wall-clock) that binds first, bounding the raw turns and fetches, and is spent in topic-priority order, so an exhausted budget drops the lowest-priority topic fail-soft rather than overrunning the run. Grounded by the deterministic SEC/FMP financials and the holding's latest **earnings-call commentary** (surfaced through this web-research loop, since FMP transcripts are off-plan — so research fills the gaps the numbers and management's own words don't), the agenda covers: competitive / business position, recent results and estimate revisions, catalysts and risks; **market narrative and sentiment** — the prevailing story driving the stock and how much of its price reflects emotion about *what might come* versus what the company does today, weighed against the fundamentals (narrative vs reality) rather than chased — a **sourced qualitative read** (every claim cited via the research loop), distinct from the quantitative sentiment proxies (options activity, momentum, volatility); and **forward opportunity and thematic fit** — the future opportunity set (TAM, optionality, pipeline) and how the company maps onto dominant market themes (e.g. the AI buildout) that can carry the stock regardless of near-term results. **Conditionally — when the holding has moved on a third-party technology event** (a competitor's or supplier's product / standard announcement that materially repriced the name or its group, the same materiality gate Trade Opportunities applies — [trade-opportunities.md §The event-impact / value-chain repricing lens](trade-opportunities.md#the-event-impact--value-chain-repricing-lens)) — the agenda adds a **technology-event impact assessment** that reads the *actual technology* and **sizes the holding's real exposure** (substitute / complement / mix-shift, the exposed revenue / profit pool, deployment timeline, switching costs), producing the typed **`technology_read`** that feeds the conviction read and the thesis-ledger falsifier ([§The position thesis ledger](#the-position-thesis-ledger)). This is what tells a **panic drop apart from a genuine impairment** — and, symmetrically, a holding that *rose* on an announcement whose benefit is overstated — rather than letting both flow through as "the price moved." It is one input among many, bounded by the same per-holding research budget (an extra topic when triggered, dormant otherwise).
4. **Distillation** — the reasoner in **non-thinking mode** consolidates each topic's **full findings response** into the compact object the interpretation stage reads — a consolidation over the **complete** per-topic outputs, never a re-distillation of in-loop summaries (research is never planned over already-distilled notes; see [web-research.md](web-research.md)) — so interpretation reasons over a clean synthesis rather than the research transcript. It is **a single pass by default, or hierarchical** (a tier-1 distillation per topic-tree → a reduce) when a holding's research is large, by the same deterministic, orchestrator-chosen, budget-driven rule the suite shares ([web-research.md §The research loop and context management](web-research.md#the-research-loop-and-context-management)); Portfolio has no cross-lens contradiction check to relocate, so the reduce here is purely consolidation. This runs on the same resident 122B by default — one model across research, distill, and interpret avoids per-holding model swaps; a dedicated fast 35B model is a benchmark-gated option, used only if distillation wall-clock proves a measured bottleneck *and* it co-resides cleanly (see [local-models.md §The model roster and per-task routing](local-models.md#the-model-roster-and-per-task-routing)).
5. **Interpretation and grading** — first, a **deterministic target refinement** runs if distillation produced a typed **`research_forward_assumption`** — a material forward fact the structured feeds lacked (a guidance figure, a signed-contract value, a commodity / ASP turn) carried as a strict object (fact type, numeric value, units, period / as-of date, source URL, confidence, the target assumption it affects, and conflict handling): the **engine — not the model —** recomputes the affected scenario target with it as an explicit, logged assumption, so the number stays engine-computed (compute-don't-guess) while the forward view reflects what research learned. A malformed, unsourced, or non-numeric claim is rejected (it cannot move a target), and a fact conflicting with a structured feed is resolved by the assumption's declared conflict rule. The backward-looking sub-scores are untouched, and absent a valid assumption the Step-2 targets stand. Then the 122B reasoner in thinking mode interprets the (refined) analysis and the distilled research into the **intrinsic verdict**: it sets the grade, conviction, and horizon, **selects and justifies the base-case target**, and commits to a **standalone action lean** (what it would do if the holding stood alone, before portfolio context) — but reads every number (sub-scores, metrics, targets) from the deterministic engine rather than inventing it, and the *final* portfolio action is deferred to the construction stage that has the whole book in view ([§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)). This stage also **rewrites the thesis ledger** — revising the thesis, re-weighting the bear/base/bull monitor, and re-setting the falsifiers and triggers in light of what this run learned.
6. **Continuity check.** The verdict is **audited** against the prior run's, using the engine's deterministic **input delta** (Step 2). Interpretation (Step 5) attributes every verdict move to an input that actually moved — an **external** change (market data, company information, or research / narrative) — or, where the inputs sat still, to a **labeled self-correction** of the model's own prior read. The app then **validates** the attribution — every "external" label must resolve to a concrete input-delta entry, a source-backed research finding, or the logged forward assumption, or it is **downgraded to self-correction** (or fails schema validation) — so the distinction can't be gamed. Output stays firm and does not swing run-to-run absent hard supporting data, so a self-correction is the earned, flagged exception, not a quiet re-grade (see [thesis-continuity.md](thesis-continuity.md)). The check also **validates the thesis-ledger rewrite**: every falsifier the run marks tripped and every action trigger it marks fired must map to the engine's deterministic crossing (for a quantitative condition) or a source-backed finding (for a qualitative one), or it is rejected — the same attribution rule, so the ledger can't be quietly rewritten to fit the new verdict (see [§The position thesis ledger](#the-position-thesis-ledger)).

## The holding verdict

Each holding's output is a structured, schema-validated record that deliberately keeps two things apart — the **intrinsic verdict** (what the holding is worth on its own merits) and the **portfolio action** (what to do about it given the rest of the book). A holding can be an A-grade business and a *trim* because it is oversized, crowded, or duplicative; a B-grade business can be an *add* because it diversifies the book at better risk-adjusted upside. Collapsing the two hides that reasoning; separating them — and auditing each on its own — is what keeps an "A-grade" from silently becoming "add aggressively."

### Intrinsic verdict

Decided in the per-holding loop with **no whole-book context** — it is a property of the holding alone:

- **Composite grade** (A–F) over the engine's deterministically-computed sub-scores — **quality**, **valuation**, **momentum**, and **risk** — so the letter rolls up from real metrics, not a model's gestalt.
- **Conviction** — the verdict's confidence, lowered when evidence is thin or degraded and shaped by the **narrative-vs-reality** read (a price driven by multiple-expansion the estimates don't support caps conviction even on a strong company); below the evidence floor it abstains entirely (see [§Evidence floor](#evidence-floor)).
- **Horizon outlook** — separate short-, mid-, and long-term reads.
- **Price targets** — end-of-month and end-of-year, computed by the financial-analysis engine as scenario outputs with their **methodology and assumptions exposed** (including any research-sourced forward assumption folded in post-research — see [§The per-holding pipeline](#the-per-holding-pipeline) Step 5), aligned to the ledger's bear/base/bull monitor; the model selects and justifies the base case rather than inventing the number.
- **Standalone action lean** — what the action *would* be if the holding stood alone (a pure read of grade × conviction × upside/downside × risk on the same ladder), set **before** portfolio constraints. It is the input the construction stage reconciles, and showing it beside the final action is what makes the intrinsic-vs-portfolio split visible.
- **Financial analysis** — a concise read of the company's financial health.

### Portfolio action

Decided **after** the roll-up has the whole book in view (see [§Portfolio roll-up and construction](#portfolio-roll-up-and-construction)), so the action reflects the portfolio, not just the holding:

- **Action**, on a fixed ladder: **sell all → trim → hold → add → add aggressively**, made concrete with a **target portfolio-weight range and an estimated share/dollar adjustment**. No orders are ever placed.
- **The action-sizing spine.** The action and its size are reconciled from an explicit, mostly-deterministic input set rather than a model's gestalt: the **grade** and **conviction**, the **upside / downside** the targets imply against the current price, the **risk tier**, the **existing weight**, the **concentration headroom** against the profile's limits, the **correlation / exposure overlap** with the rest of the book (a name held directly and inside a fund, or several names sharing one factor, size down together), and the **cash / buying-power and tax** constraints. The engine computes these and **bounds the feasible action set** — *add aggressively* is offered only when buying power and concentration headroom support it; a *sell all* sitting on a large taxable gain carries that cost into the call — and the model chooses within those bounds and justifies it. The spine is what prevents a strong grade from sizing past what the portfolio can bear.

### What changed

A structured **audit of the prior verdict**, not a bare diff, **split to match the separation above**:

- The **intrinsic** half attributes every moved value (grade, any sub-score, conviction, target, horizon, a re-weighted scenario, a tripped falsifier or fired trigger) old → new to an **external change** — **market data** (price / momentum / positioning), **company information** (earnings, guidance, a new filing, a fundamentals shift), or **research / narrative** (a catalyst or narrative change), tied to the specific engine input-delta entry or source-backed finding that moved — or to a **self-correction**, where the inputs did *not* materially change and the model is revising its own prior read.
- The **action** half attributes a changed action or target weight to either a **moved intrinsic verdict** *or* a **moved portfolio context** (the holding became oversized, cash was freed for a higher-conviction name, an overlap emerged) — so an action can change with the intrinsic verdict unchanged, and that reads honestly rather than as a mystery re-grade.

A self-correction is the **flagged exception** to firm-output continuity — it swings the verdict with no new facts — so it is labeled as such rather than dressed as an external move, and repeated self-corrections on one holding are a calibration signal. The line also carries the **position delta** (new / increased / decreased / unchanged since last run; an unseen holding reads as `new`).

The fixed action ladder and grade vocabulary are load-bearing: like the report's fixed regime labels ([storage.md](storage.md)), they keep verdicts comparable across runs and prevent the model from retreating into hedged, non-comparable language. The grade is **intrinsic** to the holding; the **action** additionally reflects the investor profile (horizon, risk tolerance, tax sensitivity), available cash, and the whole-book context the construction stage supplies.

## Portfolio roll-up and construction

After the per-holding pass produces every **intrinsic verdict** and standalone action lean, a final stage takes the whole book in view and does the two things the per-holding loop structurally cannot — because that loop decides each holding before any other holding's verdict exists. It runs in two parts.

First, **deterministic whole-book aggregates** (application layer): concentration and sector / factor exposure (with fund exposure folded in at the **sector / country level** from each held fund's weightings — single-name look-through into fund constituents is off-plan with `etf/holdings`, so a name held both directly and inside an ETF aggregates at the **exposure** level, not per-name, unless SEC N-PORT supplies constituents), correlation / overlap clusters, the cash / buying-power position, **the risk / exposure contribution of any material not-rated positions** (fixed-income duration / credit weight, options notional / delta — graded nowhere but real exposure), and, per holding, the **action-sizing spine inputs** ([§The holding verdict](#the-holding-verdict)) — existing weight, concentration headroom, overlap contribution, upside/downside, risk tier, tax. These are computed, not modeled, and they are the inputs the next part is bounded by.

Then a **portfolio-construction synthesis** (122B) reconciles each holding's intrinsic standalone lean against those aggregates into its **final action and target weight**, and produces the portfolio-level view: overall risk posture, a cash / deployment stance (what to trim to fund which adds), the concentration and exposure read, and **positions closed since the last run** (the exited names from the holdings diff — graded nowhere, but acknowledged here rather than silently dropped) — all read against both the report's house view and the investor profile. This is the only stage that can say "trim the A-grade because it is 11% of the book and correlated with two other holdings, and route the proceeds to the B+ that diversifies it." The per-holding **action half of the what-changed audit** is written here, where the portfolio context exists, and is app-validated the same way the intrinsic half is — a "became oversized" claim must map to a real aggregate, not a model assertion. Beyond the sum of individual holdings, this is where the job answers "what does the portfolio as a whole look like, how should it be shaped, and how does it sit relative to the current market thesis."

## Continuity and isolation

The job retains its most recent N runs. The prior run feeds the next (continuity above), and run results are embedded into the **Portfolio Analysis memory partition only** — its learnings are never read by the report or by Trade Opportunities, and it never reads theirs (see [local-models.md §Run history and continuity](local-models.md#run-history-and-continuity)). This keeps holding-grading calibration cumulative within the job and uncontaminated by other jobs' context.

## Storage and display

Each run persists its per-holding verdicts, the per-holding **thesis ledger** (carried forward to seed the next run's continuity check), and the portfolio roll-up, together with an **audit record** that makes the run traceable and reviewable: the holdings snapshot it ran against, the report(s) and sources it used (with retrieval timestamps), the distilled findings and the **computed financial metrics, the derived reads (revision / narrative-vs-reality, forensic flags, positioning), the input delta and the per-value what-changed attribution, and the price-target methodology — including its discount-rate assumption and any research-sourced forward assumption applied in the post-distillation refinement (with its source) —** behind each verdict, the model ids and quantizations, the prompt/schema version, and any degraded-input flags. (Source URLs, timestamps, and the distilled findings make a verdict's basis inspectable; full page snapshots are not stored, so a run is traceable rather than bit-for-bit reproducible.) Retention keeps the last N runs (parallel to report retention — [storage.md](storage.md)). The **Portfolio page** renders each holding's verdict — the **intrinsic** read (grade and sub-scores, conviction, outlook, targets, the bear/base/bull monitor) beside the **portfolio action** (action, target weight, sizing rationale), with financials and the what-changed line — alongside the portfolio roll-up, and shows not-rated positions with their reason (see [interface.md](interface.md)).

## Failure posture

The execution gate requires the local model daemon and roster ([local-models.md](local-models.md)) **and a connected Schwab account** ([schwab-integration.md](schwab-integration.md)) — holdings and the options signal both come from Schwab, so a missing or lapsed connection blocks the job (manual-import holdings do not satisfy the gate). Because holdings are independent, the job **checkpoints per holding** — each holding's completed stages persist, so a cancellation or a single model failure resumes the unfinished holdings rather than restarting the (potentially hours-long) run, and recent research is cached within a freshness window. Within a run, web research is fail-soft (thinner evidence, lower conviction), while a hard model or persistence failure fails the run, recorded like any other job ([scheduling.md](scheduling.md)).
