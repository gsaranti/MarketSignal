# BUILD — Market Signal

## What we're building

Market Signal is a local-first macOS desktop app (Tauri 2 / Rust backend, Vue 3 frontend) that runs one recurring job — a **Weekly Market Report** generated every Sunday 9:00 AM local — and presents an evolving, professional market thesis rather than reactive daily commentary. A deterministic Rust pipeline gathers market data, macro data, and news; a constrained set of LLM "agents" reason over a curated packet to produce a Markdown report; the app renders it to HTML for display/PDF and keeps long-term continuity through vector memory. Everything runs on the user's machine except external API/model calls. The repo today is the stock Tauri+Vue scaffold (`src-tauri/src/lib.rs:2-5` is still the `greet` demo); this brief describes the system to build on top of it.

## The load-bearing decision: the app layer orchestrates; agents are pure stages

The single call the rest of the architecture is most sensitive to is the **boundary between the deterministic application layer and the agents**. Per `docs/agents.md`: *"The pipeline is not tool-driven by the main agent"* and *"The application layer executes external API calls and deterministic data retrieval."* Agents never touch the network, the database, or the filesystem. Each agent stage is a pure function: **structured input → structured output**. The Rust app layer owns the entire 18-step control flow (`docs/weekly-report-workflow.md`), all I/O, all limits, and all persistence; agents only consume the packet handed to them and emit schema-conformant results.

Commit to this as the spine. Concretely:

- Every agent stage sits behind a Rust trait (e.g. `MainAgent`, `AnalystAgent`, `HeadlineFilter`, `ResearchRouter`) whose method takes a typed request and returns a typed, validated response. The model HTTP call is an implementation detail of the adapter, swappable for a deterministic stub. **These trait methods are synchronous** — agent stages are pure `fn`s; the blocking provider HTTP call (`reqwest::blocking`) is offloaded via `spawn_blocking` at the application-layer seam (the Tauri command), so `tokio`/async lives only in app-layer I/O — the scheduler's timer — never in the agent trait itself (the bounded research executor shipped synchronous; see the Step-9 note below). (Established by the real-adapter slice; see `src-tauri/src/model_agent.rs` and `generate_report_manual` in `lib.rs`.)
- **Research planning is the router's job, not the main agent's** (resolved in walk C1): the fixed routing model produces the executable research plan at Step 8; the application layer executes it (Step 9); the main agent only shapes research indirectly through the inputs it surfaces. **The Step-11 condensed packet deviates from the docs** — `docs/weekly-report-workflow.md §Step 11` and `agents.md §Main Agent` assign packet-building to the main agent, but it ships as a **deterministic app-layer assembler** (`research_packet::build_condensed_packet`), not a model stage. Conscious extension: by Step 11 the upstream funnel (~500 headlines → ~10 stories → ~5 routed topics → bounded evidence) has already condensed, so assembling the token-bounded packet is plumbing, not reasoning — and it keeps faith with the pure-stage spine (agents reason; the app layer condenses and owns token control). Downstream code must not give the main agent a live tool loop.
- Research execution is hard-bounded — *"maximum 50 research requests per job ... maximum duration of 30 minutes for the research phase ... maximum dynamic-branching depth of 2"* (`docs/weekly-report-workflow.md §Step 9`). These bounds live in the executor, not the model. **As built (Step-9 slice, `src-tauri/src/research_executor.rs`), the executor is synchronous** — it runs inside the same `spawn_blocking` as the rest of the pipeline and issues sequential `reqwest::blocking` searches (≤50 calls × one backend timeout stays well inside 30 min), so the 30-minute budget is an elapsed-time check at each request boundary via an injectable `Clock` seam, not a `tokio` timeout; the request / time / depth gates are polled at request boundaries exactly like cancellation. Dynamic branching ships as **machinery only**: the depth-2 cap is enforced by the executor (a `BranchPolicy` cannot breach it). The real follow-up generator now ships — `DeltaBranchPolicy`, **deterministic delta-rules keyed off the change view**: a tracked series (oil `DCOILWTICO`, the 10y yield `DGS10`) moving past a threshold **in the doc's direction** ("oil spikes" / "yields rise sharply") emits one second-order follow-up query, attached to the first matching research topic; a `Direction` gate keeps a sharp *decline* from firing a rise-flavored follow-up, and a single fired rule emits at most once (bounded by the trait's one-follow-up-per-finding shape). `NoBranch` stays the trait's no-op default. **The research half is now wired into `generate_report`** (the research-half wiring slice): `pipeline::assemble_research_packet` runs news → filter → route → execute (selecting `DeltaBranchPolicy` vs `NoBranch` via `select_branch_policy` from the change view) → `build_condensed_packet`, and the packet rides into the main agent on `MainAgentInput.research` — an additive `Option<ResearchPacket>`; `baseline`/`deltas` stay top-level, so the packet's own copies of them are inert. **This wiring deviates from `docs/weekly-report-workflow.md §Step 9` (the line that treats a stuck or failing model call in any stage as a job failure).** As built the research half is **fully fail-soft**: news gather, the headline filter, the router, and the executor each degrade to empty on failure and the run always reaches the agent with whatever research landed — only the Step-3 coverage floor gates a run. Conscious extension (resolved with the user): a flaky Tavily / OpenAI / Anthropic call yields a *thinner* report rather than losing the weekly run; the cost is that a degraded run raises no failed-job warning, so the degradation surfaces only through the run tracker's per-request rows (one per real call, with a `failed` status) and stderr. Tavily's Step-7 gather is itself per-topic fail-soft (one bad topic no longer discards the rest or GDELT). Cancellation is polled at each research stage boundary — before the filter and router calls — and at each executor request boundary, so a mid-research cancel skips the remaining model calls rather than spending them.

Why it's load-bearing: this boundary decides the module graph, the testing strategy (agents become offline-stubbable pure functions), the data contracts (the research packet and each analyst's output schema are the API between halves), and the safety model (no unbounded agent I/O). If it were instead a tool-calling agent driving the run, the data model, the limits, and the test approach would all be different.

## Data model & storage

Three stores, by responsibility (`docs/storage.md`):

- **Filesystem** — canonical Markdown reports, named `YYYY-MM-DD-market-signal-weekly-report.md` (`docs/export.md §Export Naming`), plus the `/research-inbox` and `/research-archive` folders.
- **SQLite** — report records, report metadata, job history, warning states, and **per-report baseline snapshots** (the serialized Step-3 scan, behind the per-report change view — see below). The structured **report-summary metadata** is a JSON object the main agent populates; required fields are `report_id` (UUID), `report_type` (always `weekly_market`), `created_at` (ISO-8601), **`risk_posture`** ∈ {risk-on, risk-off, mixed}, **`market_cycle`** ∈ {late-cycle, recessionary, recovery}, `thesis_stance` ∈ {bullish, bearish, mixed, uncertain}, `header_summary_bullets` (3–6). The regime field was split into the two orthogonal axes `risk_posture`/`market_cycle` in walk Q2 — schema code must carry both, not a single `market_regime`. Optional arrays: `key_risks`, `unresolved_questions`, `forward_outlook_themes`.
- **Vector memory (SQLite)** — long-term semantic memory: one embedding per report summary and one per durable learning (`text-embedding-3-large`), each an atomic unit (no chunking). **Engine deviation (the vector-memory store slice, resolved with the user 2026-06-11):** the docs originally named LanceDB, but the store ships as a `vector_memory` table in the same SQLite database — little-endian-f32 BLOB embeddings with exact brute-force cosine in Rust (`src-tauri/src/vector_memory.rs`). Rationale: at this corpus's scale (≤30 summaries plus low-thousands of learnings over years) an unindexed LanceDB runs the same exhaustive scan, while the `lancedb` crate costs an arrow/lance/DataFusion dependency tree (~3M SLoC), a `protoc` build-machine requirement, 0.x breaking-change churn, and an async-only API against the deliberately synchronous spine. The engine swap stays contained behind two seams: the `vector_memory` module owns all store access, and the `embedding::Embedder` trait (deterministic stub + live OpenAI adapter, the model-adapter house pattern) owns text→vector. **As built, the Step-17 summary write is wired**: after a report persists, the structured summary's deterministic text rendering (`summary_memory_text` — the summary metadata, never report Markdown) is embedded and stored best-effort in the persist step, mirroring the baseline-snapshot block — a flaky embedding call or store failure costs the memory row, never the report. As built, both retrievals (Step-4 pre-research, Step-10 research-informed), both memory inputs (`RouterInput.memory` carries the Step-4 pull, now also surfaced to the main agent's audit on `MainAgentInput.audit_memory`; the packet's `memory` carries the Step-10 pull), and the durable-learning writes (`MainAgentOutput.durable_learnings`, persisted in the persist step) are all wired into `generate_report`. **Durable-learning dedup now ships** (the learning-dedup slice, `main` @ `80f4f9f`): the Step-17 persist drops a learning whose embedding is within a cosine threshold (`LEARNING_DEDUP_THRESHOLD`, 0.65 — an app-layer policy beside `LEARNINGS_PER_REPORT_CAP`, calibrated to real `text-embedding-3-large` geometry and tunable, not doc-pinned) of an existing `learning` row. `vector_memory::nearest_learning_similarity` (top-1 cosine over `learning` rows only, kind-filtered) reuses the embedding the persist loop already computed — no extra paid call — and scans this run's own prior inserts, so within-run *and* cross-run restatements both collapse; the incumbent (older) row wins. It is **fail-open**: a dedup-scan error keeps the learning, so the check can only ever drop a redundant row, never lose a real one — holding the leg's best-effort posture (the inline loop was extracted into `pipeline::persist_durable_learnings` as the unit-test seam; trim/cap, cancellation poll, and provenance are unchanged). One test caveat: the deterministic `StubEmbedder` collapses distinct prose to ~1.0 cosine (constant `v[0]=1.0` common-mode + positive byte increments), so it cannot drive a cosine-threshold test — the dedup *mechanism* uses a separating test embedder (`BasisEmbedder` / `DistinctEmbedder`) instead, while the threshold's *value* (0.65, down from a vacuously-high 0.93 that no real restatement reached) and `MEMORY_TOP_K` are calibrated against real geometry by the `#[ignore]`d live probes `tuning_dedup_threshold_calibration` / `tuning_topk_selectivity_probe`. **The Step-4 pull's audit consumer now ships** (the audit-memory slice, `main` @ `65811e1`): the same pre-research pull rides to the main agent on a new top-level `MainAgentInput.audit_memory` channel — a sibling of `baseline`/`deltas`, deliberately *not* inside the packet, so the packet keeps carrying only the Step-10 pull (the doc's §Step-10 replace-not-merge rule). `assemble_research_packet` returns `AssembledResearch { packet, audit_memory }` (the pull is cloned into routing, the original surfaced for the audit), computed inside the existing `research` run-tracker step so its embedding request row stays bracketed; fail-soft/cancel posture unchanged. The model adapter renders it as a prompt block distinct from the Step-10 memory. **The audit's prior-report context and gate now ship structurally** (the Step-2 prior-report-context slice, `main` @ `ea7a68f`): the main agent receives the recent prior reports — structured summary metadata **plus head-truncated Markdown bodies** — on a new top-level `MainAgentInput.recent_reports` channel (`Vec<RecentReport>`), loaded best-effort by `pipeline::load_recent_reports_for_audit` (the most-recent `MAIN_AGENT_RECENT_REPORTS`, 3, via `storage::list_recent_reports_with_paths`; each body head-truncated to `RECENT_REPORT_BODY_CAP`, 12_000 chars, with the inbox truncation-marker convention). These reports are the **Retrospective Audit's auditable object and its gate**: the section is written iff `recent_reports` is present, and `audit_memory` is narrowed to a *steering* input — its `[learning · …]` / `[summary · …]` fragments point at what to scrutinise and feed the thesis/strategy but no longer license the section. The gate stays **prompt-level** (the model writes the Markdown section), but its *input* is now **structural/deterministic** — recent-report presence ⇔ ≥1 prior report — rather than the earlier `audit_memory[summary]` recall proxy. Retrieval stays kind-blind (no per-kind quota), so learnings keep steering. The loader is additive/fail-soft like its router sibling (`load_recent_report_context`): a DB failure degrades the list to empty, an unreadable Markdown file drops that one report's body (its summary still carries), and neither ever gates the run. The two recent-report loaders (router summaries / main-agent bodies) **now share their boilerplate** (the loader-consolidation slice, `main` @ `e529d9d`): `pipeline::read_db_fail_soft<T: Default>` (this slice's `read_recent_fail_soft`, since renamed when the fold below generalized it past recent-report reads) owns the `open → init_schema → read → degrade-to-default-with-labeled-stderr` idiom both call, and `storage::list_recent_reports` is a summary-only projection over `list_recent_reports_with_paths` so the SQL + serde round-trip live once (the third consumer having arrived at the query layer — the sidebar list @ 30 plus the two loaders). The two loaders stay **distinct functions** and keep their **separate caps** (`ROUTER_RECENT_REPORTS` / `MAIN_AGENT_RECENT_REPORTS`, both 3 today) — only the duplicated idiom collapsed, so routing's read result is byte-for-byte unchanged (the one delta is the audit loader's stderr line losing its `Step-2` prefix under the shared template). A **follow-up fold then routed the pipeline's other two fail-soft DB reads onto the same shell** (renaming the helper `read_recent_fail_soft` → `read_db_fail_soft`, since it is no longer recent-report-specific): `retrieve_memory` (the Step-4 / Step-10 vector-memory pulls — its pre-open empty-query guard and `bounded_query` cap stay *outside* the shell, so a blank query still never opens the DB and the paid embedding call is still skipped), and `compute_prior_deltas` (the per-report change view — `Option<BaselineDeltas>` rides the `<T: Default>` bound since `None` is its default, so a DB or decode failure now degrades to a *logged* `None` where it was previously silent, while an absent prior snapshot stays the silent, expected `None`). All three fail-soft reads now share the single `open → init_schema → degrade` shell; the only cost was operator-log wording (a relocated colon under the neutral `{context}: degraded to empty` template — no test asserts stderr text). `MAIN_AGENT_RECENT_REPORTS` and `RECENT_REPORT_BODY_CAP` are app-layer tunables (not doc-pinned), joining the tuning bundle. Retention asymmetry is deliberate and must be honored in deletion code: only the most recent **30 reports** are kept (deleting a report cascades its Markdown, metadata, and vector *summary* row together — no HTML leg, since HTML is rendered on demand and never persisted (amended 2026-06-12) — `vector_memory::delete_report_summary` is the cascade's vector leg, the 30-report retention cascade having shipped and running best-effort after each report persists), **but durable learnings survive report deletion** — guaranteed by the row's `kind`, not its `report_id`, so a learning tagged with a deleted report's id still survives.

One deliberate exception to *persisted config lives in SQLite*: the **Light/Dark appearance** preference is stored in webview `localStorage`, not `app_settings` — it is pure presentation with no backend consumer (agents never see HTML; PDF export reuses the same themed DOM), and a synchronous pre-mount read in `main.ts` avoids a first-paint theme flash. `src/theme.ts` is its only writer.

The **Step-3 baseline scan** handed to the main agent (`BaselineMarketData`) is the live, **in-memory packet** — twelve groups: `indices`, `internals`, `sectors`, `macro_levels`, `labor_levels`, `calendar`, `index_performance` (multi-horizon index returns), the equity-breadth `movers` and `earnings`, and the valuation + finer-rotation `sector_pe`, `industries`, and `market_risk_premium` (the Step-3 micro-enrichment and valuation slices), plus a **`gaps` missing-data manifest**. The series adapters merge their contributions into it. **Partial failures degrade rather than abort** (the Step-3 partial-tolerance slice): an adapter no longer fails the scan on a flaky series or a down provider — it records every series/release it can't resolve as a `DataGap` tagged with its group and a reason (`out-of-scope` for a permanent/premium absence — excluded from coverage; `unavailable` / `rejected` / `malformed` for a this-run failure — transport/5xx, auth/quota, or a contract violation), and that manifest rides on the packet into the main agent's prompt so the model reasons over what's absent rather than inferring it. The **single coverage floor lives in the app layer**, not the adapters: `pipeline::enforce_coverage` is the sole point a too-thin baseline fails the run, and the floor is `indices` **and** at least one of {`internals`, `macro_levels`} each clearing `COVERAGE_FLOOR` (0.6 of a group's expected-minus-out-of-scope series) — so a rejected FMP or FRED key still fails the run loudly, while `sectors`, `labor_levels`, the additive `calendar` / `index_performance` / `movers` / `earnings` / `sector_pe` / `industries` / `market_risk_premium`, and whichever posture group didn't clear all degrade to recorded gaps. The `CompositeMarketDataSource` merges each provider's data *and* gaps, and now propagates only a *catastrophic* (non-data) child `Err`, not a data failure.

**Each run's baseline is also persisted** (the baseline-history slice): the app serializes the packet into a SQLite `baseline_snapshots` table — one row per report (`report_id`, an app-minted `captured_at`, a `BASELINE_SCHEMA_VERSION` stamp, and the JSON), keeping the most recent **14** (its own cap, self-pruned, decoupled from the 30-report report retention; `report_id` is stored so a future cascade can join). On the next run the app reads the latest prior snapshot and computes a deterministic **per-report change view** (`baseline_delta::compute_deltas`) — level-by-level deltas over the *level-bearing* groups only (`indices`, `internals`, `macro_levels`, `labor_levels`, `sector_pe`, `market_risk_premium`, joined by series id; `sectors`, returns, and set-valued groups excluded), with a series present in only one run surfaced as a `new`/`missing` transition that **carries the gap reason** so a data-feed outage reads differently from a series leaving the market. The view is **cadence-honest** — anchored on the elapsed interval `as_of − prior.captured_at` (reports aren't guaranteed weekly; the `as_of` anchor is deliberately distinct from the agent-minted `created_at`, joined by `report_id`), not an assumed week — and rides into the prompt on `MainAgentInput.deltas`. It is **additive / non-floor / fail-soft**: a first report, a DB error, or an undecodable older snapshot degrades to no deltas and never gates a run (forward-compat via `#[serde(default)]` on every `BaselineMarketData` field), and the snapshot write is best-effort (failures logged to stderr, never losing a generated report). The Step-8 research router reads the same computed view, and (the RouterInput slice) also receives the doc's fourth routing input, **recent report context**: the last **3** persisted `ReportSummary` rows (`pipeline::load_recent_report_context`), read best-effort exactly like the deltas — a first report or any DB failure degrades to empty and never gates — and rendered into the router prompt as a thesis-continuity block. Two deliberate scope calls: the doc's "recent **Markdown** report context" reaches *routing* in **structured summary form** (stance, key risks, unresolved questions, forward themes); the full **Markdown bodies** now ship to the **main agent** via the Step-2 prior-report-context slice (`MainAgentInput.recent_reports`; see Vector memory above), on a separate loader that leaves this router read untouched — the bodies deliberately do *not* feed routing; and "upcoming known market-moving events" needed no new field, since it already rides in on `baseline.calendar`. `RouterInput`'s seven doc inputs are now all wired — the last two, the Step-4 vector-memory pull (`memory`) and the Step-6 parsed inbox documents (`inbox_documents`), landed in the vector-memory and inbox-parsing slices respectively.

## Scheduling & runtime

A tray-resident app (closing the window must not quit it) runs a Rust timer that fires the weekly job at Sunday 9 AM local. Honesty about limits is part of the spec: jobs run only while the app is running; sleep ⇒ **missed** (never retroactively replayed); offline/unreachable provider ⇒ **failed**; a second concurrent run ⇒ **skipped** (single workflow at a time); a user-cancelled run ⇒ **cancelled** (a fifth terminal state, recorded like skipped — no report, and no failed-job warning, since it was intentional). Missed detection happens on next open by comparing last-run against the expected window. Manual execution reuses the identical workflow and validation.

**Live run observability + cancellation** ride on a Tauri-free `progress` seam so the deterministic spine stays unit-testable (`docs/run-tracking.md`). A `ProgressReporter` trait plus a per-run `RunContext` (the reporter, a shared cancel flag, and a sequence) are threaded into `generate_report` and the real adapters/agent via `with_context` builders — so **neither the `MarketDataSource` nor the `MainAgent` trait signature changes**; tests pass a no-op context. While a job runs the app streams to an open window: per-step progress, one **request row per actual HTTP call** (a `request-started` before each call and a `request-finished` after — a short-circuited series sends none and emits none), and the main agent's report **streamed token-by-token** (the SSE body is consumed on the blocking thread and the decoded `markdown` field emitted while the *full* envelope is accumulated and parsed exactly as the non-streaming path — so the token decoder is a side-channel that can't corrupt the report; the resumable decoder recombines UTF-16 surrogate pairs so non-BMP text — emoji — renders in the tracker as it does in the report). The frontend renders this as the **run tracker** (replacing the report pane while a run is in flight; latest-run-only, in-session). Cancellation is cooperative — a shared flag polled at step/request boundaries and mid-stream, never interrupting an in-flight request — and the flag is reset only once a run owns the concurrency slot, so a skipped competing run can't wipe an active run's cancellation. Two load-bearing UI invariants: a **run is never a report** (a row appears only on persisted success, so a cancel/fail removes nothing), and the terminal `run-finished` event is emitted **before** a job-history write error can propagate (so a DB failure can't strand the UI mid-run).

The **execution gate** (`docs/configuration.md`, `docs/weekly-report-workflow.md §Step 1`) blocks any run until: all four agent models are configured; **both** OpenAI and Anthropic tokens exist (always required — the fixed internal stages span both providers, walk C2); the required external credentials (Tavily, FMP, and FRED) are present; and the network is reachable. Failures surface in the **Persistent Warning Area**, which now has **five** de-duplicating categories (walk Q4): missing agent configuration, missing API tokens, missing provider credentials, failed jobs, missed scheduled jobs.

## Testing approach

The spine makes the pipeline testable offline: because agents and data adapters are traits, the orchestrator can be driven end-to-end against deterministic stubs and fixture packets, with no live keys. Cover: the bounded-research executor's three limits; the 30-report retention cascade *and* durable-learning survival; near-duplicate **learning dedup** (within-run and cross-run drop above the cosine threshold, fail-open on a scan error, driven by a separating test embedder since `StubEmbedder` collapses distinct prose to ~1.0 cosine); the validation gate's pass/block matrix; the **Step-3 coverage-floor matrix** (`enforce_coverage` — indices + at-least-one-posture-group pass/degrade, out-of-scope excluded); missed-vs-failed-vs-skipped-vs-**cancelled** state transitions (the cancelled run is driven by a data source that requests cancellation mid-scan, and a skipped competing run is asserted not to reset an active run's cancel flag); and research-inbox **fail-soft** parsing (an unparseable file is skipped + logged, left in the inbox, job continues — walk Q5). The baseline-history slice adds its own offline coverage: the delta engine's rise/fall/`new`/`missing` join and gap-reason carry, the cadence-honest elapsed pass-through, the snapshot store's latest/prune retention and forward-compat decode of an older blob, and an end-to-end two-run diff through `generate_report` (run 2 sees deltas vs run 1; both snapshots persist). The Step-2 prior-report-context slice adds: the audit loader's body read + head-truncate + marker and its fail-soft arms (a missing Markdown file keeps the summary with an empty body; an unopenable DB degrades to an empty list), `list_recent_reports_with_paths`' cap/order/path-pairing, and the `build_user_prompt` recent-reports block (present when reports are present, absent when empty, heading distinct from both memory blocks). The `progress` seam stays out of the way of every other test via a no-op `RunContext`; its own logic — the resumable streamed-token (`markdown`-field) decoder and the SSE delta/envelope reconstruction — is unit-tested against fixtures (including UTF-16 surrogate-pair recombination, matched against serde_json's own decode), and the `#[ignore]`d `live_generate_and_stream_smoke` exercises it against the *live* SSE wire for **both** dialects (OpenAI `delta.content`, Anthropic `input_json_delta`), asserting the streamed tokens reconstruct `out.markdown` exactly; it feeds a populated `MainAgentInput` fixture so a weak model's stub reply doesn't fail `generate`'s 3–6 header-bullets validation before the streaming assertions run. Real-provider adapters get thin integration tests behind a feature flag, and — the per-adapter offline-round-trip slice (`main` @ `481c60e`) — **offline wire-path coverage** of every gated adapter: a shared `#[cfg(test)] test_http::MockHttp` localhost server (lifted out of `http_retry.rs`'s test module, now recording each request's target) drives FMP/FRED/BLS/Tavily/fmp_news through `with_base_url` over the full URL-build → `send_with_retry` → `interpret_response` → domain-output path with no live key. 11 round-trips (a happy-path domain assertion plus one error-status case per adapter) assert the **exact** endpoint path (`request_paths()` `assert_eq!`, so a suffix typo like `/quote-v2` fails) and, where it varies, the per-call query var reaching the wire; scripts are single-reply on non-retryable statuses, so no `BASE_BACKOFF` sleeps — the retry/backoff/body-reread mechanics stay covered by `http_retry`'s own round-trips against the same mock. GDELT is excluded (single-shot fail-soft, not routed through `send_with_retry`). **The frontend unit gate is two runners under `npm test`, split by file extension**: pure modules (`tests/**/*.test.ts`, e.g. the chart renderer) run on Node's built-in runner via TypeScript type-stripping — no build step; Vue **SFC component tests** (`tests/**/*.spec.ts`) run on **Vitest** (`vitest.config.ts` — `@vitejs/plugin-vue` + happy-dom + `@vue/test-utils`, `include` scoped to `*.spec.ts` so it never grabs the `node:test` files), mounting real components to assert behavior and accessibility against the design system. The harness shipped with `tests/components/ResearchDocuments.spec.ts` (the inbox failed-row a11y contract — title tooltip, `aria-describedby` reason↔Delete pairing, two-step delete) as its first spec. **The Tauri-mock SFC pattern now ships** (`main` @ `59e20b1`): `tests/helpers/tauri.ts` is the reusable double for the four `@tauri-apps/api` modules `App.vue` imports (`core`/`event`/`window`/`app`) — a command-routing `invoke` that **throws on any unhandled command** (a new `onMounted` call fails loud, never silently `undefined`) plus default response shapes — and a spec wires it via hoisted `vi.fn()`s + `vi.mock` + a `beforeEach` `makeInvokeRouter()`. **Correction to the original framing: `App.vue` is the sole Tauri-calling SFC** — `Settings.vue` is presentational (props in, four events out) and is covered **mock-free**, like `ResearchDocuments`. `App.spec.ts` asserts the `onMounted` bootstrap contract and the `save_settings` / `set_job_enabled` round-trips through real template wiring; `Settings.spec.ts` pins the emit contract (omit-untouched credentials, the appearance/schedule toggles). **The event-folding extension now ships** (`main` @ `758d83e`): `tests/helpers/tauri.ts` also exports **`emitterFor(listenMock, event)`** — it captures the callback the mocked `listen` registered (reads `.mock.calls` structurally, stays `vi`-free) so a spec can drive App's `handleProgress` reducer by feeding `job-progress` `ProgressMessage`s the way the backend's stream would; a per-spec **`mountWithTracker`** helper folds a `run-started` + a `@view-tracker` emit to put a live run on screen. `App.spec.ts`'s coverage now spans **run-tracker event folding** (request routing by `group`, agent-token accumulation, `step-finished` status+detail, `run-finished` terminal + still-running-step reconciliation, straggler `run_id` filtering), **report selection** (`@select`→`load_report`, the list-vs-load error-channel separation), and the **cancel path** (`@cancel`→`cancel_run`, the `!runActive` guard, `@view-tracker`/`@close` glue). Load-bearing test idiom: every assertion reads through **child-component props** (`JobTrackerView`/`LatestReportView`/`JobStatusPanel`/`RecentReportsSidebar`), because `App.vue` is `<script setup>` with **no `defineExpose`** — internal refs are unreachable via `wrapper.vm`. The Node engine range is pinned to the intersection of both runners' support — `>=22.18 <23 || >=24` (type-stripping needs 22.18+; Vitest 4 excludes the EOL Node 23 line).

## First vertical slice: manual report, stub agent, end to end

The thinnest runnable pass that exercises the load-bearing boundary without touching the financial-data adapters or live API keys:

1. **One Tauri command** — replace the scaffold `greet` with `generate_report_manual` in `src-tauri/src/lib.rs`, wired into `invoke_handler`.
2. **One agent stage, stubbed** — a `MainAgent` trait with a deterministic stub impl that returns a fixed structured result: a small valid report Markdown body plus a report-summary object (`report_id`, `report_type="weekly_market"`, `created_at`, `risk_posture`, `market_cycle`, `thesis_stance`, `header_summary_bullets`). This proves the structured-in/out contract that the real adapters will later satisfy.
3. **One database write + one file write** — create the SQLite `reports` table and insert the record + summary JSON; write the canonical `YYYY-MM-DD-market-signal-weekly-report.md` to the reports directory.
4. **One screen** — the Vue **Latest Report View** calls the command, receives the Markdown, and renders it via markdown-it (styled with design-system tokens).
5. **One passing test** — a Rust integration test that invokes the pipeline with the stub and asserts both the SQLite row (with `risk_posture` + `market_cycle` populated) and the `.md` file exist.

This slice runs offline, instantiates the app→agent-stage→persist→render flow that the whole system is built on, and turns the architecture's central bet (deterministic orchestration around a pure agent stage) into something executable.
