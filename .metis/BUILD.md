# BUILD — Market Signal

## What we're building

Market Signal is a local-first macOS desktop app (Tauri 2 / Rust backend, Vue 3 frontend) that runs one recurring job — a **Weekly Market Report** generated every Sunday 9:00 AM local — and presents an evolving, professional market thesis rather than reactive daily commentary. A deterministic Rust pipeline gathers market data, macro data, and news; a constrained set of LLM "agents" reason over a curated packet to produce a Markdown report; the app renders it to HTML for display/PDF and keeps long-term continuity through vector memory. Everything runs on the user's machine except external API/model calls. The repo today is the stock Tauri+Vue scaffold (`src-tauri/src/lib.rs:2-5` is still the `greet` demo); this brief describes the system to build on top of it.

## The load-bearing decision: the app layer orchestrates; agents are pure stages

The single call the rest of the architecture is most sensitive to is the **boundary between the deterministic application layer and the agents**. Per `docs/agents.md`: *"The pipeline is not tool-driven by the main agent"* and *"The application layer executes external API calls and deterministic data retrieval."* Agents never touch the network, the database, or the filesystem. Each agent stage is a pure function: **structured input → structured output**. The Rust app layer owns the entire 18-step control flow (`docs/weekly-report-workflow.md`), all I/O, all limits, and all persistence; agents only consume the packet handed to them and emit schema-conformant results.

Commit to this as the spine. Concretely:

- Every agent stage sits behind a Rust trait (e.g. `MainAgent`, `AnalystAgent`, `HeadlineFilter`, `ResearchRouter`) whose method takes a typed request and returns a typed, validated response. The model HTTP call is an implementation detail of the adapter, swappable for a deterministic stub. **These trait methods are synchronous** — agent stages are pure `fn`s; the blocking provider HTTP call (`reqwest::blocking`) is offloaded via `spawn_blocking` at the application-layer seam (the Tauri command), so `tokio`/async lives only in app-layer I/O — the scheduler's timer — never in the agent trait itself (the bounded research executor shipped synchronous; see the Step-9 note below). (Established by the real-adapter slice; see `src-tauri/src/model_agent.rs` and `generate_report_manual` in `lib.rs`.)
- **Research planning is the router's job, not the main agent's** (resolved in walk C1): the fixed routing model produces the executable research plan at Step 8; the application layer executes it (Step 9); the main agent only shapes research indirectly through the inputs it surfaces. **The Step-11 condensed packet deviates from the docs** — `docs/weekly-report-workflow.md §Step 11` and `agents.md §Main Agent` assign packet-building to the main agent, but it ships as a **deterministic app-layer assembler** (`research_packet::build_condensed_packet`), not a model stage. Conscious extension: by Step 11 the upstream funnel (~500 headlines → ~10 stories → ~5 routed topics → bounded evidence) has already condensed, so assembling the token-bounded packet is plumbing, not reasoning — and it keeps faith with the pure-stage spine (agents reason; the app layer condenses and owns token control). Downstream code must not give the main agent a live tool loop.
- Research execution is hard-bounded — *"maximum 50 research requests per job ... maximum duration of 30 minutes for the research phase ... maximum dynamic-branching depth of 2"* (`docs/weekly-report-workflow.md §Step 9`). These bounds live in the executor, not the model. **As built (Step-9 slice, `src-tauri/src/research_executor.rs`), the executor is synchronous** — it runs inside the same `spawn_blocking` as the rest of the pipeline and issues sequential `reqwest::blocking` searches (≤50 calls × one backend timeout stays well inside 30 min), so the 30-minute budget is an elapsed-time check at each request boundary via an injectable `Clock` seam, not a `tokio` timeout; the request / time / depth gates are polled at request boundaries exactly like cancellation. Dynamic branching ships as **machinery only**: the depth-2 cap is enforced by the executor (a `BranchPolicy` cannot breach it). The real follow-up generator now ships — `DeltaBranchPolicy`, **deterministic delta-rules keyed off the change view**: a tracked series (oil `DCOILWTICO`, the 10y yield `DGS10`) moving past a threshold **in the doc's direction** ("oil spikes" / "yields rise sharply") emits one second-order follow-up query, attached to the first matching research topic; a `Direction` gate keeps a sharp *decline* from firing a rise-flavored follow-up, and a single fired rule emits at most once (bounded by the trait's one-follow-up-per-finding shape). `NoBranch` stays the trait's no-op default. **The research half is now wired into `generate_report`** (the research-half wiring slice): `pipeline::assemble_research_packet` runs news → filter → route → execute (selecting `DeltaBranchPolicy` vs `NoBranch` via `select_branch_policy` from the change view) → `build_condensed_packet`, and the packet rides into the main agent on `MainAgentInput.research` — an additive `Option<ResearchPacket>`; `baseline`/`deltas` stay top-level, so the packet's own copies of them are inert. **This wiring deviates from `docs/weekly-report-workflow.md §Step 9` (the line that treats a stuck or failing model call in any stage as a job failure).** As built the research half is **fully fail-soft**: news gather, the headline filter, the router, and the executor each degrade to empty on failure and the run always reaches the agent with whatever research landed — only the Step-3 coverage floor gates a run. Conscious extension (resolved with the user): a flaky Tavily / OpenAI / Anthropic call yields a *thinner* report rather than losing the weekly run; the cost is that a degraded run raises no failed-job warning, so the degradation surfaces only through the run tracker's per-request rows (one per real call, with a `failed` status) and stderr. Tavily's Step-7 gather is itself per-topic fail-soft (one bad topic no longer discards the rest or GDELT). Cancellation is polled at each research stage boundary — before the filter and router calls — and at each executor request boundary, so a mid-research cancel skips the remaining model calls rather than spending them.

Why it's load-bearing: this boundary decides the module graph, the testing strategy (agents become offline-stubbable pure functions), the data contracts (the research packet and each analyst's output schema are the API between halves), and the safety model (no unbounded agent I/O). If it were instead a tool-calling agent driving the run, the data model, the limits, and the test approach would all be different.

## Data model & storage

Three stores, by responsibility (`docs/storage.md`):

- **Filesystem** — canonical Markdown reports, named `YYYY-MM-DD-market-signal-weekly-report.md` (`docs/export.md §Export Naming`), plus the `/research-inbox` and `/research-archive` folders.
- **SQLite** — report records, report metadata, generated HTML, job history, warning states, and **per-report baseline snapshots** (the serialized Step-3 scan, behind the per-report change view — see below). The structured **report-summary metadata** is a JSON object the main agent populates; required fields are `report_id` (UUID), `report_type` (always `weekly_market`), `created_at` (ISO-8601), **`risk_posture`** ∈ {risk-on, risk-off, mixed}, **`market_cycle`** ∈ {late-cycle, recessionary, recovery}, `thesis_stance` ∈ {bullish, bearish, mixed, uncertain}, `header_summary_bullets` (3–6). The regime field was split into the two orthogonal axes `risk_posture`/`market_cycle` in walk Q2 — schema code must carry both, not a single `market_regime`. Optional arrays: `key_risks`, `unresolved_questions`, `forward_outlook_themes`.
- **LanceDB** — long-term semantic memory: one embedding per report summary and one per durable learning (`text-embedding-3-large`), each an atomic unit (no chunking). Retention asymmetry is deliberate and must be honored in deletion code: only the most recent **30 reports** are kept (deleting a report cascades its Markdown, HTML, metadata, and vector *summary* reference together), **but durable learnings survive report deletion**.

One deliberate exception to *persisted config lives in SQLite*: the **Light/Dark appearance** preference is stored in webview `localStorage`, not `app_settings` — it is pure presentation with no backend consumer (agents never see HTML; PDF export reuses the same themed DOM), and a synchronous pre-mount read in `main.ts` avoids a first-paint theme flash. `src/theme.ts` is its only writer.

The **Step-3 baseline scan** handed to the main agent (`BaselineMarketData`) is the live, **in-memory packet** — twelve groups: `indices`, `internals`, `sectors`, `macro_levels`, `labor_levels`, `calendar`, `index_performance` (multi-horizon index returns), the equity-breadth `movers` and `earnings`, and the valuation + finer-rotation `sector_pe`, `industries`, and `market_risk_premium` (the Step-3 micro-enrichment and valuation slices), plus a **`gaps` missing-data manifest**. The series adapters merge their contributions into it. **Partial failures degrade rather than abort** (the Step-3 partial-tolerance slice): an adapter no longer fails the scan on a flaky series or a down provider — it records every series/release it can't resolve as a `DataGap` tagged with its group and a reason (`out-of-scope` for a permanent/premium absence — excluded from coverage; `unavailable` / `rejected` / `malformed` for a this-run failure — transport/5xx, auth/quota, or a contract violation), and that manifest rides on the packet into the main agent's prompt so the model reasons over what's absent rather than inferring it. The **single coverage floor lives in the app layer**, not the adapters: `pipeline::enforce_coverage` is the sole point a too-thin baseline fails the run, and the floor is `indices` **and** at least one of {`internals`, `macro_levels`} each clearing `COVERAGE_FLOOR` (0.6 of a group's expected-minus-out-of-scope series) — so a rejected FMP or FRED key still fails the run loudly, while `sectors`, `labor_levels`, the additive `calendar` / `index_performance` / `movers` / `earnings` / `sector_pe` / `industries` / `market_risk_premium`, and whichever posture group didn't clear all degrade to recorded gaps. The `CompositeMarketDataSource` merges each provider's data *and* gaps, and now propagates only a *catastrophic* (non-data) child `Err`, not a data failure.

**Each run's baseline is also persisted** (the baseline-history slice): the app serializes the packet into a SQLite `baseline_snapshots` table — one row per report (`report_id`, an app-minted `captured_at`, a `BASELINE_SCHEMA_VERSION` stamp, and the JSON), keeping the most recent **14** (its own cap, self-pruned, decoupled from the 30-report report retention; `report_id` is stored so a future cascade can join). On the next run the app reads the latest prior snapshot and computes a deterministic **per-report change view** (`baseline_delta::compute_deltas`) — level-by-level deltas over the *level-bearing* groups only (`indices`, `internals`, `macro_levels`, `labor_levels`, `sector_pe`, `market_risk_premium`, joined by series id; `sectors`, returns, and set-valued groups excluded), with a series present in only one run surfaced as a `new`/`missing` transition that **carries the gap reason** so a data-feed outage reads differently from a series leaving the market. The view is **cadence-honest** — anchored on the elapsed interval `as_of − prior.captured_at` (reports aren't guaranteed weekly; the `as_of` anchor is deliberately distinct from the agent-minted `created_at`, joined by `report_id`), not an assumed week — and rides into the prompt on `MainAgentInput.deltas`. It is **additive / non-floor / fail-soft**: a first report, a DB error, or an undecodable older snapshot degrades to no deltas and never gates a run (forward-compat via `#[serde(default)]` on every `BaselineMarketData` field), and the snapshot write is best-effort (failures logged to stderr, never losing a generated report). The Step-8 research router will read the same computed view.

## Module boundaries

- **`app` (Rust orchestrator)** — the 18-step pipeline, the bounded research executor, the scheduler, validation/gating, warning-state management, the **baseline-snapshot persistence + deterministic per-report delta computation** (`baseline_delta`), and the `progress` run-observability + cancellation seam (see Scheduling & runtime). This is where determinism lives.
- **`adapters` (Rust)** — `data_sources` (FMP/FRED/BLS REST via reqwest — FMP serves equity-market data (indices, VIX, gold and **silver**, sector performance, company financials, **multi-horizon index performance** — weekly / MTD / YTD / 52-week-range returns from FMP's free historical end-of-day prices — the equity-breadth **movers** (gainers / losers / most-actives) and free **earnings calendar**, and the **valuation + finer-rotation** snapshots: per-**sector** and per-**industry** P/E plus the strongest/weakest industries by move, gathered for **both NASDAQ and NYSE** — exchange-specific, so the growth-vs-value valuation spread is surfaced — and the US **equity-risk-premium**), FRED the macro series including the **dollar index, commodities (oil, natural gas), and Treasury yields** plus the **credit spreads (high-yield + investment-grade OAS, with the BBB and single-B buckets for credit-quality dispersion) and 10y–3m / 10y–2y curve spreads**, the **equity-volatility term structure (S&P 3-month VIX / VXV, Nasdaq-100 VXN)**, the macro-levels group (Fed-funds target range, inflation breakevens, consumer sentiment, PCE; the headline activity reports PPI, retail sales, JOLTS, GDP; and the weekly risk/cycle gauges — **financial-conditions indices NFCI/ANFCI/STLFSI4, jobless claims initial+continued, the Fed balance sheet, and the 30-year mortgage rate**), and the **economic-release calendar** (the prior-week + upcoming US release schedule, from FRED's free release-dates API), BLS labor; data access stays in Rust, no OpenBB/Python sidecar; Tavily + GDELT for news) and `models` (OpenAI + Anthropic HTTP). Live-verified 2026-06-05: FMP's free tier gates the dollar index / oil / gas behind premium, which is why those market-internal series moved to FRED. Live-verified 2026-06-06: FMP's economic-calendar endpoint is likewise premium (HTTP 402), so the Step-3 calendar's release schedule is sourced from FRED too — no free source carries analyst consensus, so the calendar's `expected` value is reserved for a future paid tier. Live-verified 2026-06-06: FMP's free historical-EOD endpoint serves all four indices and the VIX, so the index-performance enrichment stays free; FRED's documented limit is 120 req/min with no daily cap, so the ~45-request weekly scan is well within budget. Live-verified 2026-06-07: FMP's **movers**, **earnings calendar**, per-**sector** / per-**industry** P/E, **market-risk-premium**, and **silver** are all free-tier (constituent lists, batch quotes, and copper are premium); the sector / industry snapshots are **exchange-specific** (a no-`exchange` call defaults to NASDAQ; NYSE and AMEX are also free), so the adapter pins both NASDAQ and NYSE, labels each row by its **wire** exchange, and fails a leg whose returned board doesn't match the request rather than silently mislabelling or duplicating one board. **FMP's free tier is hard-capped at 250 requests/day** — a real constraint on live smoke/probe runs. The gated adapters (FMP/FRED/BLS/Tavily) share a bounded retry/backoff for transient HTTP-status/transport failures (429/5xx/dropped connections, `Retry-After`-aware); GDELT is excluded — its escalating IP lockout makes retrying harmful, so it keeps its single-shot fail-soft. Fixed internal stages are pinned here: GPT-5 mini (headline filtering, data extraction), Claude Sonnet (research routing), `text-embedding-3-large` (embeddings) — non-configurable, distinct from the four user-selectable agent models.
- **`agents` (prompt + schema contracts)** — main agent, Bull/Bear/Balanced analysts (run **concurrently**, no ordering dependency — walk Q6), and the **16 analyst skills as a shared library** surfaced by progressive disclosure: agents first receive each skill's frontmatter, then request the full skill (prompt + output schema) on demand from the packet (walk Q1).
- **`frontend` (Vue 3)** — Latest Report View, the **Run Tracker** (live per-step + per-request job progress with streamed agent output, shown in place of the report while a run is in flight; `docs/run-tracking.md`), Recent Reports Sidebar, Research Documents, Persistent Warning Area, Settings (`docs/interface.md`). All UI is built against the design system in `market-signal-design-system/` (its `SKILL.md` and tokens are binding per project CLAUDE.md). Markdown→HTML rendering uses **markdown-it**, which is JS — so HTML generation lives on the frontend/webview side; the backend persists the rendered HTML. Agents never see HTML.

The existing scaffold gives us exactly one seam to extend: the Tauri command boundary (`invoke_handler` at `src-tauri/src/lib.rs:11`). Cargo deps to add: a SQLite layer (`rusqlite` or `sqlx`), `lancedb`, `reqwest`, `tokio`, `uuid`, `chrono`, and the Tauri tray/notification features.

## Scheduling & runtime

A tray-resident app (closing the window must not quit it) runs a Rust timer that fires the weekly job at Sunday 9 AM local. Honesty about limits is part of the spec: jobs run only while the app is running; sleep ⇒ **missed** (never retroactively replayed); offline/unreachable provider ⇒ **failed**; a second concurrent run ⇒ **skipped** (single workflow at a time); a user-cancelled run ⇒ **cancelled** (a fifth terminal state, recorded like skipped — no report, and no failed-job warning, since it was intentional). Missed detection happens on next open by comparing last-run against the expected window. Manual execution reuses the identical workflow and validation.

**Live run observability + cancellation** ride on a Tauri-free `progress` seam so the deterministic spine stays unit-testable (`docs/run-tracking.md`). A `ProgressReporter` trait plus a per-run `RunContext` (the reporter, a shared cancel flag, and a sequence) are threaded into `generate_report` and the real adapters/agent via `with_context` builders — so **neither the `MarketDataSource` nor the `MainAgent` trait signature changes**; tests pass a no-op context. While a job runs the app streams to an open window: per-step progress, one **request row per actual HTTP call** (a `request-started` before each call and a `request-finished` after — a short-circuited series sends none and emits none), and the main agent's report **streamed token-by-token** (the SSE body is consumed on the blocking thread and the decoded `markdown` field emitted while the *full* envelope is accumulated and parsed exactly as the non-streaming path — so the token decoder is a side-channel that can't corrupt the report). The frontend renders this as the **run tracker** (replacing the report pane while a run is in flight; latest-run-only, in-session). Cancellation is cooperative — a shared flag polled at step/request boundaries and mid-stream, never interrupting an in-flight request — and the flag is reset only once a run owns the concurrency slot, so a skipped competing run can't wipe an active run's cancellation. Two load-bearing UI invariants: a **run is never a report** (a row appears only on persisted success, so a cancel/fail removes nothing), and the terminal `run-finished` event is emitted **before** a job-history write error can propagate (so a DB failure can't strand the UI mid-run).

The **execution gate** (`docs/configuration.md`, `docs/weekly-report-workflow.md §Step 1`) blocks any run until: all four agent models are configured; **both** OpenAI and Anthropic tokens exist (always required — the fixed internal stages span both providers, walk C2); the required external credentials (Tavily, FMP, and FRED) are present; and the network is reachable. Failures surface in the **Persistent Warning Area**, which now has **five** de-duplicating categories (walk Q4): missing agent configuration, missing API tokens, missing provider credentials, failed jobs, missed scheduled jobs.

## Testing approach

The spine makes the pipeline testable offline: because agents and data adapters are traits, the orchestrator can be driven end-to-end against deterministic stubs and fixture packets, with no live keys. Cover: the bounded-research executor's three limits; the 30-report retention cascade *and* durable-learning survival; the validation gate's pass/block matrix; the **Step-3 coverage-floor matrix** (`enforce_coverage` — indices + at-least-one-posture-group pass/degrade, out-of-scope excluded); missed-vs-failed-vs-skipped-vs-**cancelled** state transitions (the cancelled run is driven by a data source that requests cancellation mid-scan, and a skipped competing run is asserted not to reset an active run's cancel flag); and research-inbox **fail-soft** parsing (an unparseable file is skipped + logged, left in the inbox, job continues — walk Q5). The baseline-history slice adds its own offline coverage: the delta engine's rise/fall/`new`/`missing` join and gap-reason carry, the cadence-honest elapsed pass-through, the snapshot store's latest/prune retention and forward-compat decode of an older blob, and an end-to-end two-run diff through `generate_report` (run 2 sees deltas vs run 1; both snapshots persist). The `progress` seam stays out of the way of every other test via a no-op `RunContext`; its own logic — the resumable streamed-token (`markdown`-field) decoder and the SSE delta/envelope reconstruction — is unit-tested against fixtures (the *live* SSE wire format is not yet exercised). Real-provider adapters get thin integration tests behind a feature flag; UI components get component tests against the design system.

## First vertical slice: manual report, stub agent, end to end

The thinnest runnable pass that exercises the load-bearing boundary without touching the financial-data adapters or live API keys:

1. **One Tauri command** — replace the scaffold `greet` with `generate_report_manual` in `src-tauri/src/lib.rs`, wired into `invoke_handler`.
2. **One agent stage, stubbed** — a `MainAgent` trait with a deterministic stub impl that returns a fixed structured result: a small valid report Markdown body plus a report-summary object (`report_id`, `report_type="weekly_market"`, `created_at`, `risk_posture`, `market_cycle`, `thesis_stance`, `header_summary_bullets`). This proves the structured-in/out contract that the real adapters will later satisfy.
3. **One database write + one file write** — create the SQLite `reports` table and insert the record + summary JSON; write the canonical `YYYY-MM-DD-market-signal-weekly-report.md` to the reports directory.
4. **One screen** — the Vue **Latest Report View** calls the command, receives the Markdown, and renders it via markdown-it (styled with design-system tokens).
5. **One passing test** — a Rust integration test that invokes the pipeline with the stub and asserts both the SQLite row (with `risk_posture` + `market_cycle` populated) and the `.md` file exist.

This slice runs offline, instantiates the app→agent-stage→persist→render flow that the whole system is built on, and turns the architecture's central bet (deterministic orchestration around a pure agent stage) into something executable. The immediate next slices: swap the stub for a real OpenAI/Anthropic `MainAgent` adapter (and the config/token gate); persist generated HTML + add PDF export; then build out the data-source adapters (FMP/FRED/BLS REST).
